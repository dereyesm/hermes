## 1. Abstract

HERMES (**Heterogeneous Event Routing for Multi-agent Ephemeral Sessions**) is a
lightweight, file-based communication protocol for autonomous AI agents that
operate within stateless sessions. Inspired by the TCP/IP protocol stack, HERMES
provides a five-layer architecture that enables agents organized into isolated
namespaces to exchange state, alerts, events, and data through a shared message
bus -- without requiring servers, databases, or persistent processes.

HERMES is designed for environments where agents are invoked on demand, share a
file system, and must coordinate across trust boundaries while keeping a human
operator in authoritative control.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
"SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be
interpreted as described in RFC 2119.

### 2.1 Definitions

**Agent (Skill)** -- An autonomous or semi-autonomous AI entity that reads from
and writes to the Amaru bus. An agent operates within a single namespace and
executes tasks on behalf of a human operator.

**Namespace (Dimension)** -- A logical isolation boundary that groups agents,
configuration files, and external service credentials. Namespaces enforce
security policy. An agent MUST NOT access resources outside its own namespace
unless explicitly permitted by the routing table.

**Clan** -- The complete set of namespaces and agents under a single HERMES
deployment. A clan shares one bus, one routing table, and one archive.

**Bus** -- The append-only message log (JSONL file) through which all
inter-namespace communication flows. The bus is the sole communication channel
between namespaces.

**Instance** -- A single running session of an agent. Because sessions are
stateless, each instance begins by reading the bus (SYN) and ends by writing to
it (FIN).

**Router** -- A designated agent (or agent role) with read access to ALL
namespaces and the full bus. The router detects desynchronization and proposes
corrections. There MUST be exactly one router role per clan.

**Operator** -- The human who owns the clan. The operator has final authority
over all actions. HERMES informs; the operator decides.

**Packet** -- A single JSON object (one line) in the bus file, representing one
message between namespaces.

**TTL (Time-To-Live)** -- The number of days a packet remains active on the bus
before it is archived.

**SYNC HEADER** -- A metadata block embedded in each namespace's configuration
file that summarizes the namespace's last known state and pending message counts.

## 3. Protocol Stack

HERMES defines a five-layer stack. Each layer has a single responsibility and a
clear boundary with adjacent layers.

| Layer | Name        | TCP/IP Analog | HERMES Function                | Implementation             |
|-------|-------------|---------------|--------------------------------|----------------------------|
| L0    | Physical    | Physical      | File system                    | Directory paths per namespace |
| L1    | Frame       | Data Link     | Packet format                  | JSONL with standardized header |
| L2    | Routing     | Network (IP)  | Routing table                  | Static routes file mapping namespaces to paths and agents |
| L3    | Transport   | TCP           | Message bus with acknowledgment | Active bus file + archive file |
| L4    | Application | Application   | Agents and orchestrators       | Each agent reads/writes the bus per protocol |

### 3.1 L0 -- Physical Layer

The physical layer is the file system. Each namespace maps to one or more
directories containing its configuration, memory, and agent definitions.

An implementation MUST define a base sync directory containing:

```
<sync-root>/
  bus.jsonl            -- active messages
  bus-archive.jsonl    -- expired messages
  routes.<ext>         -- routing table
  PROTOCOL.<ext>       -- this specification (optional, recommended)
```

Namespace directories live outside the sync root. The routing table (L2)
provides the mapping.

### 3.2 L1 -- Frame Layer

Every packet on the bus is a single JSON object occupying exactly one line in the
bus file (JSONL format). A conforming packet MUST contain the following fields:

```json
{
  "ts":   "YYYY-MM-DD",
  "src":  "<source-namespace>",
  "dst":  "<target-namespace> | *",
  "type": "<message-type>",
  "msg":  "<payload>",
  "ttl":  <integer>,
  "ack":  []
}
```

| Field  | Type            | Description |
|--------|-----------------|-------------|
| `ts`   | ISO 8601 date   | Emission timestamp. MUST be set by the sender. |
| `src`  | string          | The namespace that created this packet. |
| `dst`  | string or `"*"` | Target namespace, or `"*"` for broadcast to all namespaces. |
| `type` | enum string     | Message classification (see Section 3.2.1). |
| `msg`  | string          | Compact payload. SHOULD be a single line, max 200 characters. One packet MUST carry exactly one topic (atomicity rule). |
| `ttl`  | integer         | Days until expiration, counted from `ts`. |
| `ack`  | array of string | Namespaces that have consumed this packet. |

#### 3.2.1 Message Types

Implementations MUST support the following core types:

| Type         | Semantics                                      | Default TTL |
|--------------|-------------------------------------------------|-------------|
| `state`      | A namespace's state has changed                 | 7           |
| `alert`      | Urgent information another namespace must see   | 5           |
| `event`      | Something happened (informational)              | 3           |
| `request`    | A namespace needs something from another        | 7           |
| `data_cross` | Permitted data transfer across namespace boundary | 7         |

Implementations MAY define additional types (e.g., `dispatch`, `system`) as long
as unrecognized types are preserved without modification by conforming readers.

#### 3.2.2 Payload Conventions

1. **Atomicity**: One packet = one topic. If there are two things to
   communicate, send two packets.
2. **Projection prefix**: When a `data_cross` payload contains projected or
   estimated values (not verified data), the `msg` field MUST begin with the
   prefix `PROJ:`.
3. **Source annotation**: When a `data_cross` payload includes numerical values,
   implementations SHOULD append a source tag in the format `@<source-id>` at
   the end of the `msg` field.

### 3.3 L2 -- Routing Layer

The routing table is a static file that maps each namespace to:

- Its configuration file paths
- Its memory/state file paths
- Its agent directory paths
- Its permitted external services (if any)
- Its permitted cross-namespace data flows

The routing table MUST also declare **adjacency** -- which namespaces are
permitted to exchange packets. By default, namespaces are isolated; only
explicitly declared routes are valid.

A conforming implementation MUST reject or flag packets whose `src`/`dst` pair
is not present in the routing table, unless `dst` is `"*"` (broadcast).

### 3.4 L3 -- Transport Layer

The transport layer manages the lifecycle of packets on the bus.

#### 3.4.1 SYN -- Session Start

When an agent instance begins, it MUST execute the SYN procedure:

1. Read the bus file.
2. Filter packets where `dst` matches the agent's namespace OR `dst` is `"*"`,
   AND the agent's namespace is NOT in the `ack` array.
3. Present pending packets to the operator (or log them).
4. If any unacknowledged packets have an age exceeding 3 days, the
   implementation SHOULD emit a staleness warning.

#### 3.4.2 FIN -- Session End

When an agent instance terminates, it MUST execute the FIN procedure:

1. If the session produced state changes relevant to other namespaces, append
   new packet(s) to the bus file.
2. For each packet consumed during this session, add the agent's namespace to
   that packet's `ack` array.
3. Update the SYNC HEADER in the namespace's configuration file.

#### 3.4.3 Acknowledgment

A packet is considered **consumed** by a namespace when that namespace's
identifier appears in the `ack` array. A packet is considered **fully
acknowledged** when all intended recipients have acknowledged it:

- For unicast (`dst` = specific namespace): one entry in `ack`.
- For broadcast (`dst` = `"*"`): all non-source namespaces in `ack`.

#### 3.4.4 Expiration and Archival

A packet is **expired** when the current date exceeds `ts` + `ttl` days. Expired
packets MUST be moved from the active bus file to the archive file. They MUST NOT
be deleted. The archive serves as an audit trail.

#### 3.4.5 SYNC -- Router Reconciliation

The router agent SHOULD periodically execute a full reconciliation:

1. Read ALL SYNC HEADERS from every namespace.
2. Read the full bus.
3. Detect inconsistencies: unacknowledged packets, stale SYNC HEADERS,
   expired packets still on the active bus.
4. Report desynchronizations to the operator.
5. With operator approval: update SYNC HEADERS, acknowledge packets, archive
   expired packets.

### 3.5 L4 -- Application Layer

The application layer is where agents operate. Each agent:

- Belongs to exactly one namespace.
- Reads from and writes to the bus using L3 procedures.
- MAY have access to external services (APIs, tools), but ONLY those permitted
  by its namespace's entry in the routing table.
- MUST NOT invoke external services belonging to another namespace.

Agents MAY be organized hierarchically within a namespace (e.g., a lead agent
that delegates to specialist agents), but this internal structure is opaque to
the protocol. HERMES only governs inter-namespace communication.

## 4. SYNC HEADER

Each namespace's configuration file MUST contain a SYNC HEADER block with the
following fields:

| Field         | Type    | Description                                        |
|---------------|---------|----------------------------------------------------|
| `version`     | integer | Monotonically increasing counter, incremented on each state change. |
| `last_sync`   | date    | The date of the most recent SYN or FIN operation.  |
| `state`       | string  | One-line summary of the namespace's current state. |
| `pending_out` | integer | Count of packets emitted by this namespace that have not been fully acknowledged. |
| `pending_in`  | integer | Count of packets addressed to this namespace that have not been consumed. |

The SYNC HEADER is the authoritative snapshot of a namespace's synchronization
status. It MUST be updated during every FIN procedure.

## 5. Design Principles

HERMES is built on five non-negotiable design principles:

### 5.1 File-Based -- No Servers, No Databases

All protocol state lives in plain text files on the file system. There are no
daemons, no databases, no network services. Any agent that can read and write
files can participate. This makes HERMES trivially deployable, version-
controllable, and auditable.

### 5.2 Stateless Sessions

Agents do not maintain persistent connections or in-memory state across
invocations. Every session starts from the file system (SYN) and commits back to
it (FIN). This matches the operational model of CLI-based AI agents that are
invoked on demand.

### 5.3 Human-in-the-Loop

HERMES is an information protocol, not an execution framework. The bus carries
awareness between namespaces; the human operator retains authority over all
consequential actions. An implementation MUST NOT take irreversible actions based
solely on bus messages without operator confirmation.

### 5.4 Firewall by Default

Namespaces are isolated by default. Communication requires an explicit entry in
the routing table. External service credentials MUST NOT cross namespace
boundaries under any circumstances. Data MAY cross boundaries only through
`data_cross` packets and only along routes declared in the routing table.

### 5.5 Auditability

No data is silently discarded. Expired packets are archived, not deleted. SYNC
HEADERS provide a verifiable snapshot of each namespace's state. The bus file is
append-only during normal operation (modifications are limited to updating `ack`
arrays).

## 6. Conformance Levels

An implementation claims conformance at one of three levels:

### 6.1 Level 1 -- Minimal

The implementation MUST:
- Read and write conforming packets (L1).
- Execute SYN and FIN procedures (L3).
- Respect namespace isolation (L4).

### 6.2 Level 2 -- Standard

The implementation MUST satisfy Level 1, and additionally:
- Validate packets against the routing table (L2).
- Manage TTL expiration and archival (L3).
- Maintain SYNC HEADERS (L3/L4).

### 6.3 Level 3 -- Full

The implementation MUST satisfy Level 2, and additionally:
- Provide a router reconciliation function (L3, SYNC procedure).
- Detect and report desynchronization across all namespaces.
- Support all core message types and payload conventions (L1).

## 7. Security Considerations

### 7.1 Credential Isolation

External service credentials (API keys, OAuth tokens, session cookies) MUST NOT
appear in bus packets, SYNC HEADERS, or any protocol file. Credentials are bound
to a namespace and MUST NOT be referenced or transmitted across namespace
boundaries.

### 7.2 Bus File Permissions

The bus file and archive file SHOULD be readable and writable only by the
operator and authorized agent processes. Implementations SHOULD set restrictive
file permissions (e.g., `0600` on POSIX systems).

### 7.3 Payload Sensitivity

The `msg` field SHOULD NOT contain secrets, personally identifiable information,
or sensitive data. If sensitive data must be referenced, the payload SHOULD
contain a pointer (e.g., a file path within the source namespace) rather than
the data itself.

### 7.4 Routing Table Integrity

The routing table defines the security perimeter. Modifications to the routing
table MUST require operator approval. Implementations SHOULD treat the routing
table as immutable during a session.

### 7.5 Archive as Audit Trail

The archive file is the protocol's audit log. It MUST NOT be truncated or
deleted during normal operation. Operators MAY rotate or compress archives
outside of active sessions.

### 7.6 Broadcast Scope

Broadcast packets (`dst: "*"`) are delivered to ALL namespaces. Senders MUST
ensure that broadcast payloads do not contain information restricted to a
specific namespace.
