# ARC-1918: Private Spaces and Namespace Firewalls

| Field | Value |
|-------|-------|
| **Number** | ARC-1918 |
| **Title** | Private Spaces and Namespace Firewalls |
| **Lineage** | Inspired by RFC 1918 — Address Allocation for Private Internets |
| **Status** | IMPLEMENTED |
| **Created** | 2026-02-28 |
| **Requires** | HERMES Core Protocol |

## Abstract

This specification defines the isolation model for HERMES namespaces. Each namespace operates within a private space that contains its own configuration, memory, agents, and tool access. A firewall governs which external tools each namespace may invoke and under which conditions data may cross namespace boundaries. The model ensures that credentials never leave their originating namespace and that cross-namespace data movement requires explicit human authorization.

The design mirrors RFC 1918's principle: private address spaces are unreachable from outside unless explicitly routed. In HERMES, a namespace's tools, credentials, and memory are its private address space.

## Motivation

AI agent orchestration systems that manage multiple domains (engineering, finance, operations, etc.) face a fundamental trust boundary problem. An agent working in a finance namespace should never accidentally invoke an engineering namespace's deployment tools. A corporate namespace's API credentials must never be readable by a personal namespace's agents.

Without formal isolation, a single misconfigured agent can:
- Send messages using the wrong identity
- Access tools outside its authorization scope
- Leak sensitive data across organizational boundaries
- Execute actions in a namespace where it has no authority

ARC-1918 provides the specification for preventing all of these failure modes.

## Terminology

| Term | Definition |
|------|-----------|
| **Namespace** | A logically isolated domain of work (e.g., `engineering`, `finance`, `operations`) |
| **Private Space** | The set of files, configuration, memory, and agents belonging to a namespace |
| **Firewall** | The access control layer that governs tool access and data flow per namespace |
| **Tool** | Any external service, API, or MCP server that an agent can invoke |
| **Controller** | A privileged namespace that can read state across all namespaces but cannot execute actions in them |
| **Data-cross** | A sanctioned transfer of specific data from one namespace to another |

## 1. Private Space Definition

Each namespace MUST have its own private space, consisting of:

```
<namespace_root>/
  CLAUDE.md              # Namespace configuration and state
  .claude/
    skills/              # Agents scoped to this namespace
    memory/              # Persistent memory for this namespace
    settings.json        # Tool (MCP) configuration for this namespace
```

### 1.1 Ownership Rules

- Every file within a namespace's private space is OWNED by that namespace.
- No agent from another namespace may read, write, or delete files in a foreign private space.
- The only exception is the Controller namespace (see Section 5), which has read-only access to SYNC HEADER sections.

### 1.2 Agent Scoping

Agents (skills) are scoped to the namespace where they are defined:

- An agent defined in `engineering/.claude/skills/` can only be invoked from within the `engineering` namespace.
- Agents MAY travel between namespaces through a promotion mechanism (see Section 6), but they adopt the firewall rules of their current namespace upon arrival.
- Global agents (defined at the orchestrator level) have NO tool access. They exist solely for deliberation and coordination.

### 1.3 Memory Isolation

Each namespace maintains its own memory store:

- Memory files are private to the namespace.
- Cross-namespace memory access is PROHIBITED.
- Shared state is communicated exclusively through the Amaru bus using typed messages.

## 2. Firewall Rule Specification

Each namespace declares its firewall rules in its configuration. Rules follow a default-deny model: any tool not explicitly listed in the ALLOW table is denied.

### 2.1 Rule Format

```yaml
namespace: engineering
firewall:
  allow:
    - tool: github
      scope: full
    - tool: jira
      scope: full
    - tool: slack
      scope: [read, write]
  deny:
    - tool: personal-email
    - tool: banking-api
    - tool: social-media
```

### 2.2 ALLOW Table

The ALLOW table enumerates every tool the namespace may access, along with the permitted scope:

| Field | Type | Description |
|-------|------|-------------|
| `tool` | string | Identifier of the external tool or MCP server |
| `scope` | string or list | `full`, `read-only`, or a list of permitted operations |

If a tool does not appear in the ALLOW table, access is implicitly denied.

### 2.3 DENY Table

The DENY table provides explicit documentation of prohibited tools. While implicit denial (absence from ALLOW) is sufficient for enforcement, the DENY table serves two purposes:

1. **Human readability** -- makes the security boundary obvious at a glance.
2. **Audit trail** -- documents intentional exclusions and their rationale.

### 2.4 Evaluation Order

```
1. Check DENY table -- if tool is listed, REJECT immediately
2. Check ALLOW table -- if tool is listed with valid scope, PERMIT
3. Default -- REJECT (deny by default)
```

### 2.5 Example: Multi-Namespace Firewall

| Namespace | ALLOW | DENY |
|-----------|-------|------|
| `engineering` | github, jira, ci-pipeline, monitoring | personal-email, banking-api |
| `finance` | banking-api, accounting-svc, personal-email | github, ci-pipeline, jira |
| `operations` | monitoring, ticketing, docs | github, banking-api |
| `controller` | _(none -- read-only state access)_ | _(all tools)_ |

No tool appears in the ALLOW column of two namespaces that should not share it. The controller has no tool access at all.

## 3. Data-Cross Protocol

Certain data types MAY cross namespace boundaries, but only through the Amaru bus and only with explicit permission.

### 3.1 Data-Cross Message Format

Cross-namespace data travels as a `data_cross` message on the bus:

```jsonl
{
  "ts": "2026-02-28",
  "src": "operations",
  "dst": "finance",
  "type": "data_cross",
  "msg": "monthly_hosting_costs_1200_as_infrastructure@billing_system",
  "ttl": 7,
  "ack": []
}
```

### 3.2 Permission Table

Each HERMES instance MUST define a permission table specifying which data types may cross which boundaries:

| Source | Destination | Permitted Data | Category at Destination |
|--------|-------------|----------------|------------------------|
| `operations` expenses | `finance` | Cost amounts, categories | "Operations" |
| `engineering` infrastructure | `finance` | Hosting/tooling costs | "Engineering" |
| `operations` facilities | `finance` | Rent, utilities | "Facilities" |

### 3.3 Data-Cross Rules

1. **Only data, never tools or credentials.** A `data_cross` message carries categorized information (amounts, statuses, labels). It MUST NOT carry API keys, tokens, passwords, or tool configurations.
2. **Projection marking.** If the data represents an estimate or projection rather than a verified fact, the message MUST use the `PROJ:` prefix in the `msg` field.
3. **Source attribution.** Monetary amounts MUST include a source annotation using `@source` suffix (e.g., `monthly_costs_1200@billing_system`).
4. **Human approval.** The receiving namespace SHOULD NOT auto-consume `data_cross` messages without human review. Implementations MAY enforce this as a hard requirement.
5. **One topic per message.** Each `data_cross` message carries exactly one data point. Batching multiple unrelated values into a single message is prohibited.

### 3.4 Prohibited Crosses

The following data types MUST NEVER cross namespace boundaries:

- Credentials (API keys, tokens, passwords, certificates)
- Tool configurations (MCP server URIs, OAuth client IDs)
- Session state (conversation history, agent memory)
- Identity artifacts (account emails, user IDs specific to a namespace's services)

## 4. Identity Isolation Model

### 4.1 Credential Binding

Each namespace binds to exactly one identity context:

```yaml
namespace: engineering
identity:
  account: engineering-team@company.com
  tools_authenticated_as: engineering-team@company.com
```

### 4.2 Isolation Guarantees

1. **No credential sharing.** If `engineering` authenticates to GitHub as `eng-team@corp.com`, the `finance` namespace MUST NOT use that credential, even if both namespaces could theoretically access GitHub.
2. **No identity spoofing.** An agent in namespace A MUST NOT send messages, emails, or API calls that appear to originate from namespace B's identity.
3. **Separate authentication.** If two namespaces need access to the same external service, each MUST authenticate independently with its own credentials.
4. **Credential storage.** Credentials are stored within the namespace's private space and are not readable by any other namespace or by the Amaru bus.

### 4.3 Identity Verification

Before any tool invocation, the runtime MUST verify:

```
1. The invoking agent belongs to the current namespace
2. The tool is in the namespace's ALLOW table
3. The credential being used belongs to the current namespace's identity
4. The action is consistent with the declared scope
```

Failure at any step MUST result in a hard rejection, not a warning.

## 5. Controller Privileges and Limitations

A controller namespace (e.g., a global orchestrator or strategy layer) has elevated read access but zero execution capability.

### 5.1 What the Controller CAN Do

| Capability | Mechanism |
|-----------|-----------|
| Read SYNC HEADERS from all namespaces | Direct file read of the SYNC HEADER section only |
| Read all messages on the Amaru bus | Standard bus read (bus.jsonl) |
| Detect desynchronization across namespaces | Compare SYNC HEADERS against bus state |
| Propose corrective actions | Emit `state` or `alert` messages to the bus |
| Route and dispatch tasks | Emit `dispatch` messages assigning agents to tasks |

### 5.2 What the Controller CANNOT Do

| Prohibition | Rationale |
|------------|-----------|
| Invoke any external tool | Controller has an empty ALLOW table |
| Write to any namespace's private space | Ownership rules (Section 1.1) |
| Read namespace memory or agent definitions | Only SYNC HEADERS are visible |
| Execute actions on behalf of a namespace | Only the namespace's own agents can execute |
| Override firewall rules | Firewall is declarative and immutable at runtime |
| Auto-approve data-cross messages | Human approval required (Section 3.3) |

### 5.3 Separation of Concerns

The controller embodies the principle: **observe and coordinate, never execute.**

This mirrors the SDN (Software-Defined Networking) model where the control plane makes routing decisions but the data plane performs the actual packet forwarding. The controller decides what should happen; the namespaces decide whether and how to do it.

## 6. Agent Mobility

Agents may be promoted to operate across namespace boundaries under strict conditions:

1. **Promotion requires a governance event** recorded on the bus (`dojo_event` type).
2. **Upon entering a new namespace, the agent adopts that namespace's firewall rules.** It loses access to its origin namespace's tools and gains access to the destination namespace's tools.
3. **The agent DOES NOT carry credentials.** It authenticates using the destination namespace's identity.
4. **The agent DOES NOT carry memory.** It starts with a clean state in the new namespace, receiving only what the destination namespace explicitly shares.

## 7. Security Considerations

### 7.1 Threat Model

| Threat | Mitigation |
|--------|-----------|
| Agent in namespace A invokes namespace B's tools | Firewall default-deny + identity verification (Sections 2, 4) |
| Credential leakage through bus messages | Bus carries only state/data, never credentials (Section 3.4) |
| Controller escalation to execution | Controller has empty ALLOW table, enforced at runtime (Section 5.2) |
| Data exfiltration via data-cross | Permission table + human approval + source attribution (Section 3) |
| Identity spoofing across namespaces | Credential binding + separate authentication (Section 4) |
| Memory poisoning from foreign namespace | Memory isolation + no cross-namespace reads (Section 1.3) |

### 7.2 Implementation Requirements

Implementations MUST:

1. Enforce firewall rules before every tool invocation, not just at session start.
2. Validate namespace ownership of credentials at authentication time.
3. Reject `data_cross` messages that contain patterns matching credential formats (API keys, tokens, connection strings).
4. Log all cross-namespace data movements for audit purposes.
5. Provide a mechanism for human review of the firewall configuration.

### 7.3 Non-Goals

This specification does NOT address:

- Encryption of bus messages (assumed to be on a trusted local filesystem)
- Authentication of HERMES instances to each other (see future ATR on peering)
- Rate limiting or quota management across namespaces
- Multi-user access control (HERMES assumes a single human operator)

## Compatibility

ARC-1918 is a foundational specification. It constrains behavior defined in the Amaru Core Protocol by adding:

- Firewall evaluation before tool invocation
- Identity verification before credential use
- Permission table checks before `data_cross` message delivery

Existing bus message types (`state`, `alert`, `event`, `request`, `dispatch`, `dojo_event`) are unaffected. Only `data_cross` messages are subject to the additional permission table validation.

## Reference Implementation

The reference implementation adds:

- `hermes.firewall.Firewall` -- evaluates ALLOW/DENY rules for a namespace
- `hermes.identity.IdentityContext` -- binds credentials to namespaces and validates ownership
- `amaru.bus.DataCrossValidator` -- validates `data_cross` messages against the permission table
- Configuration schema for firewall rules in namespace `settings.json`
