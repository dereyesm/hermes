> *"HTTP succeeded because RFC 7231 defined semantics independently of transport —
> GET means GET whether over TCP, QUIC, or a carrier pigeon. Agent protocols need
> the same: a shared semantic layer that lets gateways translate between worlds
> without losing meaning."*

## 1. Abstract

This specification defines the semantic mapping between HERMES internal message formats and external agent communication protocols, enabling gateways (ARC-3022) to translate bidirectionally between HERMES and protocols such as Google A2A and Anthropic MCP. It establishes field-level equivalences, message type mappings, and translation flows so that a HERMES clan can participate in external protocol ecosystems without exposing its internal structure.

RFC 7231 separated HTTP semantics from wire format, allowing multiple transports to share the same request/response meanings. ARC-7231 does the same for agent communication: it defines what operations mean — query, invoke, notify, update — independently of whether the transport is JSONL on a file system, JSON-RPC over HTTP/2, or stdio-based tool calls.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in ARC-2119, which follows RFC 2119.

| Term | Definition |
|------|-----------|
| **Bridge** | A gateway function that translates between HERMES internal formats and an external agent protocol |
| **External Protocol** | An agent communication protocol outside the HERMES ecosystem (e.g., A2A, MCP) |
| **Semantic Mapping** | A defined equivalence between concepts in two protocols, specifying how to translate one to the other |
| **Translation Flow** | The end-to-end path of a message as it crosses the gateway from one protocol format to another |
| **Agent Card** | A2A's agent description document, equivalent to a HERMES Agent Profile (ARC-2606) |
| **Tool** | An MCP concept representing a callable function; maps to a HERMES agent capability with invocation semantics |
| **Resource** | An MCP concept representing a readable data source; maps to HERMES namespace read permissions (ARC-1918) |
| **Inbound Bridge** | Translation from an external protocol format into HERMES JSONL |
| **Outbound Bridge** | Translation from HERMES JSONL into an external protocol format |

## 3. A2A to HERMES Mapping

### 3.1 Concept Mapping

Google A2A (Agent-to-Agent, v0.3.0 — July 2025) defines a horizontal agent communication protocol with three transport bindings: JSON-RPC 2.0 over HTTPS, gRPC with Protocol Buffers, and HTTP REST. It supports Server-Sent Events for streaming, push notifications via webhooks, and Agent Cards for discovery. Donated to the Linux Foundation in June 2025 with 150+ backing organizations. The following table maps A2A concepts to their HERMES equivalents.

| A2A Concept | HERMES Equivalent | Translation Notes |
|---|---|---|
| Agent Card | Agent Profile (ARC-2606) | Gateway publishes HERMES profile as A2A Agent Card. See Section 3.2 for field mapping. |
| Task | Message sequence with CID correlation (ARC-0768) | A2A task lifecycle maps to a CID-correlated request/response pair. |
| Message (JSON-RPC) | JSONL message (ARC-5322) | Gateway translates JSON-RPC envelope to/from JSONL. One JSON-RPC call = one HERMES message. |
| Artifact | Message with `encoding="ref"` (ARC-5322 future) | Large payloads via file reference. Until encoding evolution is implemented, artifacts are summarized in the 120-char payload with an external reference. |
| Streaming (SSE) | Polling bus.jsonl / future ARC-6455 | HERMES is async-first. Real-time streaming requires ARC-6455 (PLANNED). Gateway MAY buffer SSE events and write them as sequential bus messages. |
| Task State (submitted, working, completed, failed, input-required, auth-required) | REL message lifecycle (ARC-0768) | `submitted` = message created; `working` = ACKed; `completed` = `[RE:token]` received; `failed` = escalation on TTL expiry; `input-required` = request message from target; `auth-required` = no HERMES equivalent (gateway returns error). |
| Push Notification | `alert` message type (ARC-5322) | A2A push notifications translate to HERMES alerts with appropriate TTL. |
| Context (contextId) | CID correlation group (ARC-0768) | A2A v0.3 `contextId` groups related tasks. Maps to a shared CID prefix in HERMES for conversational continuity. |
| Signed Agent Card | Profile with Ed25519 signature (ARC-8446) | A2A v0.3 supports cryptographic signature on Agent Cards. HERMES profiles can include Ed25519-signed metadata for integrity verification. |
| gRPC binding | Not applicable | A2A v0.3 added gRPC transport. HERMES gateways MAY implement a gRPC bridge endpoint as an alternative to JSON-RPC. The translation semantics are identical. |

### 3.2 Agent Card to HERMES Profile Field Mapping

The gateway MUST be able to translate between an A2A Agent Card and a HERMES Agent Profile (ARC-2606). The following field mapping defines the equivalence.

#### 3.2.1 Outbound: HERMES Profile to A2A Agent Card

When a gateway publishes a HERMES agent to the A2A ecosystem, it constructs an Agent Card from the HERMES profile:

| A2A Agent Card Field | HERMES Profile Source | Translation Rule |
|---|---|---|
| `name` | `alias` | Direct mapping. The external alias becomes the A2A agent name. |
| `description` | `description` (from Clan Profile, ARC-2606 Section 5.1) | If absent in the agent profile, use the clan-level description. |
| `url` | Gateway endpoint | The gateway MUST expose an HTTP endpoint when bridging to A2A. This URL does not exist in the HERMES profile; the gateway generates it. |
| `provider.organization` | `clan_id` | The clan identifier serves as the provider organization name. |
| `provider.url` | Gateway endpoint or Agora directory URL | Implementation-specific. |
| `version` | `protocol_versions[0]` | Use the highest supported HERMES protocol version. |
| `capabilities.streaming` | `false` (unless ARC-6455 implemented) | HERMES is async-first. Set to `true` only if the gateway implements real-time bridge. |
| `capabilities.pushNotifications` | `true` | HERMES supports `alert` messages, which map to push notifications. |
| `skills[].id` | `capabilities[].path` | Each HERMES capability path becomes an A2A skill ID. |
| `skills[].name` | Derived from `capabilities[].path` | Extract the specialization segment (last path component). |
| `skills[].description` | Not available | HERMES capability paths are taxonomic, not descriptive. The gateway MAY generate descriptions from the ontology. |
| `authentication.schemes` | Gateway auth config | TOFU maps to `none` initially. When ARC-7519 is implemented, expose `bearer` with JWT. |
| `extensions` | Not applicable | A2A v0.3 extension declarations. The gateway SHOULD declare `hermes-bridge` as an extension for interoperability signaling. |

#### 3.2.2 Inbound: A2A Agent Card to HERMES Profile

When a gateway discovers an A2A agent and wants to represent it internally:

| HERMES Profile Field | A2A Agent Card Source | Translation Rule |
|---|---|---|
| `alias` | `name` | Sanitize to `[a-z0-9-]` format (ARC-2606 Section 4.2). Truncate to 64 chars. |
| `clan_id` | `provider.organization` | Sanitize to `[a-z0-9-]` format. If absent, derive from URL hostname. |
| `capabilities[].domain` | Inferred from `skills[].id` | Map to closest ARC-2606 ontology domain. Unmapped skills use `x-` prefix. |
| `capabilities[].path` | `skills[].id` | If the skill ID follows a hierarchical format, preserve it. Otherwise, construct a path using the inferred domain. |
| `capabilities[].confidence` | Not available | Default to `secondary` for externally discovered agents. |
| `availability` | Inferred from `capabilities.streaming` | `streaming: true` maps to `always-on`; otherwise `on-demand`. |
| `protocol_versions` | Not applicable | Record as `["a2a-bridge"]` to mark this as a bridged profile. |
| `visibility` | — | Always `direct-request` for externally bridged agents. |

### 3.3 A2A Task to HERMES Message Sequence

An A2A task lifecycle maps to a CID-correlated HERMES message sequence:

```
A2A Client                    Gateway                    HERMES Bus
    |                            |                            |
    | tasks/send (JSON-RPC)      |                            |
    |--------------------------->|                            |
    |                            | Translate to JSONL:         |
    |                            | type: "dispatch"            |
    |                            | msg: "[CID:a2a-task-42] ..." |
    |                            |--------------------------->|
    |                            |                            |
    |                            |     [internal processing]  |
    |                            |                            |
    |                            | Read response from bus:     |
    |                            | msg: "[RE:a2a-task-42] ..." |
    |                            |<---------------------------|
    |                            |                            |
    | tasks/get response         |                            |
    | (JSON-RPC result)          |                            |
    |<---------------------------|                            |
    |                            |                            |
```

**Translation rules for A2A task operations:**

| A2A Operation | HERMES Translation |
|---|---|
| `tasks/send` | Create `dispatch` message with `[CID:task-id]`. Route to target agent's namespace. |
| `tasks/get` | Query bus for message with `[RE:task-id]`. Return result or current status. |
| `tasks/cancel` | Create `alert` message with `[RE:task-id] CANCELLED`. |
| `tasks/sendSubscribe` (SSE) | Create `dispatch` message. Gateway polls bus for `[RE:task-id]` and emits SSE events. Polling interval SHOULD NOT exceed 1 second. |

### 3.4 A2A Task State Translation

| A2A Task State | HERMES Condition | Detection Method |
|---|---|---|
| `submitted` | Message created on bus | CID message exists, no ACK. |
| `working` | Message ACKed by target namespace | Target namespace in `ack` array, no `[RE:token]` message. |
| `input-required` | No direct equivalent | Gateway MAY translate a `request` message from the target agent as `input-required`. |
| `completed` | Resolution message received | `[RE:token]` companion message exists on bus. |
| `failed` | REL escalation triggered | TTL expired without `[RE:token]`; escalation alert generated (ARC-0768 Section 6). |
| `canceled` | Cancel alert received | Alert message with `[RE:token] CANCELLED` on bus. |

## 4. MCP to HERMES Mapping

### 4.1 Concept Mapping

Anthropic MCP (Model Context Protocol) defines a vertical protocol between language models and tools/resources, using JSON-RPC over stdio or Streamable HTTP. The following table maps MCP concepts to their HERMES equivalents.

| MCP Concept | HERMES Equivalent | Translation Notes |
|---|---|---|
| Tool | Agent capability (ARC-2606) with invocation semantics | Each MCP tool maps to a HERMES capability. Tool calls translate to `dispatch` messages. |
| Resource | Namespace read permission (ARC-1918) | MCP resources map to data accessible within a namespace, governed by the firewall. |
| Prompt | Message template | A pre-formatted message pattern stored in the gateway configuration. |
| Server | Agent namespace | Each MCP server corresponds to one HERMES namespace behind the gateway. |
| Streamable HTTP | bus.jsonl append + read | MCP's Streamable HTTP transport maps to bus file operations. Requests = append; responses = read. |
| Sampling | `request` message | MCP sampling (model invocation) maps to a HERMES request targeting an LLM-backed agent. |
| Roots | Namespace filesystem scope | MCP roots define the filesystem boundary, analogous to HERMES namespace isolation (ARC-1918). |
| Notification | `event` or `alert` message | MCP notifications map to DGM-mode messages (ARC-0768). |

### 4.2 MCP Tool to HERMES Capability Mapping

#### 4.2.1 Outbound: HERMES Capability as MCP Tool

When a gateway exposes HERMES agents as MCP tools:

| MCP Tool Field | HERMES Source | Translation Rule |
|---|---|---|
| `name` | `alias` + capability path | Construct as `{alias}_{specialization}`. Example: `aureus_debt_strategy`. |
| `description` | Derived from capability path | Generate from ontology. Example: `"finance/personal/debt-strategy"` becomes `"Personal finance debt strategy analysis"`. |
| `inputSchema` | Not natively available | The gateway MUST define input schemas per tool in its bridge configuration. HERMES capabilities are semantic, not schematic. |

#### 4.2.2 Inbound: MCP Tool Call to HERMES Message

When the gateway receives an MCP `tools/call` request:

```
MCP Client                    Gateway                    HERMES Bus
    |                            |                            |
    | tools/call (JSON-RPC)      |                            |
    | tool: "aureus_debt_strategy"|                           |
    | args: {...}                 |                            |
    |--------------------------->|                            |
    |                            | Translate:                  |
    |                            | type: "dispatch"            |
    |                            | dst: "finance"              |
    |                            | msg: "[CID:mcp-call-7] ..." |
    |                            |--------------------------->|
    |                            |                            |
    |                            |     [agent processes]      |
    |                            |                            |
    |                            | Read resolution:            |
    |                            | msg: "[RE:mcp-call-7] ..." |
    |                            |<---------------------------|
    |                            |                            |
    | tools/call result          |                            |
    | (JSON-RPC response)        |                            |
    |<---------------------------|                            |
    |                            |                            |
```

**Translation rules:**

1. The gateway MUST parse the tool name to resolve the target namespace and agent.
2. Tool call arguments MUST be serialized into the `msg` field if they fit within the payload limit, or stored as a referenced file if they exceed it (using `encoding="ref"` when available).
3. The gateway MUST generate a unique CID for each tool call and track it until resolution.
4. If the tool call produces a result exceeding the HERMES payload limit, the gateway MUST store the full result externally and include a reference in the `[RE:token]` message.

### 4.3 MCP Resource to HERMES Namespace Permission

| MCP Resource Operation | HERMES Translation |
|---|---|
| `resources/list` | Query gateway configuration for published namespace resources. The gateway maintains a resource registry mapping namespace data to MCP resource URIs. |
| `resources/read` | Create `query` message targeting the resource's namespace. The namespace agent reads the requested data and responds via `[RE:token]`. The gateway translates the response to MCP resource content. |
| `resources/subscribe` | Create a standing query. The gateway polls the bus for `state` or `event` messages from the resource namespace and emits MCP notifications on change. |

### 4.4 MCP Prompt to HERMES Message Template

MCP prompts are pre-defined message templates that guide model interactions. In HERMES, these map to gateway-configured message templates:

```yaml
# Gateway bridge configuration
mcp_bridge:
  prompts:
    - name: "analyze-debt"
      hermes_template:
        type: "dispatch"
        dst: "finance"
        msg_prefix: "[CID:{{cid}}] Analyze debt scenario:"
      arguments:
        - name: "scenario"
          description: "Debt scenario description"
          required: true
```

When an MCP client invokes `prompts/get`, the gateway returns the prompt definition. When the prompt is used in a conversation, the resulting tool call follows the standard translation flow (Section 4.2.2).

### 4.5 MCP Server to HERMES Namespace

Each MCP server instance maps to a HERMES namespace behind the gateway:

| MCP Server Aspect | HERMES Equivalent |
|---|---|
| Server identity | Namespace identifier (ARC-0791) |
| Server capabilities (tools, resources, prompts) | Agent capabilities within the namespace (ARC-2606) |
| Server lifecycle (initialize, shutdown) | Namespace session lifecycle (SYN/FIN, ARC-0793) |
| Server notifications | `event` messages from the namespace to the gateway |

The gateway MUST NOT expose multiple HERMES namespaces as a single MCP server. The one-to-one mapping preserves HERMES namespace isolation (ARC-1918).

## 5. Translation Flow Architecture

### 5.1 Inbound Flow (External to HERMES)

```
External Client          Gateway Bridge          Gateway Core          HERMES Bus
       |                      |                      |                     |
  1.   | Protocol request     |                      |                     |
       |--------------------->|                      |                     |
       |                      |                      |                     |
  2.   |               Parse external format         |                     |
       |               Validate schema               |                     |
       |               Extract semantics             |                     |
       |                      |                      |                     |
  3.   |                      | HERMES message       |                     |
       |                      |--------------------->|                     |
       |                      |                      |                     |
  4.   |                      |        Inbound validation (ARC-3022 §8)    |
       |                      |        Identity translation                |
       |                      |        Rate limiting                       |
       |                      |                      |                     |
  5.   |                      |                      | Append to bus       |
       |                      |                      |-------------------->|
       |                      |                      |                     |
```

**Step details:**

1. External client sends a request in its native protocol format (A2A JSON-RPC, MCP JSON-RPC, etc.).
2. The bridge component parses the external format, validates it against the protocol schema, and extracts the semantic intent (query, invoke, notify, update).
3. The bridge constructs a HERMES JSONL message using the mappings defined in Sections 3 and 4.
4. The gateway core applies standard inbound validation: source verification, payload inspection, rate limiting (ARC-3022 Section 8).
5. The validated message is appended to the internal bus.

### 5.2 Outbound Flow (HERMES to External)

```
HERMES Bus          Gateway Core          Gateway Bridge          External Client
     |                    |                      |                       |
1.   | Message for        |                      |                       |
     | bridged agent      |                      |                       |
     |------------------>|                      |                       |
     |                    |                      |                       |
2.   |        Outbound filter (ARC-3022 §7)      |                       |
     |        Identity translation               |                       |
     |                    |                      |                       |
3.   |                    | Filtered message     |                       |
     |                    |--------------------->|                       |
     |                    |                      |                       |
4.   |                    |        Translate to external format           |
     |                    |        Apply protocol-specific envelope       |
     |                    |                      |                       |
5.   |                    |                      | Protocol response     |
     |                    |                      |---------------------->|
     |                    |                      |                       |
```

### 5.3 Reference Architecture: 3GPP SBA Pattern

The translation flow follows the 3GPP Service-Based Architecture (TS 23.501) pattern, where Network Functions (NFs) expose services via standardized HTTP/2 interfaces. In HERMES:

| 3GPP SBA Concept | HERMES Bridge Equivalent |
|---|---|
| NF Service Producer | HERMES agent behind the gateway |
| NF Service Consumer | External protocol client (A2A/MCP) |
| SBI (Service-Based Interface) | Gateway bridge endpoint |
| NRF (NF Repository Function) | Agora directory + gateway profile publisher |
| Service Registration | Gateway publishes Agent Card / MCP server manifest |
| Service Discovery | External client discovers HERMES agent via Agora or protocol-native discovery |

The key insight from 3GPP SBA is that **each internal function is exposed as a service with a standardized interface**, regardless of its internal implementation. The HERMES gateway bridge applies the same principle: internal agents, regardless of their implementation, are exposed as A2A agents or MCP tools through standardized interface translation.

## 6. Unified Operation Semantics

### 6.1 Operation Pattern Taxonomy

Inspired by the BBF TR-369/USP CRUD+Operate+Notify pattern, HERMES defines five universal operation patterns that map consistently across all supported protocols. This taxonomy ensures that the gateway can translate any operation regardless of the source or target protocol.

| Operation Pattern | Intent | HERMES Type | A2A Equivalent | MCP Equivalent |
|---|---|---|---|---|
| **Query** (read state) | Retrieve information without side effects | `request` (with query semantics in payload) | `tasks/get` | `resources/read` |
| **Update** (set state) | Modify existing state | `state` | `tasks/update` (via `tasks/send`) | — (no native equivalent) |
| **Create** (new entity) | Instantiate a new resource or task | `event` (type: create) | `tasks/send` (new task) | — (no native equivalent) |
| **Invoke** (execute) | Trigger an action with expected result | `dispatch` | `tasks/send` (with action payload) | `tools/call` |
| **Notify** (async signal) | Inform without expecting response | `alert` | Streaming SSE event | `notifications/*` |

### 6.2 Operation Pattern Details

#### 6.2.1 Query

A query reads state without modifying it. In HERMES, queries are `request` messages whose payload indicates read intent.

```
HERMES:  {"ts":"2026-03-15","src":"gateway","dst":"finance","type":"request",
          "msg":"[CID:q-001] Query current debt balance","ttl":5,"ack":[]}

A2A:     {"jsonrpc":"2.0","method":"tasks/get","params":{"id":"q-001"},"id":1}

MCP:     {"jsonrpc":"2.0","method":"resources/read","params":{"uri":"hermes://finance/debt-balance"},"id":1}
```

#### 6.2.2 Update

An update modifies existing state. This maps naturally to HERMES `state` messages.

```
HERMES:  {"ts":"2026-03-15","src":"finance","dst":"*","type":"state",
          "msg":"Debt strategy updated: snowball → avalanche method","ttl":7,"ack":[]}

A2A:     (Emitted as SSE event on active task subscription)

MCP:     (Emitted as resources/updated notification)
```

#### 6.2.3 Create

A create operation instantiates a new entity. In HERMES, this is an `event` message that announces creation.

```
HERMES:  {"ts":"2026-03-15","src":"controller","dst":"finance","type":"event",
          "msg":"New analysis task created: Q2 budget projection","ttl":3,"ack":[]}

A2A:     {"jsonrpc":"2.0","method":"tasks/send","params":{"message":{...}},"id":1}

MCP:     (No native create; modeled as tools/call that returns a new resource)
```

#### 6.2.4 Invoke

An invoke triggers an action and expects a result. This is the most common bridge operation.

```
HERMES:  {"ts":"2026-03-15","src":"gateway","dst":"finance","type":"dispatch",
          "msg":"[CID:inv-42] Analyze debt restructuring for scenario-7","ttl":5,"ack":[]}

A2A:     {"jsonrpc":"2.0","method":"tasks/send",
          "params":{"message":{"role":"user","parts":[{"text":"Analyze debt..."}]}},"id":1}

MCP:     {"jsonrpc":"2.0","method":"tools/call",
          "params":{"name":"aureus_debt_strategy","arguments":{"scenario":"scenario-7"}},"id":1}
```

#### 6.2.5 Notify

A notify sends information without expecting a response. Fire-and-forget semantics.

```
HERMES:  {"ts":"2026-03-15","src":"finance","dst":"*","type":"alert",
          "msg":"Budget threshold 90% reached. Discretionary freeze recommended","ttl":5,"ack":[]}

A2A:     (SSE event: {"type":"status","state":"working","message":{...}})

MCP:     {"jsonrpc":"2.0","method":"notifications/message","params":{"level":"warning","data":{...}}}
```

### 6.3 Bidirectional Consistency

The gateway MUST ensure that a round-trip translation preserves semantic intent:

```
External Request → HERMES Message → Internal Processing → HERMES Response → External Response
```

At each arrow, the operation pattern (Query, Update, Create, Invoke, Notify) MUST be preserved. A Query in A2A MUST translate to a Query in HERMES and back to a Query response in A2A. The gateway MUST NOT silently change the operation pattern during translation.

## 7. Gateway Bridge Configuration

### 7.1 Bridge Configuration Schema

The gateway bridge is configured as an extension to the gateway configuration (ARC-3022 Section 12):

```yaml
# gateway.yaml — Bridge extension
bridge:
  enabled: true
  protocols:
    a2a:
      enabled: true
      endpoint: "/a2a"           # HTTP path for A2A JSON-RPC
      version: "0.3"             # A2A protocol version (v0.3.0, July 2025)
      streaming: false           # SSE support (requires ARC-6455)
      agent_card_path: "/.well-known/agent.json"  # A2A discovery

    mcp:
      enabled: true
      transport: "streamable-http"  # streamable-http | stdio
      endpoint: "/mcp"           # HTTP path for MCP JSON-RPC (if HTTP)
      tools: []                  # Auto-populated from published capabilities
      resources: []              # Manually configured resource mappings
      prompts: []                # Manually configured prompt templates

  # CID generation
  cid_prefix: "brg"             # Prefix for bridge-generated CIDs
  cid_format: "{prefix}-{protocol}-{timestamp}-{seq}"

  # Response timeout
  response_timeout_seconds: 300  # Max wait for HERMES response before returning error

  # Payload handling
  large_payload_dir: ".bridge/payloads/"  # Directory for payloads exceeding 120 chars
  max_external_payload_bytes: 65536       # Max accepted from external clients
```

### 7.2 Tool/Capability Auto-Registration

When `bridge.protocols.mcp.tools` is empty, the gateway SHOULD auto-generate the MCP tool list from published agent capabilities:

```
FOR EACH agent IN published_agents:
    FOR EACH cap IN agent.capabilities:
        tool_name = sanitize(agent.alias + "_" + cap.specialization)
        tool_description = generate_from_ontology(cap.path)
        tool_input_schema = load_or_generate_schema(agent.alias, cap.path)
        REGISTER tool(tool_name, tool_description, tool_input_schema)
```

The operator MAY override auto-generated tools with explicit definitions in the configuration.

## 8. Error Translation

### 8.1 Error Mapping

Protocol-specific errors MUST be translated to meaningful equivalents:

| HERMES Condition | A2A Error | MCP Error | HTTP Status |
|---|---|---|---|
| Target namespace not found | Task failed: agent not found | Tool not found (`-32601`) | 404 |
| Rate limit exceeded | Task rejected: rate limited | Rate limited (`-32000`) | 429 |
| Payload too large | Task failed: payload exceeded | Invalid params (`-32602`) | 413 |
| TTL expired without resolution | Task failed: timeout | Internal error (`-32603`) | 504 |
| Outbound filter blocked | Task failed: not authorized | Permission denied (`-32000`) | 403 |
| Inbound validation failed | Task rejected: invalid request | Invalid request (`-32600`) | 400 |
| Gateway bridge not configured | Task failed: unsupported | Method not found (`-32601`) | 501 |

### 8.2 Error Response Format

When translating HERMES errors to external protocols, the gateway MUST:

1. Include the operation pattern (Section 6.1) in the error context.
2. Never expose internal namespace names, agent names, or bus contents in error messages.
3. Use the external protocol's native error format (JSON-RPC error object for both A2A and MCP).
4. Log the full error context internally for debugging, including the CID and internal routing details.

## 9. Security Considerations

### 9.1 Gateway as Trust Boundary

The gateway bridge extends ARC-3022's trust boundary to encompass external protocol interactions. All security guarantees from ARC-3022 Section 14 apply:

1. **Identity isolation.** Internal agent names, namespace identifiers, and bus contents MUST NOT be exposed through any bridge endpoint. External clients interact only with published aliases and capabilities.
2. **Default-deny outbound.** The bridge MUST NOT transmit any data that would not pass the standard outbound filter (ARC-3022 Section 7). Bridge responses are subject to the same filtering as Agora responses.
3. **Validated inbound.** All external protocol requests MUST pass through inbound validation (ARC-3022 Section 8) before reaching the internal bus. The bridge adds protocol-specific validation (JSON-RPC schema, tool name resolution) on top of standard validation.

### 9.2 Authentication Upgrade Path

| Phase | Authentication Method | HERMES Spec |
|---|---|---|
| Initial | TOFU (Trust-On-First-Use) | ARC-3022 Section 11.3 |
| Phase 2 | Ed25519 signatures on bridge messages | ARC-8446 (PLANNED) |
| Phase 2+ | JWT bearer tokens for HTTP bridge endpoints | ARC-7519 (PLANNED) |
| Future | OAuth 2.0 with external identity providers | Future spec |

The gateway MUST support TOFU from the initial implementation. When ARC-7519 is implemented, bridge endpoints SHOULD require JWT authentication for all non-discovery requests.

### 9.3 Payload Inspection

External protocol payloads may contain:

- **Executable content.** The bridge MUST reject payloads containing executable code, scripts, or shell commands unless the target capability explicitly accepts them.
- **Oversized data.** The bridge MUST enforce `max_external_payload_bytes` and reject payloads exceeding the limit.
- **Internal references.** The bridge MUST sanitize payloads that reference internal file paths, namespace identifiers, or bus locations.

### 9.4 Rate Limiting

The bridge MUST enforce rate limits independently of the standard gateway rate limits:

| Scope | Recommended Limit |
|---|---|
| Per-client, per-protocol | 60 requests per minute |
| Per-tool invocation | 20 calls per minute |
| Per-resource read | 100 reads per minute |
| Discovery (Agent Card / tool list) | 10 requests per minute |

Rate limits are additive with ARC-3022 inbound rate limits. A request that exceeds either limit MUST be rejected.

### 9.5 Protocol Downgrade Attacks

An attacker MAY attempt to exploit translation gaps between protocols. For example:

- Sending an A2A request that, when translated to HERMES, bypasses namespace firewall rules.
- Crafting an MCP tool call whose arguments, when serialized to a HERMES payload, inject control characters.

The bridge MUST validate the translated HERMES message against ARC-5322 after translation, before appending to the bus. If the translated message fails ARC-5322 validation, the bridge MUST reject the original external request.

## 10. Compatibility

### 10.1 Protocol Version Support

The bridge MUST declare which versions of external protocols it supports. Version negotiation follows each external protocol's native mechanism:

- **A2A**: Version declared in the Agent Card's `version` field.
- **MCP**: Version negotiated during the `initialize` handshake.

### 10.2 Graceful Degradation

When the bridge encounters an external protocol feature that has no HERMES equivalent:

1. The bridge MUST NOT silently drop the feature.
2. The bridge SHOULD return a protocol-appropriate error indicating the feature is unsupported.
3. The bridge MAY log the unsupported feature for future implementation tracking.

Features with no current HERMES equivalent include:

| Feature | Protocol | Workaround |
|---|---|---|
| Streaming responses | A2A (SSE) | Buffer and return final result. Log that streaming was requested. |
| Multimodal content (images, audio) | A2A, MCP | Return error: unsupported media type. Future ARC-7540 may address. |
| Tool annotations (readOnlyHint, destructiveHint) | MCP | Ignore hints; apply HERMES firewall rules (ARC-1918) instead. |
| Dynamic tool discovery | MCP | Serve static tool list from gateway config. Refresh on profile update. |

### 10.3 Backward Compatibility with HERMES

The bridge is an extension. A HERMES deployment that does not implement ARC-7231 remains fully functional — it simply cannot participate in A2A or MCP ecosystems. The bridge is optional, same as the gateway itself (ARC-3022 Section 15.1).

### 10.4 Dependencies

| Specification | Dependency Type | What ARC-7231 Uses |
|---|---|---|
| ARC-3022 | Required | Gateway architecture, outbound filter, inbound validator, identity translation |
| ARC-2606 | Required | Agent Profile schema, Capability Ontology, discovery protocol |
| ARC-5322 | Required | Message format, validation algorithm, message types |
| ARC-0768 | Required | CID/RE correlation for request/response mapping, transport modes |
| ARC-1918 | Required | Namespace isolation model, firewall rules for resource mapping |
| ARC-0791 | Required | Addressing conventions for namespace and agent resolution |
| ARC-6455 | Optional | Real-time streaming bridge (when implemented) |
| ARC-7519 | Optional | JWT authentication for bridge endpoints (when implemented) |
| ARC-8446 | Optional | Cryptographic message signing for bridge messages (when implemented) |

## 11. Examples

### 11.1 A2A Client Invokes HERMES Agent

**Scenario**: An A2A client discovers a HERMES agent via its Agent Card and sends a task.

```
1. A2A Client fetches Agent Card:
   GET https://gateway.example.com/.well-known/agent.json
   Response:
   {
     "name": "aureus",
     "description": "Personal finance agent specializing in debt strategy",
     "url": "https://gateway.example.com/a2a",
     "provider": {"organization": "momosho-d"},
     "version": "0.3.0",
     "capabilities": {"streaming": false, "pushNotifications": true},
     "skills": [
       {"id": "finance/personal/debt-strategy", "name": "Debt Strategy"},
       {"id": "finance/personal/budget", "name": "Budget Optimization"}
     ],
     "authentication": {"schemes": ["none"]},
     "extensions": ["hermes-bridge"]
   }

2. A2A Client sends task:
   POST https://gateway.example.com/a2a
   {
     "jsonrpc": "2.0",
     "method": "tasks/send",
     "params": {
       "id": "task-ext-001",
       "message": {
         "role": "user",
         "parts": [{"text": "Analyze debt scenario: 3 cards, $27M total, min payments $1.2M/month"}]
       }
     },
     "id": 1
   }

3. Gateway translates to HERMES JSONL:
   {"ts":"2026-03-15","src":"gateway","dst":"finance","type":"dispatch",
    "msg":"[CID:brg-a2a-20260315-001] Analyze debt: 3 cards $27M min $1.2M/mo","ttl":5,"ack":[]}

4. Internal agent processes and responds:
   {"ts":"2026-03-15","src":"finance","dst":"gateway","type":"state",
    "msg":"[RE:brg-a2a-20260315-001] Avalanche method saves $2.1M over 18mo","ttl":7,"ack":[]}

5. Gateway translates back to A2A:
   {
     "jsonrpc": "2.0",
     "result": {
       "id": "task-ext-001",
       "status": {"state": "completed"},
       "artifacts": [
         {"parts": [{"text": "Avalanche method saves $2.1M over 18mo"}]}
       ]
     },
     "id": 1
   }
```

### 11.2 MCP Client Calls HERMES Tool

**Scenario**: An MCP client calls a HERMES agent capability exposed as a tool.

```
1. MCP Client initializes:
   {"jsonrpc":"2.0","method":"initialize",
    "params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"test"}},
    "id":1}
   Response: {"jsonrpc":"2.0","result":{"protocolVersion":"2025-03-26",
    "capabilities":{"tools":{},"resources":{}}},"id":1}

2. MCP Client lists tools:
   {"jsonrpc":"2.0","method":"tools/list","id":2}
   Response: {"jsonrpc":"2.0","result":{"tools":[
     {"name":"aureus_debt_strategy","description":"Debt restructuring analysis",
      "inputSchema":{"type":"object","properties":{"scenario":{"type":"string"}},"required":["scenario"]}},
     {"name":"aureus_budget","description":"Budget optimization",
      "inputSchema":{"type":"object","properties":{"period":{"type":"string"}},"required":["period"]}}
   ]},"id":2}

3. MCP Client calls tool:
   {"jsonrpc":"2.0","method":"tools/call",
    "params":{"name":"aureus_debt_strategy","arguments":{"scenario":"3 cards, $27M total"}},
    "id":3}

4. Gateway translates to HERMES:
   {"ts":"2026-03-15","src":"gateway","dst":"finance","type":"dispatch",
    "msg":"[CID:brg-mcp-20260315-001] Debt strategy: 3 cards $27M total","ttl":5,"ack":[]}

5. Agent responds:
   {"ts":"2026-03-15","src":"finance","dst":"gateway","type":"state",
    "msg":"[RE:brg-mcp-20260315-001] Avalanche saves $2.1M/18mo. Ref: .bridge/payloads/brg-mcp-20260315-001.json","ttl":7,"ack":[]}

6. Gateway translates back to MCP:
   {"jsonrpc":"2.0","result":{"content":[
     {"type":"text","text":"Avalanche method recommended. Saves $2.1M over 18 months. See detailed analysis."}
   ],"isError":false},"id":3}
```

### 11.3 Bidirectional Translation Walkthrough

**Scenario**: A HERMES clan wants to call an external A2A agent through the bridge.

```
1. Internal agent creates dispatch to gateway:
   {"ts":"2026-03-15","src":"engineering","dst":"gateway","type":"dispatch",
    "msg":"[CID:out-001] Query external agent alpine/lex-mountain re: zoning law","ttl":5,"ack":[]}

2. Gateway identifies target as A2A-bridged agent (from cached profile):
   - Target: alpine-collective / lex-mountain
   - Protocol: A2A
   - Endpoint: https://alpine.example.com/a2a

3. Gateway translates to A2A:
   POST https://alpine.example.com/a2a
   {
     "jsonrpc": "2.0",
     "method": "tasks/send",
     "params": {
       "id": "out-001",
       "message": {
         "role": "user",
         "parts": [{"text": "Query regarding zoning law compliance"}]
       }
     },
     "id": 1
   }

4. External A2A agent responds.
5. Gateway translates response back to HERMES:
   {"ts":"2026-03-15","src":"gateway","dst":"engineering","type":"state",
    "msg":"[RE:out-001] AGORA:alpine/lex-mountain: Zoning compliant per code 4.2.1","ttl":7,"ack":[]}

Note: The AGORA: prefix (ARC-3022 §13.2) marks this as externally sourced.
```
