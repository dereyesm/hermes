---

> *"In TR-369, the Controller does not micromanage the Agent — it sends an Operate
> command, sets the target, and trusts the Agent to report back. The Controller's
> job is orchestration, not execution. HERMES inherits this discipline: the daemon
> dispatches, the agent executes, the bus records. The operator decides what the
> system cannot."*

## 1. Abstract

This specification defines the **Agent Service Platform (ASP)** for HERMES
deployments. The ASP establishes a unified model for three related problems that
emerge at scale:

1. **Bus Convergence** — reconciling the internal session bus with the clan bus
   into a single append-only log, with the Gateway (ARC-3022) as the boundary
   filter enforcing default-deny on cross-clan traffic.

2. **Agent Registration** — a declarative mechanism by which agents (sensors,
   skill sub-agents, scheduled workers) announce their identity, capabilities,
   dispatch rules, and resource limits to the daemon (ARC-4601).

3. **Dispatch Protocol** — the rules by which the daemon selects an agent,
   delivers a work item, and processes the result — including an operator
   approval gate for dispatches that exceed defined trust thresholds.

The ASP draws directly from BBF TR-369 (USP), which defines how a Controller
manages Agents via a Message Transfer Protocol. The mapping is precise: the
HERMES daemon is the USP Controller, registered agents are USP Agents, the bus
is the MTP, and the gateway is the USP Proxy. This lineage is made explicit in
the reference table in Section 4.

The ASP preserves HERMES Phase 0 compatibility: file-based deployments without
a daemon continue to function. Registration profiles and bus convergence are
optional enhancements layered on top of the existing protocol.

## 2. Motivation

HERMES Phase 0 defined a file-based bus with ephemeral session agents. As clans
grow and agents proliferate, three structural gaps emerge:

### 2.1 The Two-Bus Problem

A clan operating with a persistent daemon accumulates two buses in practice:

- An **internal bus** used by session agents (skills, routers, internal tools)
  to exchange state and coordinate within the clan.
- A **clan bus** that carries messages to and from external destinations.

Without a formal convergence model, implementations diverge: some write
internal messages to a private file, others to a shared file. The boundary
between what stays internal and what crosses the clan perimeter is implicit,
ad-hoc, and therefore unreliable.

TR-369 solved an analogous problem in home networking: the Controller sees all
Objects, but only a defined profile is exposed through the MTP to the ACS.
ARC-0369 applies this same discipline — the bus is unified; the gateway decides
what crosses.

### 2.2 Undeclared Agents

In Phase 0, agents are invoked manually per session. There is no registry, no
declared capability, no resource contract. As autonomous agents emerge (daemons,
scheduled workers, event-driven sensors), the daemon has no way to:

- Discover what agents are available
- Know which bus events each agent handles
- Enforce resource limits without inline configuration

TR-369 defines the USP Agent Object Model — a structured profile that each
Agent registers with the Controller. ARC-0369 adapts this for HERMES: agents
declare a profile file, and the daemon loads it at startup.

### 2.3 Unguarded Dispatch

ARC-4601 defines dispatch slots and timeout guardrails, but treats all
`dispatch`-type messages identically. Some dispatches should execute
autonomously; others require operator confirmation before the daemon spawns a
process. Without a formal approval gate, implementations either block all
autonomous execution (too conservative) or allow all of it (too permissive).

The TR-369 model includes an explicit Subscription and Notification mechanism:
the Controller notifies the operator before acting on certain conditions.
ARC-0369 formalizes this as an **approval gate** — a per-dispatch-rule flag
that routes the work item through operator confirmation before execution.

## 3. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
"SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be
interpreted as described in RFC 2119 / ARC-2119.

| Term | Definition |
|------|-----------|
| **Agent Service Platform (ASP)** | The complete system defined by this spec: unified bus, agent registry, dispatch protocol, and lifecycle model. |
| **Internal Bus** | Messages whose `src` and `dst` both resolve to namespaces within the local clan. These messages MUST NOT cross the clan boundary. |
| **Clan Bus** | The unified JSONL bus file shared by all namespaces in a clan. Contains both internal and outbound messages. The gateway reads the clan bus and filters outbound traffic. |
| **Agent Profile** | A JSON file in the `agents/` directory declaring an agent's identity, capabilities, dispatch rules, and resource limits. Loaded by the daemon at startup. |
| **Dispatch Rule** | A condition defined in an agent profile that triggers autonomous dispatch when matched by an incoming bus message. |
| **Approval Gate** | A flag on a dispatch rule requiring the daemon to write an escalation message and await operator confirmation before executing the dispatch. |
| **Operator** | The human who owns the clan. The operator has final authority. When the approval gate fires, the operator's next-session acknowledgment unblocks the dispatch. |
| **Sensor Agent** | An agent that only reads external sources and writes to the bus. It does not receive dispatch commands. Analogous to a USP Agent with Notify-only profile. |
| **Worker Agent** | An agent that receives `dispatch` messages and executes tasks. Analogous to a USP Agent with Operate profile. |
| **Platform Agent** | An agent that combines Sensor + Worker roles. Full USP Agent analogy. |
| **Registry** | The in-memory map of `agent_id → AgentProfile` loaded by the daemon at startup by scanning the `agents/` directory. |
| **USP** | User Services Platform (BBF TR-369). The telecom standard from which ARC-0369 derives its Controller/Agent model and dispatch semantics. |

### 5.2 Plane Assignment

Per ARC-2314, components are assigned to planes:

| Component | Plane | Rationale |
|-----------|-------|-----------|
| Bus (`bus.jsonl`) | Transport | Message transfer medium |
| Daemon BusObserver | Control | Observes and classifies messages |
| Registry | Control | Loads and resolves agent profiles |
| Dispatch Engine | Orchestration | Selects agent, spawns process |
| Approval Gate | Orchestration | Routes unconfirmed dispatches to operator |
| Gateway | Control | Boundary filter, outbound ACL |
| Registered Agents | User | Execute tasks, write results |

### 5.3 Phase 0 Compatibility

All features defined in this spec are additive. A clan without a daemon:

- Continues to read and write `bus.jsonl` using session-based scripts
- Does not benefit from bus convergence enforcement (internal vs. outbound)
- Does not use the agent registry
- Does not use the dispatch protocol

Phase 0 clans SHOULD treat all messages as internal until a gateway is
deployed. When a daemon is deployed, ARC-0369 MUST be followed.

## 6. Bus Convergence

### 6.1 The Unified Bus Model

ARC-0369 defines a single clan bus. There is exactly ONE `bus.jsonl` file per
clan. All session agents, scheduled workers, and the daemon read from and
write to this file.

The distinction between "internal" and "outbound" is determined at routing
time by the daemon's MessageEvaluator, not by separate file paths.

**Definition — Internal Message**: A message where both `src` and `dst` resolve
to namespaces within the local clan, OR where `dst == "*"` and the intent is
broadcast within the clan. Internal messages MUST NOT be forwarded by the
gateway.

**Definition — Outbound Message**: A message where `dst` resolves to a namespace
outside the local clan. Outbound messages MUST pass through the gateway. The
gateway MUST apply its outbound filter per ARC-3022 (default-deny).

### 6.2 Message Classification

The daemon MUST classify every new message on the bus into exactly one
category:

| Category | Condition | Action |
|----------|-----------|--------|
| `internal` | `dst` resolves to a local namespace, OR `dst == "*"` | Evaluate for dispatch; do NOT forward |
| `outbound` | `dst` resolves to an external clan namespace | Forward through gateway (subject to outbound ACL) |
| `inbound` | Written by gateway after receiving from remote clan | Evaluate for dispatch; do NOT re-forward |
| `expired` | `ttl` exceeded since `ts` | Archive; no action |

### 6.3 Source Integrity

The daemon MUST verify that the `src` field of every message written to the
bus matches a known local namespace or a registered agent ID. Messages with
unknown `src` values SHOULD be logged as warnings and MAY be rejected.

This prevents a compromised process from spoofing a trusted namespace.

### 6.4 Internal-Only Namespaces

Clans MAY designate namespaces as internal-only in `gateway.json`. Messages
from internal-only namespaces MUST be classified as `internal` regardless of
their `dst` value. The gateway MUST drop any message whose `src` is in the
internal-only list.

Example internal-only namespaces: orchestration routers, local skill agents,
dispatch controllers. These namespaces produce operational state that is
meaningful inside the clan but irrelevant (and potentially sensitive) to
external clans.

```json
{
  "clan_id": "example-clan",
  "internal_namespaces": ["router", "dojo", "skill-a", "skill-b"]
}
```

### 6.5 Convergence Invariant

At any point in time, the bus MUST satisfy:

1. Every message is on exactly one bus file (no shadowing across files).
2. Every internal message has been ACKed by its `dst` namespace, OR its TTL
   has not yet expired.
3. No outbound message is forwarded more than once (the gateway ACKs with its
   own namespace tag after forwarding).

## 7. Agent Registration

### 7.1 Registry Directory

The agent registry lives in the `agents/` subdirectory of the clan directory.
Each registered agent MUST have exactly one profile file:

```
{clan_dir}/
└── agents/
    ├── sensor-alpha.json
    ├── worker-beta.json
    └── platform-gamma.json
```

The daemon MUST scan this directory at startup and load all `*.json` files as
agent profiles. The daemon SHOULD watch the directory for changes and reload
profiles without restart (hot reload). File names MUST match the `agent_id`
field in the profile.

### 7.2 Agent Profile Schema

```json
{
  "agent_id": "string",
  "display_name": "string",
  "version": "semver string",
  "role": "sensor | worker | platform",
  "description": "string",
  "capabilities": ["string"],
  "dispatch_rules": [
    {
      "rule_id": "string",
      "trigger": {
        "type": "event-driven | scheduled",
        "match_type": "string | null",
        "match_src": "string | null",
        "match_msg_prefix": "string | null",
        "cron": "string | null"
      },
      "command_template": "string | null",
      "approval_required": "boolean",
      "approval_timeout_hours": "integer"
    }
  ],
  "resource_limits": {
    "max_turns": "integer",
    "timeout_seconds": "integer",
    "allowed_tools": ["string"],
    "max_concurrent": "integer"
  },
  "enabled": "boolean"
}
```

### 7.3 Field Specification

#### 7.3.1 Identity Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `agent_id` | string | YES | Unique identifier within the clan. MUST match the file name without extension. Pattern: `[a-z0-9][a-z0-9-]*` |
| `display_name` | string | YES | Human-readable name for logs and notifications |
| `version` | string | YES | Semantic version of the agent profile |
| `role` | enum | YES | `sensor`: writes to bus only. `worker`: receives dispatch only. `platform`: both. |
| `description` | string | YES | One-sentence purpose statement |
| `capabilities` | array | YES | List of capability identifiers this agent offers (e.g., `["email-scan", "notify-telegram"]`) |
| `enabled` | boolean | YES | If `false`, the daemon MUST skip this agent for all matching |

#### 7.3.2 Dispatch Rules

Each element in `dispatch_rules` defines one trigger → action mapping.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `rule_id` | string | YES | Unique identifier within this agent's rules |
| `trigger.type` | enum | YES | `event-driven`: fire on matching bus message. `scheduled`: fire on cron schedule. |
| `trigger.match_type` | string | NO | Bus message `type` value to match. Required for `event-driven`. |
| `trigger.match_src` | string | NO | Bus message `src` value to match. `null` means any source. |
| `trigger.match_msg_prefix` | string | NO | Prefix match on bus message `msg` field. `null` means any payload. |
| `trigger.cron` | string | NO | Standard 5-field cron expression. Required for `scheduled`. |
| `command_template` | string | NO | Override the default dispatch command. `null` uses the daemon's configured default. |
| `approval_required` | boolean | YES | If `true`, daemon MUST obtain operator approval before executing this dispatch. |
| `approval_timeout_hours` | integer | NO | Hours to wait for approval before escalating. Default: 24. |

#### 7.3.3 Resource Limits

Resource limits are per-dispatch-invocation. They override the daemon-level
defaults from ARC-4601 for this specific agent.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `max_turns` | integer | NO | Maximum agent turns per dispatch invocation. Overrides `dispatch_max_turns`. |
| `timeout_seconds` | integer | NO | Timeout per invocation. Overrides `dispatch_timeout`. |
| `allowed_tools` | array | NO | Tools permitted in this dispatch. Overrides `dispatch_allowed_tools`. |
| `max_concurrent` | integer | NO | Maximum concurrent invocations of this agent. 1 = serial. Default: 1. |

### 7.4 Profile Validation

The daemon MUST validate each loaded profile:

1. `agent_id` matches filename (without `.json` extension)
2. `role` is one of: `sensor`, `worker`, `platform`
3. `dispatch_rules` array is present (may be empty for sensor role)
4. For each rule with `trigger.type == "event-driven"`: `trigger.match_type` MUST be present
5. For each rule with `trigger.type == "scheduled"`: `trigger.cron` MUST be a valid 5-field cron expression
6. `approval_required` is a boolean

Profiles that fail validation MUST be skipped. The daemon MUST log an error
with the profile filename and the specific validation failure. A failed profile
MUST NOT block the daemon from loading other profiles.

### 7.5 Example: Sensor Agent Profile

```json
{
  "agent_id": "mail-scanner",
  "display_name": "Email Scanner",
  "version": "1.0.0",
  "role": "sensor",
  "description": "Scans inbox and writes summaries to bus.",
  "capabilities": ["email-scan", "inbox-summarize"],
  "dispatch_rules": [
    {
      "rule_id": "scheduled-scan",
      "trigger": {
        "type": "scheduled",
        "cron": "0 */4 * * *"
      },
      "approval_required": false
    }
  ],

    "max_turns": 5,

  },

}
```

```json

  "display_name": "Report Builder",

  "role": "worker",
```

## 8. Dispatch Protocol

### 8.1 Overview

The dispatch protocol defines how the daemon selects a registered agent,
optionally gates on operator approval, and delivers a work item. The
protocol maps directly to the TR-369 Controller sending an `Operate` message
to a USP Agent.

### 8.2 Trigger Evaluation

The daemon's MessageEvaluator MUST apply the following logic for each new
internal bus message:

```
for each enabled agent in registry:
  for each rule in agent.dispatch_rules:
    if rule.trigger.type == "event-driven":
      if message.type matches rule.trigger.match_type (if set):
        if message.src matches rule.trigger.match_src (if set):
          if message.msg starts with rule.trigger.match_msg_prefix (if set):
            → CANDIDATE MATCH
    if rule.trigger.type == "scheduled":
      → evaluated by scheduler, not by MessageEvaluator
```

If a message matches rules from multiple agents, the daemon MUST evaluate
each match independently. Multiple agents MAY be dispatched for the same
message, subject to their individual `max_concurrent` limits.

### 8.3 Dispatch Gate

For each candidate match, the daemon applies the dispatch gate:

```
                  ┌─────────────────────────┐
  CANDIDATE  ──►  │  approval_required?     │
                  └────────┬────────────────┘
                           │
               ┌───────────┴──────────────┐
               │ NO                       │ YES
               ▼                          ▼
        DIRECT DISPATCH           APPROVAL GATE
        (Section 8.4)             (Section 8.5)
```

### 8.4 Direct Dispatch

When `approval_required == false`:

1. Check agent's `max_concurrent` limit. If at capacity: enqueue or drop
   (per `queue_overflow` policy, default: `drop-newest`).
2. Render the command: substitute `{{payload}}`, `{{agent_id}}`, `{{rule_id}}`
   in `command_template` (or use daemon default with message payload).
3. Spawn the subprocess in a dispatch slot (ARC-4601 §7).
4. ACK the triggering bus message with the agent's namespace tag.
5. After completion, write result to bus per ARC-4601 §7.4.

### 8.5 Approval Gate

When `approval_required == true`:

1. Write an `event` message to the bus:
   ```
   type: "event"
   src:  daemon namespace
   dst:  "*"
   msg:  "APPROVAL_REQUIRED:<agent_id>:<rule_id>:<original_msg_truncated>"
   ttl:  <approval_timeout_hours / 24, rounded up>
   ```
2. Do NOT execute the dispatch.
3. Monitor the bus for a message matching:
   - `type == "dispatch"`
   - `msg == "APPROVE:<agent_id>:<rule_id>:<original_ts>"`
   — This is the operator's approval signal.
4. On approval received: execute Direct Dispatch (Section 8.4).
5. On timeout (approval_timeout_hours exceeded without approval):
   - Write an `alert` message: `APPROVAL_TIMEOUT:<agent_id>:<rule_id>`
   - Drop the pending dispatch.

The operator SHOULD write the approval message in their next session. No
automated process may write this approval — it requires a human action.

### 8.6 Scheduled Dispatch

For rules with `trigger.type == "scheduled"`, the daemon MUST:

1. Parse the `trigger.cron` expression at profile load time.
2. Register the rule in an internal scheduler.
3. When the cron fires: evaluate the same approval gate as Section 8.3.
4. The scheduler fires even if no bus message triggered it. The payload
   delivered to the agent is a synthetic `dispatch` message:
   ```
   type: "dispatch"
   src:  daemon namespace
   dst:  <agent_id>
   msg:  "SCHEDULED:<rule_id>:<cron_ts>"
   ```

### 8.7 Dispatch Result

After any dispatch completes (direct or approval-gated):

| Outcome | Bus Action |
|---------|-----------|
| Exit 0, output captured | Write `event`: `[RE:<rule_id>] <output_summary>` |
| Exit non-zero | Write `alert`: `DISPATCH_FAILED:<agent_id>:<rule_id>:<exit_code>` |
| Timeout | Write `alert`: `DISPATCH_TIMEOUT:<agent_id>:<rule_id>` |
| Approval timeout | Write `alert`: `APPROVAL_TIMEOUT:<agent_id>:<rule_id>` |
| Capacity exceeded | Write `alert`: `DISPATCH_DROPPED:<agent_id>:<rule_id>:CAPACITY` |

### 8.8 Priority and Ordering

Within a single evaluation cycle, the daemon MUST process dispatch candidates
in this priority order:

1. `alert`-type triggers (highest — time-sensitive)
2. `dispatch`-type triggers (normal)
3. `event`-type triggers (lower)
4. Scheduled triggers (background)

Within the same priority tier, FIFO ordering by message `ts` MUST be used.

## 9. Agent Lifecycle

### 9.1 State Machine

Each registered agent instance (a specific invocation) transitions through
the following states:

```
  ┌──────────┐  profile load   ┌──────────┐  trigger match   ┌───────────┐
  │ INACTIVE ├────────────────►│  ACTIVE  ├─────────────────►│ PENDING   │
  └──────────┘                 └──────────┘                  │ (gate)    │
                                     ▲                        └─────┬─────┘
                                     │ complete/fail                │
                                     │                        approve/auto
                                     │                              │
                               ┌─────┴─────┐   spawn         ┌─────▼─────┐
                               │   IDLE    │◄───────────────  │ RUNNING   │
                               └─────┬─────┘                  └─────┬─────┘
                                     │                              │
                              disable/unload                   timeout/error
                                     │                              │
                               ┌─────▼─────┐                  ┌─────▼─────┐
                               │  REMOVED  │                  │  FAILED   │
                               └───────────┘                  └───────────┘
```

| State | Description | Analogous To |
|-------|-------------|-------------|
| `INACTIVE` | Profile exists on disk but not loaded | USP Agent powered off |
| `ACTIVE` | Profile loaded, ready to match triggers | USP Agent registered, awaiting subscription |
| `PENDING` | Trigger matched; awaiting approval gate | USP Controller received Notification; awaiting Operate decision |
| `RUNNING` | Subprocess spawned; executing | USP Agent executing Operate command |
| `IDLE` | Execution complete; awaiting next trigger | USP Agent idle after Inform |
| `FAILED` | Execution terminated with error or timeout | USP Agent error state |
| `REMOVED` | Profile disabled or deleted from `agents/` | USP Agent de-registered |

The lifecycle mapping to 3GPP TS 23.501 NF lifecycle:

| HERMES State | 3GPP NF Lifecycle State |
|--------------|------------------------|
| `INACTIVE` | NF Registered (but `nfStatus = SUSPENDED`) |
| `ACTIVE` | NF Registered (`nfStatus = REGISTERED`) |
| `RUNNING` | NF Processing service request |
| `FAILED` | NF `nfStatus = UNDISCOVERABLE` |
| `REMOVED` | NF De-registered |

### 9.2 Heartbeat per Agent

The daemon SHOULD include per-agent state in its gateway heartbeat payload
(ARC-4601 §6.3):

```json
{
  "node_id": "string",
  "ts": "ISO-8601 datetime",
  "agents": [
    {
      "agent_id": "mail-scanner",
      "state": "IDLE",
      "last_dispatch": "ISO-8601 datetime",
      "dispatch_count": 42,
      "failure_count": 1
    }
  ]
}
```

This allows the remote gateway and operator tooling to observe per-agent health
without parsing the bus.

### 9.3 Hot Reload

When a profile file changes in `agents/`:

- **New file**: Load profile, transition agent to `ACTIVE`.
- **Modified file**: Reload profile. If an invocation is `RUNNING`, wait for
  completion before applying new limits. Update triggers immediately.
- **Deleted file**: Transition agent to `REMOVED`. Cancel any `PENDING`
  approvals. If `RUNNING`: allow current invocation to complete, then remove.

## 10. Notification Flow

### 10.1 Bus-to-Operator Notification

When the daemon writes an `alert` message to the bus, it SHOULD also issue a
desktop notification (platform-native) to surface the alert to the operator
in real time. This is the HERMES analog to a TR-369 Notification reaching a
management system that renders a UI alert.

Notification-eligible message types:

| Message Type | Condition | Priority |
|---|---|---|
| `alert` | Any alert written to bus | High |
| `event` with `APPROVAL_REQUIRED:` prefix | Approval gate fired | High |
| `event` with `DISPATCH_FAILED:` or `DISPATCH_TIMEOUT:` | Execution failure | Medium |
| `dispatch` from external clan (inbound via gateway) | External work request | Medium |
| `state` with TTL = 1 (urgent state change) | Urgent state | Low |

The daemon MUST NOT issue desktop notifications for internal `state` messages
with normal TTL, `data_cross` messages, or `event` messages with `[RE:]`
prefix (these are dispatch results, not actionable items).

### 10.2 Notification Payload

Desktop notifications SHOULD include:

- **Title**: `HERMES — <type>` (e.g., `HERMES — ALERT`)
- **Body**: First 120 characters of `msg` field
- **Metadata**: `src` namespace and timestamp

### 10.3 Platform Implementations

| Platform | Mechanism | Notes |
|----------|-----------|-------|
| macOS | `osascript -e 'display notification ...'` | Available without external deps |
| Linux | `notify-send` | Requires `libnotify` |
| Headless / CI | Log to stderr | No display available |

The daemon MUST NOT fail if notification delivery fails. Notification is
best-effort. Failures MUST be logged at WARN level.

### 10.4 Notification Throttling

The daemon MUST NOT issue more than 5 notifications per 60-second window per
source namespace. When the threshold is exceeded, remaining notifications in
that window are suppressed and a single batch summary is issued at the end of
the window: `N notifications suppressed from <src> — see bus`.

### 10.5 Notification vs. Bus

Desktop notifications are a convenience layer only. The bus is the authoritative
record. An operator who dismisses or misses a notification MUST still be able
to recover the full context from `bus.jsonl`. No information may appear in a
notification that is not also present on the bus.

## 11. Security Considerations

### 11.1 Threat Model

| ID | Threat | Mitigation |
|----|--------|-----------|
| T1 | Rogue agent profile injection | Profile files are loaded from a controlled directory; file-system permissions restrict write access |
| T2 | Approval spoofing | Approval messages require `type=dispatch` and matching `agent_id:rule_id:ts` — cannot be pre-generated without knowing the trigger timestamp |
| T3 | Cross-namespace message leak | Internal namespace list in `gateway.json` enforced by daemon; gateway drops messages from internal-only `src` |
| T4 | Dispatch command injection | `command_template` fields MUST NOT be interpolated via shell; MUST use `subprocess.run(list_form)` |
| T5 | Notification flooding (DoS) | Throttling per Section 10.4 |
| T6 | Scheduled trigger abuse | Cron expressions validated at load time; maximum frequency SHOULD be configurable (default: no faster than every 5 minutes) |
| T7 | Resource exhaustion via dispatch | Per-agent `max_concurrent` and `timeout_seconds` enforced by daemon |
| T8 | Profile hot-reload race condition | Running invocations complete before new limits apply (Section 9.3) |

### 11.2 Minimum Frequency Limit

Scheduled dispatch rules MUST NOT define a cron expression that fires more
frequently than once every 5 minutes (i.e., intervals shorter than `*/5` on
the minute field are PROHIBITED). This prevents the scheduler from becoming a
high-frequency process spawner.

### 11.3 Approval Gate Integrity

The daemon MUST store pending approval requests in its state file
(`agent-node.state.json`) with the original triggering message's `ts` and a
hash of its `msg` field. When an operator writes an approval, the daemon MUST
verify that the `<original_ts>` in the approval message matches a stored
pending request.

### 11.4 Profile Confidentiality

Agent profiles are stored as plaintext JSON. Profiles MUST NOT contain secrets,
credentials, API keys, or authentication tokens. If a dispatch command requires
credentials, they MUST be injected via environment variables at dispatch time,
not stored in the profile.

### 11.5 Internal Namespace Enforcement

The daemon MUST enforce the `internal_namespaces` list from `gateway.json`
BEFORE the message reaches the gateway. This is defense-in-depth: the daemon
drops the forwarding request, and the gateway applies its own ACL as a second
layer.

## 12. Migration Path

### 12.1 From Phase 0 (No Daemon)

A clan operating in Phase 0 MUST take no action to comply with this spec.
All additions are backward compatible:

1. No `agents/` directory: daemon starts without registered agents. All
   dispatch rules are empty; the daemon falls back to ARC-4601 behavior.
2. No `internal_namespaces` key in `gateway.json`: all namespaces are subject
   to the gateway's standard outbound ACL.
3. No approval gate keys in existing scripts: Phase 0 scripts do not use the
   approval flow; the feature is transparent.

### 12.2 Incremental Adoption

Recommended migration order:

1. **Step 1 — Bus Convergence**: Confirm all session agents write to a single
   `bus.jsonl`. Remove any shadow files. Add `internal_namespaces` list to
   `gateway.json`.

2. **Step 2 — Agent Registration**: For each agent currently invoked via cron,
   create a profile in `agents/`. Start daemon with `hermes daemon start`.
   Verify cron-based scripts produce the same results.

3. **Step 3 — Approval Gates**: Identify any dispatch rules that touch sensitive
   operations (financial data, external notifications, file writes). Set
   `approval_required: true` on those rules.

4. **Step 4 — Notification Flow**: Verify that `alert` messages surface as
   desktop notifications. Tune throttle thresholds.

### 12.3 Co-existence of Cron and Daemon

During migration, cron scripts and the daemon MAY coexist for the same agent.
To prevent double-execution:

- Rename cron-managed agents to a staging `agent_id` prefix (e.g., `cron-mail-scanner`)
- Once daemon dispatch is verified stable, disable the cron entry
- Remove the staging prefix from the profile

The daemon's offset-based bus observation (ARC-4601 §5.2) ensures that messages
written by cron scripts are still processed and ACKed correctly.

## 13. Configuration Reference

ARC-0369 extends the `gateway.json` configuration with one new section:

```json
{
  "clan_id": "example-clan",
  "internal_namespaces": ["router", "dojo", "skill-a"],
  "asp": {
    "agents_dir": "agents",
    "hot_reload": true,
    "notification_enabled": true,
    "notification_throttle_per_minute": 5,
    "min_schedule_interval_minutes": 5,
    "approval_default_timeout_hours": 24,
    "queue_overflow": "drop-newest"
  }
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `agents_dir` | string | `"agents"` | Directory (relative to clan dir) containing agent profiles |
| `hot_reload` | bool | `true` | Whether to watch `agents_dir` for profile changes |
| `notification_enabled` | bool | `true` | Whether to issue desktop notifications |
| `notification_throttle_per_minute` | int | `5` | Max notifications per source per 60s |
| `min_schedule_interval_minutes` | int | `5` | Minimum cron interval enforcement |
| `approval_default_timeout_hours` | int | `24` | Default approval timeout if not set in rule |
| `queue_overflow` | enum | `"drop-newest"` | Behavior when `max_concurrent` exceeded: `drop-newest`, `drop-oldest`, `queue` |
