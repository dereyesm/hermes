> *"RFC 1122 didn't invent TCP/IP — it made sure everyone implemented it the same way.
> Without conformance, you don't have a protocol — you have a collection of similar programs."*

## 1. Abstract

This specification defines **three conformance levels** for HERMES implementations.
Each level builds on the previous, establishing a clear path from minimal bus
interoperability to full multi-clan network participation.

The three levels are:

1. **Bus-Compatible** — Can read and write the Amaru bus correctly.
2. **Clan-Ready** — Can operate as a full clan member with sessions, namespaces, and gateways.
3. **Network-Ready** — Can participate in multi-clan networks with cryptography, integrity, and hub connectivity.

This specification aggregates and organizes normative statements from across the
Amaru spec suite into a single conformance reference. A raw scan of 17 IMPL
specs identified 389 keyword occurrences (264 MUST, 81 SHOULD, 44 MAY). This
specification curates **98 normative statements** (L1: 26, L2: 33, L3: 39) that
represent the essential conformance surface — deduplicating overlaps, merging
co-dependent requirements, and excluding editorial uses of RFC 2119 keywords.
It does not introduce new requirements — it classifies existing ones by
conformance level.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHOULD", "SHOULD NOT",
"RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as
described in ARC-2119.

- **Implementation**: A software agent, adapter, daemon, or tool that reads or
  writes the Amaru bus.
- **Conformance level**: A named tier of requirements. An implementation claims
  a level by satisfying all MUST requirements at that level and below.
- **Normative statement**: A requirement expressed with RFC 2119 keywords.

## 3. Conformance Model

### 3.1 Level Structure

```
┌─────────────────────────────────────────┐
│         Level 3: Network-Ready          │
│   Crypto · Integrity · Hub · Bridge     │
├─────────────────────────────────────────┤
│          Level 2: Clan-Ready            │
│  Sessions · Namespaces · Gateway · ASP  │
├─────────────────────────────────────────┤
│        Level 1: Bus-Compatible          │
│   Message Format · Read/Write · ACK     │
└─────────────────────────────────────────┘
```

### 3.2 Level Inheritance

Each level includes all requirements from the levels below it. A Level 3
implementation MUST satisfy all Level 1 and Level 2 requirements.

### 3.3 Claiming Conformance

An implementation MAY claim conformance at a specific level by including the
following in its documentation or agent profile (ARC-2606):

```
hermes_conformance: "bus-compatible"  | "clan-ready" | "network-ready"
```

An implementation MUST NOT claim a level unless it satisfies all MUST
requirements at that level and all levels below.

## 4. Level 1: Bus-Compatible

**Source specs**: ARC-5322 (Message Format), ARC-0768 (Datagram Semantics), ARC-2119 (Keywords)

Bus-Compatible is the minimum bar for interoperability. An implementation at this
level can read and write messages on the Amaru bus and be understood by any
other HERMES implementation.

### 4.1 Message Format (ARC-5322)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L1-01 | Messages MUST be encoded as UTF-8 JSON. Each message MUST occupy exactly one line (no embedded newlines). | MUST |
| L1-02 | The bus file MUST use `\n` (LF, U+000A) as the line terminator. | MUST |
| L1-03 | A conforming message MUST contain exactly 7 fields: `ts`, `src`, `dst`, `type`, `msg`, `ttl`, `ack`. | MUST |
| L1-04 | The `ts` field MUST be an ISO 8601 date string (`YYYY-MM-DD`). | MUST |
| L1-05 | The `src` field MUST be a non-empty string identifying the sender namespace. | MUST |
| L1-06 | The `dst` field MUST be either a namespace identifier or `"*"` for broadcast. | MUST |
| L1-07 | A message MUST NOT have `dst` equal to `src`. | MUST |
| L1-08 | The `type` field MUST be one of the defined message types: `state`, `event`, `alert`, `dispatch`, `request`, `data_cross`, `dojo_event`. | MUST |
| L1-09 | The `msg` field MUST be a string. It SHOULD NOT exceed 120 characters for signaling messages. | MUST/SHOULD |
| L1-10 | The `ttl` field MUST be a positive integer representing days until expiry. | MUST |
| L1-11 | The `ack` field MUST be a JSON array of namespace identifiers that have processed the message. | MUST |
| L1-12 | Messages MUST NOT contain credentials, secrets, API keys, tokens, or passwords. | MUST |
| L1-13 | Additional fields beyond the 7 specified MUST NOT be present in the base format. | MUST |
| L1-14 | An implementation SHOULD support reading messages with an optional `seq` field (ARC-9001 extension). | SHOULD |
| L1-15 | An implementation SHOULD support reading messages with an optional `encoding` field for compact wire format (ARC-5322 §14). | SHOULD |
| L1-16 | An implementation MAY support the compact binary wire format (ARC-5322 §14). | MAY |

### 4.2 Bus Operations

| ID | Requirement | Keyword |
|----|-------------|---------|
| L1-17 | The bus file MUST be named `bus.jsonl` and located in the clan directory root. | MUST |
| L1-18 | An implementation MUST append messages atomically (complete line or nothing). | MUST |
| L1-19 | An implementation MUST handle malformed lines gracefully (skip, do not crash). | MUST |
| L1-20 | When processing a message, an implementation MUST append its namespace to the `ack` array. | MUST |
| L1-21 | A namespace MUST NOT appear more than once in the `ack` array. | MUST |
| L1-22 | An implementation MUST NOT modify or delete existing bus lines (append-only). | MUST |
| L1-23 | An implementation SHOULD read the bus from its last known offset, not from the beginning. | SHOULD |
| L1-24 | An implementation MAY implement bus archival by moving expired messages (TTL exceeded) to `bus-archive.jsonl`. | MAY |

### 4.3 Transport Modes (ARC-0768)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L1-25 | An implementation MUST support the `reliable` transport mode (acknowledgment required). | MUST |
| L1-26 | An implementation MAY support the `datagram` transport mode (fire-and-forget). | MAY |

**Level 1 totals**: 22 MUST, 4 SHOULD, 3 MAY = **29 normative statements**.

## 5. Level 2: Clan-Ready

**Source specs**: ARC-0793 (Reliable Transport), ARC-0791 (Addressing), ARC-1918 (Private Spaces), ARC-2314 (CUPS), ARC-2606 (Agent Profile), ARC-3022 (Gateway), ARC-0369 (ASP)

Clan-Ready implementations can participate in a clan: managing sessions, respecting
namespace boundaries, discovering peers, and routing messages through a gateway.

### 5.1 Session Lifecycle (ARC-0793)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L2-01 | Every session MUST execute the SYN protocol at start. | MUST |
| L2-02 | Every session MUST execute the FIN protocol at close, even if no state changes occurred. | MUST |
| L2-03 | An agent MUST NOT perform work in a namespace without first executing SYN. | MUST |
| L2-04 | SYN MUST read the bus, filter messages for the namespace, and report pending items. | MUST |
| L2-05 | FIN MUST write state changes to the bus, update the SYNC HEADER, and ACK consumed messages. | MUST |
| L2-06 | All FIN operations MUST complete as a logical unit (atomic finalization). | MUST |
| L2-07 | A namespace MUST NOT have concurrent active sessions writing to the same bus. | MUST |
| L2-08 | An implementation SHOULD track session duration and write a session log entry at FIN. | SHOULD |

### 5.2 Namespace Isolation (ARC-1918)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L2-09 | Each namespace MUST have its own private space (configuration, skills, memory). | MUST |
| L2-10 | An implementation MUST define a permission table specifying allowed data crossings. | MUST |
| L2-11 | A `data_cross` message MUST NOT carry credentials, tokens, or tool configurations. | MUST |
| L2-12 | Authentication credentials, session state, and memory files MUST NEVER cross namespace boundaries. | MUST |
| L2-13 | An implementation MUST enforce that namespaces cannot access each other's private spaces. | MUST |
| L2-14 | An implementation SHOULD log all cross-namespace data transfers for auditability. | SHOULD |

### 5.3 Addressing & Routing (ARC-0791)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L2-15 | Each agent MUST have a unique namespace identifier within its clan. | MUST |
| L2-16 | Namespace identifiers MUST be lowercase alphanumeric strings (may include hyphens). | MUST |
| L2-17 | Routing MUST be determined by the `dst` field: exact match for unicast, `"*"` for broadcast. | MUST |

### 5.4 Gateway (ARC-3022)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L2-18 | A clan MUST have exactly one gateway for external communication. | MUST |
| L2-19 | Every external identity MUST map to exactly one internal identity (no aliasing). | MUST |
| L2-20 | External aliases MUST NOT reveal internal names, namespaces, or clan structure. | MUST |
| L2-21 | Internal topology, bus messages, and metrics MUST NOT leak to external clans. | MUST |
| L2-22 | The gateway MUST apply a default-deny outbound filter. | MUST |
| L2-23 | Outbound messages MUST pass through the gateway filter before leaving the clan. | MUST |
| L2-24 | Internal messages MUST NOT be forwarded by the gateway. | MUST |
| L2-25 | The gateway SHOULD publish an agent profile (ARC-2606) for external discovery. | SHOULD |
| L2-26 | The profile MUST NOT contain internal namespace names, bus messages, Bounty scores, or Dojo data. | MUST |
| L2-27 | An implementation MAY support multiple peer clans in its gateway configuration. | MAY |

### 5.5 Agent Profile (ARC-2606)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L2-28 | An agent profile SHOULD declare capabilities, supported message types, and protocol version. | SHOULD |
| L2-29 | An implementation MAY publish its profile to the Agora directory for discovery. | MAY |

### 5.6 Agent Service Platform (ARC-0369)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L2-30 | An implementation SHOULD classify bus messages into categories: internal, outbound, inbound, expired. | SHOULD |
| L2-31 | An implementation SHOULD verify that `src` fields match registered namespaces. | SHOULD |
| L2-32 | An implementation MAY implement agent registration with declarative profiles. | MAY |
| L2-33 | An implementation MAY implement dispatch rules for automated message handling. | MAY |

**Level 2 totals**: 23 MUST, 5 SHOULD, 4 MAY = **32 normative statements**.

## 6. Level 3: Network-Ready

**Source specs**: ARC-8446 (Encrypted Bus), ARC-9001 (Bus Integrity), ARC-4601 (Agent Node + Hub), ARC-7231 (Bridge)

Network-Ready implementations can participate in multi-clan networks with
end-to-end encryption, bus integrity guarantees, and hub-based routing.

### 6.1 Cryptography (ARC-8446)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L3-01 | An implementation MUST generate two independent key pairs: Ed25519 (signing) and X25519 (key agreement). | MUST |
| L3-02 | The two key pairs MUST be generated independently; one MUST NOT be derived from the other. | MUST |
| L3-03 | Private keys MUST be stored with restricted permissions (`0600` on UNIX). | MUST |
| L3-04 | Private keys MUST NOT be transmitted over any network. | MUST |
| L3-05 | Implementations MUST use HKDF-SHA256 with `info=b"HERMES-ARC8446-v1"` for static key derivation. | MUST |
| L3-06 | Implementations MUST use AES-256-GCM for authenticated encryption. | MUST |
| L3-07 | A unique 12-byte nonce MUST be generated per encryption operation. | MUST |
| L3-08 | An implementation MUST verify the Ed25519 signature before attempting decryption. | MUST |
| L3-09 | If signature verification fails, the implementation MUST NOT attempt decryption. | MUST |
| L3-10 | For inter-clan messages, an implementation MUST use ECDHE with ephemeral X25519 keys (ARC-8446 §11.2). | MUST |
| L3-11 | ECDHE MUST use HKDF-SHA256 with `info=b"HERMES-ARC8446-ECDHE-v1"`. | MUST |
| L3-12 | The ECDHE signature MUST cover `ciphertext \|\| ephemeral_pub` (TLS 1.3 order). | MUST |
| L3-13 | AAD for ECDHE MUST include the ephemeral public key. | MUST |
| L3-14 | An implementation SHOULD support sealed bus messages (static DH) for intra-clan communication. | SHOULD |
| L3-15 | An implementation SHOULD support compact sealed envelopes (ARC-5322 §14 + ARC-8446). | SHOULD |
| L3-16 | An implementation MAY implement a 30-day migration window for crypto parameter changes (ARC-8446 §11.2.9). | MAY |

### 6.2 Bus Integrity (ARC-9001)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L3-17 | An implementation MUST accept messages with or without the `seq` field (backward compatibility). | MUST |
| L3-18 | When writing messages, an implementation SHOULD include a monotonically increasing `seq` field per source. | SHOULD |
| L3-19 | An implementation SHOULD detect sequence gaps and duplicates (F1: SequenceTracker). | SHOULD |
| L3-20 | An implementation SHOULD enforce write ownership — only registered owners write to their namespace (F2: OwnershipRegistry). | SHOULD |
| L3-21 | An implementation MAY implement MVCC write vectors for conflict detection (F3). | MAY |
| L3-22 | An implementation MAY maintain a conflict log in `bus-conflicts.jsonl` (F4). | MAY |
| L3-23 | An implementation MAY implement snapshot-based recovery (F5). | MAY |
| L3-24 | An implementation MAY implement sequence-aware garbage collection (F6). | MAY |
| L3-25 | SequenceTracker and OwnershipRegistry state SHOULD be persisted in the agent node state file. | SHOULD |

### 6.3 Agent Node (ARC-4601)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L3-26 | An implementation SHOULD provide a persistent daemon that observes the bus for changes. | SHOULD |
| L3-27 | The daemon MUST NOT re-read the entire bus on each event; it MUST track file offset. | MUST |
| L3-28 | Heartbeats are transport-layer signals; they MUST NOT be written to the bus. | MUST |
| L3-29 | A single process MUST NOT run both local-daemon and hub-mode simultaneously. | MUST |

### 6.4 Hub Mode (ARC-4601 §15)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L3-30 | The Hub MUST support WebSocket protocol (RFC 6455) over TLS. | MUST |
| L3-31 | The Hub MUST authenticate peers using Ed25519 challenge-response. | MUST |
| L3-32 | The Hub MUST NOT inspect, modify, or decrypt the `msg` field of transiting messages (E2E passthrough). | MUST |
| L3-33 | The Hub MUST support store-and-forward for offline peers with TTL-based eviction. | MUST |
| L3-34 | The Hub SHOULD support broadcast delivery to all connected peers (excluding sender). | SHOULD |
| L3-35 | The Hub MAY provide presence notifications (connect/disconnect events). | MAY |

### 6.5 Bridge Protocol (ARC-7231)

| ID | Requirement | Keyword |
|----|-------------|---------|
| L3-36 | An implementation MAY support translation between HERMES and A2A (Google) messages. | MAY |
| L3-37 | An implementation MAY support translation between HERMES and MCP (Anthropic) messages. | MAY |
| L3-38 | Bridge translations MUST preserve semantic intent (message type mapping). | MUST |
| L3-39 | Bridge translations MUST NOT bypass the gateway filter. | MUST |

**Level 3 totals**: 21 MUST, 7 SHOULD, 7 MAY = **35 normative statements**.

## 8. Conformance Testing

### 8.1 Test Categories

Conformance tests SHOULD be organized by level:

- `tests/conformance/level1/` — Message format validation, bus read/write, ACK semantics
- `tests/conformance/level2/` — Session lifecycle, namespace isolation, gateway filtering
- `tests/conformance/level3/` — Crypto operations, integrity checks, hub protocol

### 8.2 Test Methodology

A conformance test suite SHOULD:

1. Be language-agnostic (YAML input/output pairs where possible).
2. Test both positive cases (valid behavior) and negative cases (rejection of invalid input).
3. Include edge cases from each source spec.
4. Be versioned alongside the spec (test suite version = spec version).

### 8.3 Claiming Conformance

To claim conformance at a given level, an implementation MUST:

1. Pass all MUST tests at that level and all levels below.
2. Document any SHOULD requirements it does not implement, with rationale.
3. Include the conformance level in its agent profile (ARC-2606).

## 9. Security Considerations

- Implementations claiming Level 3 MUST implement all crypto requirements.
  Partial crypto (e.g., signing without encryption) does not satisfy Level 3.
- Implementations MUST NOT claim a higher level than they actually satisfy.
  Overclaiming conformance undermines the trust model.
- The conformance level is informational for routing purposes — a Level 3
  implementation SHOULD prefer peers that also claim Level 3 for sensitive
  communications.
