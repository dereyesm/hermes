## Abstract

This specification defines the namespace addressing scheme, routing table
format, adjacency model, and route resolution algorithm for the Amaru
protocol. It establishes how autonomous agent namespaces are identified,
how messages are routed between them, and what isolation guarantees the
topology provides. The design follows a star topology with a central
controller hub, ensuring that no two namespaces communicate directly and
that credential/tool boundaries are never crossed during routing.

## 1. Introduction

HERMES organizes agents into isolated **namespaces** -- logical domains
that encapsulate configuration, memory, agent definitions, and external
tool access. Each namespace operates independently, with its own head
agent, tool bindings, and account associations. Inter-namespace
communication occurs exclusively through a centralized message bus,
never through direct invocation.

This document specifies:

- The syntax and semantics of namespace addresses
- The structure of routing tables that map namespaces to their resources
- The adjacency model that governs which namespaces can exchange data
- The algorithm by which a message is resolved to its destination
- Mathematical foundations for pathfinding and capacity analysis

### 1.1 Terminology

| Term            | Definition                                                        |
|-----------------|-------------------------------------------------------------------|
| **Namespace**   | An isolated logical domain containing agents, config, and memory  |
| **Head agent**  | The primary agent responsible for a namespace                     |
| **Controller**  | The supervisory namespace with global visibility                  |
| **Bus**         | The append-only message transport (see ARC for transport layer)   |
| **Tool**        | An external service binding (API, integration) scoped to a namespace |
| **Adjacency**   | The set of permitted communication paths between namespaces       |
| **Data crossing**| A permitted transfer of categorized data (never credentials/tools) between namespaces |

### 1.2 Requirements Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
"SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this
document are to be interpreted as described in RFC 2119.

## 2. Namespace Addressing

### 2.1 Address Format

A namespace address is a string identifier conforming to the following
grammar:

```
address     = reserved / namespace-id
reserved    = "*" / "controller"
namespace-id = ALPHA *(ALPHA / DIGIT / "-")
ALPHA       = %x61-7A          ; lowercase a-z
DIGIT       = %x30-39          ; 0-9
```

**Rules:**

1. Addresses MUST be lowercase.
2. Addresses MUST begin with an alphabetic character.
3. Addresses MAY contain hyphens (`-`) but MUST NOT begin or end with a hyphen.
4. Addresses MUST NOT contain whitespace, underscores, dots, or special
   characters other than hyphens.
5. Addresses MUST be between 1 and 63 characters in length.
6. Addresses MUST be unique within an Amaru deployment.

**Valid examples:** `engineering`, `finance`, `community`, `dev-ops`, `q4-planning`

**Invalid examples:** `Engineering` (uppercase), `-ops` (leading hyphen),
`my_namespace` (underscore), `a.b` (dot)

### 2.2 Reserved Addresses

| Address        | Semantics                                                    |
|----------------|--------------------------------------------------------------|
| `*`            | Broadcast -- message is delivered to ALL namespaces          |
| `controller`   | The supervisory namespace with global routing visibility     |

Reserved addresses MUST NOT be used as namespace identifiers in routing
table entries.

### 2.3 Unicast vs. Broadcast

**Unicast:** A message with `dst` set to a specific namespace identifier
is delivered exclusively to that namespace.

```json
{"src": "engineering", "dst": "finance", "type": "data_cross", "msg": "..."}
```

**Broadcast:** A message with `dst` set to `*` is delivered to every
namespace. The originating namespace (`src`) is excluded from delivery
unless it explicitly appears in the consumption logic.

```json
{"src": "finance", "dst": "*", "type": "state", "msg": "..."}
```

Broadcast messages are consumed independently by each namespace. Each
namespace appends its own identifier to the `ack` array upon consumption.
A broadcast message is fully consumed when all active namespaces appear
in `ack`.

### 2.4 Address Resolution Order

When resolving a destination address:

1. Check for reserved address (`*` or `controller`).
2. Look up the address in the routing table (Section 3).
3. If no match, the message is undeliverable. Implementations SHOULD
   log a routing error and MUST NOT silently discard the message.

## 3. Routing Table

### 3.1 File Mapping Table

The routing table maps each namespace to its constituent files. Every
namespace MUST have entries for configuration, memory, and agents.

```
ROUTING TABLE: Namespace -> Files
+------------------+--------------------+--------------------+--------------------+
| Namespace        | Config File        | Memory File        | Agents Directory   |
+==================+====================+====================+====================+
| controller       | /config/ctrl.md    | /memory/ctrl.md    | /agents/ctrl/      |
| engineering      | /config/eng.md     | /memory/eng.md     | /agents/eng/       |
| finance          | /config/fin.md     | /memory/fin.md     | /agents/fin/       |
| operations       | /config/ops.md     | /memory/ops.md     | /agents/ops/       |
| editorial        | /config/edit.md    | /memory/edit.md    | /agents/edit/      |
| community        | /config/comm.md    | /memory/comm.md    | /agents/comm/      |
+------------------+--------------------+--------------------+--------------------+
```

**Format specification:**

| Column           | Type   | Description                                     |
|------------------|--------|-------------------------------------------------|
| Namespace        | string | A valid namespace address (Section 2.1)         |
| Config File      | path   | Absolute path to the namespace's configuration  |
| Memory File      | path   | Absolute path to the namespace's persistent memory |
| Agents Directory | path   | Absolute path to the directory containing agent definitions |

All paths MUST be absolute. Implementations MUST verify that referenced
files and directories exist at startup.

### 3.2 Agent and Tool Binding Table

Each namespace declares a head agent, the external tools available to
agents in that namespace, and the account under which those tools
operate.

```
ROUTING TABLE: Namespace -> Head Agent -> Tools -> Account
+------------------+------------------+----------------------+-------------------+
| Namespace        | Head Agent       | Tools                | Account           |
+==================+==================+======================+===================+
| controller       | council          | (none)               | (none)            |
| engineering      | eng-lead         | project-mgmt, vcs    | eng@example.com   |
| finance          | fin-lead         | spreadsheet, email   | fin@example.com   |
| operations       | ops-lead         | email                | ops@example.com   |
| editorial        | editor-in-chief  | email                | ed@example.com    |
| community        | comm-lead        | email                | comm@example.com  |
+------------------+------------------+----------------------+-------------------+
```

**Constraints:**

1. A namespace MAY have zero tools (e.g., the controller namespace
   operates without external tool access).
2. Tools are scoped exclusively to their namespace. A tool binding in
   namespace A MUST NOT be invoked by agents in namespace B.
3. Account identifiers are opaque strings. They MUST NOT appear in
   bus messages or cross namespace boundaries.
4. A namespace MAY have no head agent (indicated by `(none)`), in which
   case its agents operate without hierarchical coordination.

### 3.3 Cross-Namespace Data Permissions

The routing table includes an explicit permissions matrix specifying
which data crossings are allowed. Only **data** may cross namespace
boundaries. Credentials, tool bindings, and account identifiers MUST
NEVER cross.

```
CROSS-NAMESPACE DATA PERMISSIONS
+------------------+------------------+-------------------+------------------------+
| Source           | Destination      | Message Type      | Data Category          |
+==================+==================+===================+========================+
| engineering      | finance          | data_cross        | project-expenses       |
| operations       | finance          | data_cross        | operational-costs      |
| editorial        | finance          | data_cross        | subscription-fees      |
| finance          | *                | state             | financial-health       |
| community        | *                | event             | scheduled-events       |
+------------------+------------------+-------------------+------------------------+
```

**Rules:**

1. A data crossing not listed in the permissions table MUST be rejected.
2. Broadcast permissions (destination `*`) allow the source to emit to
   all namespaces but do not grant access to other namespaces' data.
3. The `data_cross` type MUST only carry categorized, non-sensitive data.
   Credentials, tokens, and tool configurations are explicitly excluded.
4. Projections (estimated or forecasted data) MUST be prefixed with
   `PROJ:` in the message payload to distinguish them from verified data.
5. Messages carrying monetary or quantitative data SHOULD include a
   source annotation using the `@source` suffix convention.

## 4. Adjacency Model

### 4.1 Star Topology

HERMES employs a **star topology** with the controller at the hub.

```
                         +--------------+
                         |  controller  |
                         |    (hub)     |
                         +------+-------+
                                |
                 +---------+----+----+---------+
                 |         |         |         |
           +-----+--+ +---+----+ +--+-----+ +-+--------+
           |  eng   | |  fin   | |  ops   | |community |
           +--------+ +--------+ +--------+ +----------+
                                    |
                              +-----+----+
                              |editorial |
                              +----------+
```

In the star topology:

- The **controller** has adjacency to ALL namespaces.
- Each **namespace** has adjacency only to the controller.
- There is NO direct adjacency between non-controller namespaces.

### 4.2 Communication Rules

1. **No direct namespace-to-namespace communication.** All traffic
   between namespaces transits through the bus, which the controller
   supervises.

2. **The controller has global visibility.** It can read the routing
   table, inspect all bus messages, detect synchronization drift, and
   propose corrective actions.

3. **Namespaces are peers.** No namespace has authority over another
   namespace. The controller coordinates but does not command.

4. **The bus is the sole transport.** There is no side-channel
   communication. Agents MUST NOT read files belonging to other
   namespaces except through the bus protocol.

### 4.3 Adjacency Matrix

The adjacency matrix `A` for a deployment with `n` namespaces is an
`(n+1) x (n+1)` matrix (including the controller) where:

```
A[controller][ns_i] = 1   for all i in 1..n
A[ns_i][controller] = 1   for all i in 1..n
A[ns_i][ns_j]       = 0   for all i,j in 1..n where i != j
A[ns_i][ns_i]       = 0   for all i  (no self-loops)
```

For a deployment with namespaces {engineering, finance, operations,
editorial, community}:

```
              ctrl  eng  fin  ops  edit comm
controller  [  0    1    1    1    1    1  ]
engineering [  1    0    0    0    0    0  ]
finance     [  1    0    0    0    0    0  ]
operations  [  1    0    0    0    0    0  ]
editorial   [  1    0    0    0    0    0  ]
community   [  1    0    0    0    0    0  ]
```

This is a bipartite-like structure: the controller set {controller} is
adjacent to every member of the namespace set {eng, fin, ops, edit,
comm}, and no edges exist within the namespace set.

### 4.4 Logical vs. Physical Adjacency

The star topology describes **logical** adjacency -- the set of
permitted communication paths. At the **physical** layer, all namespaces
read and write to the same bus file. The star constraint is enforced by:

1. The addressing scheme (Section 2) determines who receives a message.
2. The permissions table (Section 3.3) determines what data may cross.
3. The controller's sync function detects violations.

There is no physical enforcement of isolation at the file system level.
Isolation is a protocol-level guarantee, analogous to how IP routing is
enforced by routers rather than by the physical wire.

## 5. Route Resolution Algorithm

### 5.1 Message Dispatch

When an agent in namespace `S` emits a message with destination `D`:

```
FUNCTION resolve(src, dst, msg_type, payload):
    // Step 1: Validate source
    IF src NOT IN routing_table THEN
        RETURN error("unknown source namespace")

    // Step 2: Resolve destination
    IF dst == "*" THEN
        recipients = ALL namespaces IN routing_table WHERE ns != src
        GOTO step_4
    ELSE IF dst == "controller" THEN
        recipients = [controller]
        GOTO step_4
    ELSE IF dst IN routing_table THEN
        recipients = [dst]
    ELSE
        RETURN error("unresolvable destination")

    // Step 3: Check permissions
    IF msg_type == "data_cross" THEN
        IF (src, dst) NOT IN cross_permissions THEN
            RETURN error("data crossing not permitted")

    // Step 4: Enqueue
    message = {
        ts:   current_timestamp(),
        src:  src,
        dst:  dst,
        type: msg_type,
        msg:  payload,
        ttl:  default_ttl(msg_type),
        ack:  []
    }
    APPEND message TO bus
    RETURN success(message)
```

### 5.2 Message Consumption

When a session starts in namespace `N`:

```
FUNCTION consume(namespace):
    pending = []
    FOR EACH message IN bus:
        IF message.dst == namespace OR message.dst == "*" THEN
            IF namespace NOT IN message.ack THEN
                pending.APPEND(message)
    RETURN pending
```

After a message is consumed, the namespace appends its identifier to the
`ack` array.

### 5.3 TTL and Expiration

Each message type has a default TTL:

| Message Type | Default TTL (days) |
|--------------|--------------------|
| `state`      | 7                  |
| `alert`      | 5                  |
| `event`      | 3                  |
| `request`    | 7                  |
| `data_cross` | 7                  |
| `dispatch`   | 3                  |

A message is expired when `current_date - ts > ttl`. Expired messages
are moved to an archive file and removed from the active bus.

### 5.4 Stale Message Detection

During the SYN (session initialization) phase, the consuming namespace
SHOULD check for messages where:

```
current_date - ts > 3 days AND namespace NOT IN ack
```

Such messages indicate a synchronization gap and SHOULD be flagged to
the operator.

## 6. Mathematical Foundations

### 6.1 Graph Model

An Amaru deployment is modeled as a graph `G = (V, E)` where:

- `V = {controller} UNION {ns_1, ns_2, ..., ns_n}` (the set of all
  namespace nodes)
- `E = {(controller, ns_i) | i in 1..n}` (edges from controller to
  each namespace)

Since the graph is undirected for adjacency purposes:

```
|V| = n + 1
|E| = n
```

The graph is a **star graph** `S_n`, which is a special case of a
**tree** (connected, acyclic).

### 6.2 Pathfinding (Dijkstra's Algorithm)

In the star topology, the shortest path between any two non-controller
namespaces `ns_i` and `ns_j` always passes through the controller:

```
path(ns_i, ns_j) = ns_i -> controller -> ns_j
```

Applying **Dijkstra's algorithm** with uniform edge weights `w = 1`:

```
d(controller, ns_i) = 1        for all i
d(ns_i, ns_j)       = 2        for all i != j
d(ns_i, ns_i)       = 0        (trivial)
```

**Diameter** of the graph: `diam(G) = 2`

**Radius** of the graph: `rad(G) = 1` (the controller is the center)

For weighted edges (e.g., priority-based routing), Dijkstra's algorithm
generalizes naturally. Let `w(e)` be the weight of edge `e`:

```
d(ns_i, ns_j) = w(ns_i, controller) + w(controller, ns_j)
```

Since all paths between non-controller nodes have exactly one
intermediate hop, the optimal path is always unique and trivially
computed.

### 6.3 Blocking Probability (Erlang B)

When the controller processes messages sequentially, it acts as a
single-server queue. If the bus has finite capacity `C` (maximum
concurrent unacknowledged messages) and namespaces generate messages at
aggregate rate `lambda` with mean service time `mu^{-1}`, the
**Erlang B formula** gives the probability that a message is blocked
(i.e., the bus is at capacity):

```
            A^C / C!
B(C, A) = ----------------
           sum_{k=0}^{C} A^k / k!
```

Where `A = lambda / mu` is the offered traffic intensity in Erlangs.

**Application to HERMES:**

- `lambda` = average messages emitted per day across all namespaces
- `mu` = average messages consumed (acknowledged) per day
- `C` = maximum bus depth (recommended: 100 messages)

For a deployment with 5 namespaces, each generating ~2 messages/day,
and daily sync clearing the bus:

```
lambda = 10 messages/day
mu     = 10 messages/day  (daily sync clears all)
A      = 1.0 Erlang
C      = 100

B(100, 1.0) ≈ 0  (effectively zero blocking)
```

This confirms that for typical HERMES deployments, bus capacity is not
a concern. The Erlang B model becomes relevant when:

- Namespaces scale beyond ~50
- Message generation rate exceeds consumption rate (sync frequency
  drops)
- Bus capacity is artificially constrained

### 6.4 Broadcast Complexity

A broadcast message requires `n` acknowledgments (one per namespace).
Full broadcast consumption is `O(n)` in the number of namespaces. The
controller's sync operation is `O(m * n)` where `m` is the number of
active bus messages and `n` is the number of namespaces (checking each
message against each namespace's ack status).

### 8.3 Route Resolution Walkthrough

Given the routing table in Section 8.1, resolve a message from
`engineering` to `finance` of type `data_cross`:

1. **Validate source:** `engineering` exists in routing table. PASS.
2. **Resolve destination:** `finance` is not reserved, exists in routing
   table. `recipients = [finance]`.
3. **Check permissions:** `(engineering, finance)` with type `data_cross`
   exists in `cross_permissions`. PASS.
4. **Enqueue:** Message appended to bus with TTL 7.
5. **Result:** SUCCESS.

Now resolve a message from `community` to `engineering` of type
`data_cross`:

1. **Validate source:** `community` exists. PASS.
2. **Resolve destination:** `engineering` exists. `recipients = [engineering]`.
3. **Check permissions:** `(community, engineering)` with type
   `data_cross` does NOT exist in `cross_permissions`. FAIL.
4. **Result:** ERROR -- "data crossing not permitted".

## 9. Security Considerations

### 9.1 Credential Isolation

The routing table binds tools and accounts to namespaces. This binding
is a security boundary:

- Tool bindings MUST NOT be shared across namespaces.
- Account identifiers MUST NOT appear in bus messages.
- Cross-namespace communication is limited to categorized data via the
  bus; no namespace can invoke another namespace's tools.

### 9.2 Bus Integrity

The bus file is a shared resource. Implementations SHOULD consider:

- **Write validation:** Before appending to the bus, validate that the
  source namespace matches the active session's namespace.
- **Tamper detection:** Implementations MAY compute checksums over bus
  entries to detect unauthorized modification.
- **Access control:** At the file system level, the bus file SHOULD be
  readable by all namespaces and writable only by the active session.

### 9.3 Broadcast Safety

Broadcast messages (`dst: "*"`) are delivered to all namespaces. Senders
MUST ensure that broadcast payloads contain no namespace-specific
sensitive data. The `state` and `event` types are appropriate for
broadcast; `data_cross` SHOULD use unicast with explicit permissions.

### 9.4 Data Crossing Restrictions

The cross-namespace permissions table (Section 3.3) acts as a firewall:

- Only explicitly listed `(source, destination, type)` tuples are
  permitted.
- Default policy is **deny all** -- if a crossing is not listed, it is
  rejected.
- Projections MUST be marked with the `PROJ:` prefix to prevent
  unverified data from being treated as factual.

### 9.5 Controller Trust Model

The controller has global visibility but operates under the same
protocol constraints as other namespaces. It:

- CAN read all bus messages and routing tables.
- CAN detect drift and propose corrections.
- CANNOT execute actions without operator approval.
- CANNOT invoke tools belonging to other namespaces.

The controller is a coordinator, not an executor. The human operator
remains the ultimate authority.

## Appendix A: Adjacency Matrix Generator

For a deployment with `n` namespaces, the adjacency matrix can be
generated as:

```
FUNCTION generate_adjacency(namespaces):
    n = |namespaces|
    A = zero_matrix(n+1, n+1)   // index 0 = controller

    FOR i = 1 TO n:
        A[0][i] = 1              // controller -> ns_i
        A[i][0] = 1              // ns_i -> controller

    RETURN A
```

The resulting matrix always has:
- Row 0 and column 0 filled with 1s (except diagonal)
- All other entries 0

| Namespaces (n) | Messages/day (lambda) | Capacity (C) | Offered Load (A) | B(C,A) |
|----------------|-----------------------|---------------|-------------------|--------|
| 5              | 10                    | 50            | 1.0               | ~0     |
| 10             | 30                    | 100           | 3.0               | ~0     |
| 20             | 80                    | 100           | 8.0               | ~0     |
| 50             | 250                   | 100           | 25.0              | 0.0002 |
| 50             | 250                   | 200           | 25.0              | ~0     |
| 100            | 500                   | 100           | 50.0              | 0.12   |
| 100            | 500                   | 500           | 50.0              | ~0     |

**Key insight:** For deployments under 50 namespaces with daily sync,
bus capacity of 100 messages provides effectively zero blocking
probability. Beyond 50 namespaces, capacity should scale linearly with
`n`.
