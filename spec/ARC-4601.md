# ARC-4601: Agent Node Protocol

| Field       | Value                                                     |
|-------------|-----------------------------------------------------------|
| **Number**  | ARC-4601                                                  |
| **Title**   | Agent Node Protocol                                       |
| **Lineage** | Inspired by RFC 4601 (PIM-SM — Protocol Independent Multicast, Sparse Mode) |
| **Status**  | IMPLEMENTED                                               |
| **Authors** | HERMES Contributors                                       |
| **Created** | 2026-03-14                                                |
| **Requires** | ARC-0001 (Architecture), ARC-0793 (Reliable Transport), ARC-2314 (Skill Gateway Plane), ARC-3022 (Gateway), ARC-5322 (Message Format) |

---

> *"In PIM-SM, a router joins the multicast group and maintains forwarding state
> for as long as there are downstream listeners. The Agent Node is the same idea
> applied to inter-agent communication: a persistent local process that joins
> the bus, maintains observation state, and dispatches events to downstream
> consumers — bridging the gap between ephemeral sessions and continuous operation."*

## 1. Abstract

This specification defines the Agent Node Protocol for HERMES deployments. An
Agent Node is a persistent local process (daemon) that observes the HERMES bus
continuously, maintains a bidirectional link with a remote gateway, and dispatches
sub-agent processes with defined guardrails.

The Agent Node unifies two patterns that previously required separate scheduled
tasks: (1) periodic bus observation and forwarding, and (2) periodic sub-agent
invocation for autonomous tasks. By consolidating these into a single event-driven
daemon, the node reduces latency from minutes to seconds, provides liveness
signaling to the remote gateway, and enables autonomous dispatch within
operator-defined constraints.

The design preserves HERMES Phase 0 compatibility: the bus file format is
unchanged, and deployments without an Agent Node continue to function using
cron-based scripts.

## 2. Motivation

HERMES Phase 0 established file-based inter-agent communication through a shared
JSONL bus. This works well for synchronous sessions where a human operator reads
and writes to the bus. However, several use cases require continuous operation
between sessions:

1. **Latency**: Cron-based scripts introduce minutes of delay between bus write
   and external notification. A persistent observer reduces this to seconds.
2. **Liveness**: A remote gateway has no way to know whether the local deployment
   is operational. Heartbeats solve this.
3. **Autonomous dispatch**: Some bus messages (e.g., `dispatch` type) should
   trigger sub-agent invocation without human intervention, within guardrails.
4. **Consolidation**: Running separate cron jobs for observation, forwarding,
   and dispatch is fragile. A single daemon with coordinated tasks is more
   reliable.

## 3. Terminology

| Term | Definition |
|------|-----------|
| **Agent Node** | A persistent local process that observes the bus, links to a gateway, and dispatches sub-agents. Operates in the Control Plane per ARC-2314. |
| **Bus Observer** | The component that watches `bus.jsonl` for new messages using OS-level file notifications or polling. |
| **Gateway Link** | The bidirectional connection between the node and a remote gateway: SSE for inbound, HTTP POST for outbound. |
| **Dispatch Slot** | A bounded execution context for a spawned sub-agent process. The node has a configurable maximum number of concurrent slots. |
| **Heartbeat** | A periodic HTTP signal from the node to the gateway indicating liveness and operational metrics. Not a bus message. |
| **Escalation** | Routing a decision to the human operator when the node cannot act autonomously per its configured rules. |
| **Evaluation Cycle** | The periodic assessment of pending messages to determine action (dispatch, escalate, forward, or ignore). |

## 4. Architecture

### 4.1 Position in the Stack

The Agent Node sits at L4 (Application) in the ARC-0001 stack and operates in the
Control Plane per ARC-2314. It acts as a local Messenger — the persistent counterpart
to the ephemeral Heraldo role described in the clan topology.

```
┌──────────────────────────────────────────┐
│          Remote Gateway (cloud)          │
│  SSE /events  ·  POST /bus/push          │
│  POST /heartbeat                         │
└──────────────┬───────────────────────────┘
          SSE  │  HTTP POST
               │
┌──────────────┴───────────────────────────┐
│        Agent Node (local daemon)         │
│                                          │
│  ┌────────────┐ ┌───────────┐ ┌────────┐│
│  │BusObserver │ │GatewayLink│ │Dispatch││
│  │ kqueue/poll│ │ SSE+POST  │ │ slots  ││
│  │ offset-tail│ │ heartbeat │ │guardrail│
│  └─────┬──────┘ └─────┬─────┘ └───┬────┘│
│        └───────┬───────┘───────────┘     │
│                ▼                         │
│         MessageEvaluator                 │
│   dispatch | escalate | forward | ignore │
│                                          │
│  ┌──────────────────────────────────────┐│
│  │ StateManager (state.json + PID lock) ││
│  └──────────────────────────────────────┘│
└──────────────────────────────────────────┘
               │
               ▼
         bus.jsonl (ARC-5322)
```

### 4.2 State Machine

```
  ┌──────┐     run()     ┌─────────┐    drain()   ┌──────────┐   cleanup  ┌─────────┐
  │ INIT ├──────────────►│ RUNNING ├─────────────►│ DRAINING ├──────────►│ STOPPED │
  └──────┘               └─────────┘              └──────────┘           └─────────┘
```

| State | Description |
|-------|-------------|
| **INIT** | Load configuration, validate paths, acquire PID lock, initialize components |
| **RUNNING** | All three async tasks active (BusObserver, GatewayLink, Dispatcher) |
| **DRAINING** | Stop accepting new dispatches, wait for in-flight dispatches to complete |
| **STOPPED** | Persist state, release PID lock, exit |

## 5. Bus Observer

### 5.1 File Watch Strategy

The Bus Observer detects new messages in `bus.jsonl` using the most efficient
mechanism available on the platform:

| Platform | Primary | Fallback |
|----------|---------|----------|
| macOS | `select.kqueue()` (stdlib) | `os.stat()` polling |
| Linux | `inotify` (if available) | `os.stat()` polling |
| Other | — | `os.stat()` polling |

The fallback polling interval is configurable (default: 2 seconds).

### 5.2 Offset-Based Tail

The observer MUST NOT re-read the entire bus file on each event. It MUST track
the byte offset of the last successfully parsed position and read only new bytes
appended after that offset.

```
bus.jsonl:
  [bytes 0..1234]     ← already processed (offset = 1234)
  [bytes 1234..1500]  ← NEW — read, parse, emit
  offset ← 1500
```

If the file size is smaller than the stored offset (indicating the bus was
rewritten after an archive operation), the observer MUST reset the offset to 0
and re-scan the entire file.

### 5.3 Event Classification

For each new message parsed, the observer classifies it:

| Condition | Event |
|-----------|-------|
| `dst` matches node namespace or `*`, namespace not in `ack` | `NewMessage` |
| Unacked message older than `escalation_threshold` | `StaleAlert` |
| Parse failure | Warning logged, line skipped |

## 6. Gateway Link

### 6.1 Upstream (Inbound via SSE)

The node connects to the gateway's SSE endpoint to receive inbound messages.

- **Reconnection**: Exponential backoff starting at 1s, doubling to max 60s
- **Validation**: Inbound messages are validated per ARC-5322 before writing to bus
- **Deduplication**: Messages already present on the bus (matched by `src + ts + msg`)
  are not written again

### 6.2 Downstream (Outbound via HTTP POST)

When the Bus Observer detects outbound-eligible messages, the node forwards them
to the gateway's push endpoint.

- **Eligible types**: `alert`, `dispatch`, `event` (configurable)
- **ACK**: Forwarded messages are ACKed with the node's namespace tag
- **Retry**: Transient failures (429, 5xx) are retried with exponential backoff

### 6.3 Heartbeat

The node sends a periodic HTTP POST to the gateway's heartbeat endpoint.

**Interval**: Configurable, default 60 seconds.

**Payload**:
```json
{
  "node_id": "string",
  "ts": "ISO-8601 datetime",
  "bus_lines": 42,
  "dispatch_slots_used": 1,
  "dispatch_slots_total": 2,
  "uptime_seconds": 3600
}
```

The heartbeat is a transport-layer signal, NOT a bus message. It MUST NOT be
written to `bus.jsonl`. The gateway MAY use heartbeat absence (> 3 intervals)
to mark the node as offline.

## 7. Dispatch Protocol

### 7.1 Spawn Model

Sub-agents are spawned as child processes using `subprocess`. Each dispatch
occupies one slot. The node enforces:

| Guardrail | Default | Configurable |
|-----------|---------|-------------|
| Max concurrent slots | 2 | Yes |
| Timeout per dispatch | 300s | Yes |
| Allowed tools | Empty (all blocked) | Yes |
| Max turns | 10 | Yes |

### 7.2 Trigger Rules (Evaluation Cycle)

The node evaluates pending messages every `evaluation_interval` (default: 300s):

| Message Pattern | Action |
|-----------------|--------|
| `type=dispatch`, `dst` matches node namespace | **DISPATCH** — spawn sub-agent |
| `type=alert`, age > `escalation_threshold` (default 4h), unacked | **ESCALATE** — forward with escalation flag |
| `type=request` | **ESCALATE** — requires human decision |
| `type=event`, `dst` is external namespace | **FORWARD** — send via gateway link |
| Already ACKed, or `type=state` | **IGNORE** |

### 7.3 Dispatch Command

The default dispatch command invokes Claude CLI:

```
claude -p "<message payload>" \
    --max-turns <max_turns> \
    --allowedTools "<allowed_tools>" \
    --output-format json
```

The command template is configurable for non-Claude sub-agents.

### 7.4 Result Handling

After a dispatch completes:

| Outcome | Bus Action |
|---------|-----------|
| Exit 0, output captured | Write `event` with `[RE:<CID>]` containing summary |
| Exit non-zero | Write `alert`: `DISPATCH_FAILED:<CID>:<exit_code>` |
| Timeout | Write `alert`: `DISPATCH_TIMEOUT:<CID>` |

### 7.5 Escalation

When a message requires escalation:

1. Write an `alert` to the bus: `ESCALATION:<type>:<summary>`
2. Forward the alert immediately via Gateway Link
3. The operator decides action in their next session

## 8. State Management

### 8.1 State File

Location: `{clan_dir}/agent-node.state.json`

```json
{
  "pid": 12345,
  "started_at": "2026-03-14T10:00:00",
  "last_heartbeat": "2026-03-14T11:30:00",
  "bus_offset": 4567,
  "active_dispatches": [
    {"pid": 12346, "cid": "task-01", "started_at": 1710410000.0}
  ],
  "last_evaluation": "2026-03-14T11:25:00"
}
```

The state file is written atomically: write to `.tmp`, then `os.rename()`.

### 8.2 PID Lock

The node uses a directory-based lock consistent with ARC-9001:

```
{clan_dir}/agent-node.lock/
└── pid       ← contains the PID of the owning process
```

Lock acquisition uses `os.mkdir()` (atomic on POSIX). If the lock exists and the
PID inside is not alive, the lock is considered stale and reclaimed.

### 8.3 Recovery

On startup, if a state file exists:

1. Check if the recorded PID is alive → if yes, abort (node already running)
2. If PID is dead: load state, resume from stored `bus_offset`
3. Clean up any orphaned dispatch PIDs (SIGTERM, then SIGKILL after 5s)

## 9. Configuration

The Agent Node reads its configuration from the `agent_node` section of
`gateway.json`. This section is OPTIONAL — its absence means the node
is not configured for this deployment.

```json
{
  "clan_id": "momoshod",
  "display_name": "Clan MomoshoD",
  "agent_node": {
    "enabled": true,
    "namespace": "heraldo",
    "bus_path": "bus.jsonl",
    "gateway_url": "https://heraldo-gateway.onrender.com",
    "gateway_sse_path": "/events",
    "gateway_push_path": "/bus/push",
    "gateway_heartbeat_path": "/heartbeat",
    "auth_token": "",
    "heartbeat_interval": 60,
    "evaluation_interval": 300,
    "max_dispatch_slots": 2,
    "dispatch_timeout": 300,
    "dispatch_command": "claude",
    "dispatch_max_turns": 10,
    "dispatch_allowed_tools": [],
    "poll_interval": 2.0,
    "escalation_threshold_hours": 4,
    "forward_types": ["alert", "dispatch", "event"]
  }
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | bool | `false` | Whether the node should start |
| `namespace` | string | `"heraldo"` | Node's namespace for bus filtering |
| `bus_path` | string | `"bus.jsonl"` | Path to bus file (relative to clan dir) |
| `gateway_url` | string | — | Base URL of remote gateway |
| `gateway_sse_path` | string | `"/events"` | SSE endpoint path |
| `gateway_push_path` | string | `"/bus/push"` | Push endpoint path |
| `gateway_heartbeat_path` | string | `"/heartbeat"` | Heartbeat endpoint path |
| `auth_token` | string | `""` | Legacy fallback token for both endpoints |
| `gateway_key` | string | `""` | `X-Gateway-Key` header for `/bus/push` |
| `sse_token` | string | `""` | Query parameter `token` for `/events` SSE |
| `heartbeat_interval` | int | `60` | Seconds between heartbeats |
| `evaluation_interval` | int | `300` | Seconds between evaluation cycles |
| `max_dispatch_slots` | int | `2` | Maximum concurrent sub-agents |
| `dispatch_timeout` | int | `300` | Seconds before dispatch timeout |
| `dispatch_command` | string | `"claude"` | Command to invoke for dispatch |
| `dispatch_max_turns` | int | `10` | Max turns for Claude CLI dispatch |
| `dispatch_allowed_tools` | list | `[]` | Tools allowed in dispatch (empty = all blocked) |
| `poll_interval` | float | `2.0` | Fallback polling interval in seconds |
| `escalation_threshold_hours` | int | `4` | Hours before unacked alerts escalate |
| `forward_types` | list | `["alert","dispatch","event"]` | Message types eligible for forwarding |

## 10. CLI Interface

The Agent Node integrates with the HERMES CLI via a `daemon` subcommand:

```
hermes daemon start [--dir PATH] [--foreground]
hermes daemon stop  [--dir PATH]
hermes daemon status [--dir PATH]
```

| Command | Behavior |
|---------|----------|
| `start` | Start the agent node. Default: daemonize. `--foreground` runs in the current process (for process managers). |
| `stop` | Send SIGTERM to the running node (read PID from lock). |
| `status` | Print node state: PID, uptime, bus offset, active dispatches, last heartbeat. |

### 11.2 Linux (systemd)

```ini
[Unit]
Description=HERMES Agent Node
After=network.target

[Service]
Type=simple
ExecStart=/path/to/python -m hermes.cli daemon start --foreground --dir /path/to/clan
```

### 11.3 Generic

The `--foreground` flag makes the daemon suitable for any process manager
(supervisord, Docker, etc.).

## 12. Security Considerations

### 12.1 Threat Model

| ID | Threat | Mitigation |
|----|--------|-----------|
| T1 | Bus file tampering | Validate all messages per ARC-5322 before processing |
| T2 | Malicious dispatch commands | Strict tool allowlist, no shell expansion, subprocess isolation |
| T3 | Gateway impersonation | HTTPS required, `X-Gateway-Key` for push + SSE token for read |
| T4 | Resource exhaustion | Max dispatch slots, timeouts, heartbeat-based monitoring |
| T5 | PID hijacking | Verify lock PID matches live process |
| T6 | State file corruption | Atomic writes (tmp + rename), recovery on startup |

### 12.2 Principle of Least Privilege

The Agent Node process requires:
- **Read/write**: `bus.jsonl`, `agent-node.state.json`, `agent-node.lock/`
- **Read**: `gateway.json`
- **Network**: Outbound HTTPS to `gateway_url`
- **Process**: Ability to spawn `dispatch_command`

It does NOT require:
- Access to other namespaces' data
- MCP credentials
- Write access to spec files or configuration beyond its state

## 13. Backward Compatibility

- The bus file format (ARC-5322) is unchanged
- Deployments without an Agent Node continue to work with cron-based scripts
- The `agent_node` configuration section is optional
- The heartbeat is HTTP-only and does not affect the bus protocol
- Sequence numbers and writer PID (ARC-9001) are preserved in forwarded messages
