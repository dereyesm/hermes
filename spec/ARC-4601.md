# ARC-4601: Agent Node Protocol

| Field       | Value                                                     |
|-------------|-----------------------------------------------------------|
| **Number**  | ARC-4601                                                  |
| **Title**   | Agent Node Protocol                                       |
| **Lineage** | Inspired by RFC 4601 (PIM-SM — Protocol Independent Multicast, Sparse Mode) |
| **Status**  | IMPLEMENTED                                               |
| **Authors** | HERMES Contributors                                       |
| **Created** | 2026-03-14                                                |
| **Requires** | ARC-0001 (Architecture), ARC-0793 (Reliable Transport), ARC-2314 (Skill Gateway Plane), ARC-3022 (Gateway), ARC-5322 (Message Format) |

---

> *"In PIM-SM, a router joins the multicast group and maintains forwarding state
> for as long as there are downstream listeners. The Agent Node is the same idea
> applied to inter-agent communication: a persistent local process that joins
> the bus, maintains observation state, and dispatches events to downstream
> consumers — bridging the gap between ephemeral sessions and continuous operation."*

## 1. Abstract

This specification defines the Agent Node Protocol for HERMES deployments. An
Agent Node is a persistent local process (daemon) that observes the Amaru bus
continuously, maintains a bidirectional link with a remote gateway, and dispatches
sub-agent processes with defined guardrails.

The Agent Node unifies two patterns that previously required separate scheduled
tasks: (1) periodic bus observation and forwarding, and (2) periodic sub-agent
invocation for autonomous tasks. By consolidating these into a single event-driven
daemon, the node reduces latency from minutes to seconds, provides liveness
signaling to the remote gateway, and enables autonomous dispatch within
operator-defined constraints.

The design preserves HERMES Phase 0 compatibility: the bus file format is
unchanged, and deployments without an Agent Node continue to function using
cron-based scripts.

## 2. Motivation

HERMES Phase 0 established file-based inter-agent communication through a shared
JSONL bus. This works well for synchronous sessions where a human operator reads
and writes to the bus. However, several use cases require continuous operation
between sessions:

1. **Latency**: Cron-based scripts introduce minutes of delay between bus write
   and external notification. A persistent observer reduces this to seconds.
2. **Liveness**: A remote gateway has no way to know whether the local deployment
   is operational. Heartbeats solve this.
3. **Autonomous dispatch**: Some bus messages (e.g., `dispatch` type) should
   trigger sub-agent invocation without human intervention, within guardrails.
4. **Consolidation**: Running separate cron jobs for observation, forwarding,
   and dispatch is fragile. A single daemon with coordinated tasks is more
   reliable.

## 3. Terminology

| Term | Definition |
|------|-----------|
| **Agent Node** | A persistent local process that observes the bus, links to a gateway, and dispatches sub-agents. Operates in the Control Plane per ARC-2314. |
| **Bus Observer** | The component that watches `bus.jsonl` for new messages using OS-level file notifications or polling. |
| **Gateway Link** | The bidirectional connection between the node and a remote gateway: SSE for inbound, HTTP POST for outbound. |
| **Dispatch Slot** | A bounded execution context for a spawned sub-agent process. The node has a configurable maximum number of concurrent slots. |
| **Heartbeat** | A periodic HTTP signal from the node to the gateway indicating liveness and operational metrics. Not a bus message. |
| **Escalation** | Routing a decision to the human operator when the node cannot act autonomously per its configured rules. |
| **Evaluation Cycle** | The periodic assessment of pending messages to determine action (dispatch, escalate, forward, or ignore). |

## 4. Architecture

### 4.1 Position in the Stack

The Agent Node sits at L4 (Application) in the ARC-0001 stack and operates in the
Control Plane per ARC-2314. It acts as a local Messenger — the persistent counterpart
to the ephemeral Heraldo role described in the clan topology.

```
┌──────────────────────────────────────────┐
│          Remote Gateway (cloud)          │
│  SSE /events  ·  POST /bus/push          │
│  POST /heartbeat                         │
└──────────────┬───────────────────────────┘
          SSE  │  HTTP POST
               │
┌──────────────┴───────────────────────────┐
│        Agent Node (local daemon)         │
│                                          │
│  ┌────────────┐ ┌───────────┐ ┌────────┐│
│  │BusObserver │ │GatewayLink│ │Dispatch││
│  │ kqueue/poll│ │ SSE+POST  │ │ slots  ││
│  │ offset-tail│ │ heartbeat │ │guardrail│
│  └─────┬──────┘ └─────┬─────┘ └───┬────┘│
│        └───────┬───────┘───────────┘     │
│                ▼                         │
│         MessageEvaluator                 │
│   dispatch | escalate | forward | ignore │
│                                          │
│  ┌──────────────────────────────────────┐│
│  │ StateManager (state.json + PID lock) ││
│  └──────────────────────────────────────┘│
└──────────────────────────────────────────┘
               │
               ▼
         bus.jsonl (ARC-5322)
```

### 4.2 State Machine

```
  ┌──────┐     run()     ┌─────────┐    drain()   ┌──────────┐   cleanup  ┌─────────┐
  │ INIT ├──────────────►│ RUNNING ├─────────────►│ DRAINING ├──────────►│ STOPPED │
  └──────┘               └─────────┘              └──────────┘           └─────────┘
```

| State | Description |
|-------|-------------|
| **INIT** | Load configuration, validate paths, acquire PID lock, initialize components |
| **RUNNING** | All three async tasks active (BusObserver, GatewayLink, Dispatcher) |
| **DRAINING** | Stop accepting new dispatches, wait for in-flight dispatches to complete |
| **STOPPED** | Persist state, release PID lock, exit |

## 5. Bus Observer

### 5.1 File Watch Strategy

The Bus Observer detects new messages in `bus.jsonl` using the most efficient
mechanism available on the platform:

| Platform | Primary | Fallback |
|----------|---------|----------|
| macOS | `select.kqueue()` (stdlib) | `os.stat()` polling |
| Linux | `inotify` (if available) | `os.stat()` polling |
| Other | — | `os.stat()` polling |

The fallback polling interval is configurable (default: 2 seconds).

### 5.2 Offset-Based Tail

The observer MUST NOT re-read the entire bus file on each event. It MUST track
the byte offset of the last successfully parsed position and read only new bytes
appended after that offset.

```
bus.jsonl:
  [bytes 0..1234]     ← already processed (offset = 1234)
  [bytes 1234..1500]  ← NEW — read, parse, emit
  offset ← 1500
```

If the file size is smaller than the stored offset (indicating the bus was
rewritten after an archive operation), the observer MUST reset the offset to 0
and re-scan the entire file.

### 5.3 Event Classification

For each new message parsed, the observer classifies it:

| Condition | Event |
|-----------|-------|
| `dst` matches node namespace or `*`, namespace not in `ack` | `NewMessage` |
| Unacked message older than `escalation_threshold` | `StaleAlert` |
| Parse failure | Warning logged, line skipped |

## 6. Gateway Link

### 6.1 Upstream (Inbound via SSE)

The node connects to the gateway's SSE endpoint to receive inbound messages.

- **Reconnection**: Exponential backoff starting at 1s, doubling to max 60s
- **Validation**: Inbound messages are validated per ARC-5322 before writing to bus
- **Deduplication**: Messages already present on the bus (matched by `src + ts + msg`)
  are not written again

### 6.2 Downstream (Outbound via HTTP POST)

When the Bus Observer detects outbound-eligible messages, the node forwards them
to the gateway's push endpoint.

- **Eligible types**: `alert`, `dispatch`, `event` (configurable)
- **ACK**: Forwarded messages are ACKed with the node's namespace tag
- **Retry**: Transient failures (429, 5xx) are retried with exponential backoff

### 6.3 Heartbeat

The node sends a periodic HTTP POST to the gateway's heartbeat endpoint.

**Interval**: Configurable, default 60 seconds.

**Payload**:
```json
{
  "node_id": "string",
  "ts": "ISO-8601 datetime",
  "bus_lines": 42,
  "dispatch_slots_used": 1,
  "dispatch_slots_total": 2,
  "uptime_seconds": 3600
}
```

The heartbeat is a transport-layer signal, NOT a bus message. It MUST NOT be
written to `bus.jsonl`. The gateway MAY use heartbeat absence (> 3 intervals)
to mark the node as offline.

## 7. Dispatch Protocol

### 7.1 Spawn Model

Sub-agents are spawned as child processes using `subprocess`. Each dispatch
occupies one slot. The node enforces:

| Guardrail | Default | Configurable |
|-----------|---------|-------------|
| Max concurrent slots | 2 | Yes |
| Timeout per dispatch | 300s | Yes |
| Allowed tools | Empty (all blocked) | Yes |
| Max turns | 10 | Yes |

### 7.2 Trigger Rules (Evaluation Cycle)

The node evaluates pending messages every `evaluation_interval` (default: 300s):

| Message Pattern | Action |
|-----------------|--------|
| `type=dispatch`, `dst` matches node namespace | **DISPATCH** — spawn sub-agent |
| `type=alert`, age > `escalation_threshold` (default 4h), unacked | **ESCALATE** — forward with escalation flag |
| `type=request` | **ESCALATE** — requires human decision |
| `type=event`, `dst` is external namespace | **FORWARD** — send via gateway link |
| Already ACKed, or `type=state` | **IGNORE** |

### 7.3 Dispatch Command

The default dispatch command invokes Claude CLI:

```
claude -p "<message payload>" \
    --max-turns <max_turns> \
    --allowedTools "<allowed_tools>" \
    --output-format json
```

The command template is configurable for non-Claude sub-agents.

### 7.4 Result Handling

After a dispatch completes:

| Outcome | Bus Action |
|---------|-----------|
| Exit 0, output captured | Write `event` with `[RE:<CID>]` containing summary |
| Exit non-zero | Write `alert`: `DISPATCH_FAILED:<CID>:<exit_code>` |
| Timeout | Write `alert`: `DISPATCH_TIMEOUT:<CID>` |

### 7.5 Escalation

When a message requires escalation:

1. Write an `alert` to the bus: `ESCALATION:<type>:<summary>`
2. Forward the alert immediately via Gateway Link
3. The operator decides action in their next session

## 8. State Management

### 8.1 State File

Location: `{clan_dir}/agent-node.state.json`

```json
{
  "pid": 12345,
  "started_at": "2026-03-14T10:00:00",
  "last_heartbeat": "2026-03-14T11:30:00",
  "bus_offset": 4567,
  "active_dispatches": [
    {"pid": 12346, "cid": "task-01", "started_at": 1710410000.0}
  ],
  "last_evaluation": "2026-03-14T11:25:00"
}
```

The state file is written atomically: write to `.tmp`, then `os.rename()`.

### 8.2 PID Lock

The node uses a directory-based lock consistent with ARC-9001:

```
{clan_dir}/agent-node.lock/
└── pid       ← contains the PID of the owning process
```

Lock acquisition uses `os.mkdir()` (atomic on POSIX). If the lock exists and the
PID inside is not alive, the lock is considered stale and reclaimed.

### 8.3 Recovery

On startup, if a state file exists:

1. Check if the recorded PID is alive → if yes, abort (node already running)
2. If PID is dead: load state, resume from stored `bus_offset`
3. Clean up any orphaned dispatch PIDs (SIGTERM, then SIGKILL after 5s)

## 10. CLI Interface

The Agent Node integrates with the Amaru CLI via a `daemon` subcommand:

```
hermes daemon start [--dir PATH] [--foreground]
hermes daemon stop  [--dir PATH]
hermes daemon status [--dir PATH]
```

| Command | Behavior |
|---------|----------|
| `start` | Start the agent node. Default: daemonize. `--foreground` runs in the current process (for process managers). |
| `stop` | Send SIGTERM to the running node (read PID from lock). |
| `status` | Print node state: PID, uptime, bus offset, active dispatches, last heartbeat. |

## 12. Security Considerations

### 12.1 Threat Model

| ID | Threat | Mitigation |
|----|--------|-----------|
| T1 | Bus file tampering | Validate all messages per ARC-5322 before processing |
| T2 | Malicious dispatch commands | Strict tool allowlist, no shell expansion, subprocess isolation |
| T3 | Gateway impersonation | HTTPS required, `X-Gateway-Key` for push + SSE token for read |
| T4 | Resource exhaustion | Max dispatch slots, timeouts, heartbeat-based monitoring |
| T5 | PID hijacking | Verify lock PID matches live process |
| T6 | State file corruption | Atomic writes (tmp + rename), recovery on startup |

### 12.2 Principle of Least Privilege

The Agent Node process requires:
- **Read/write**: `bus.jsonl`, `agent-node.state.json`, `agent-node.lock/`
- **Read**: `gateway.json`
- **Network**: Outbound HTTPS to `gateway_url`
- **Process**: Ability to spawn `dispatch_command`

It does NOT require:
- Access to other namespaces' data
- MCP credentials
- Write access to spec files or configuration beyond its state

## 13. Backward Compatibility

- The bus file format (ARC-5322) is unchanged
- Deployments without an Agent Node continue to work with cron-based scripts
- The `agent_node` configuration section is optional
- The heartbeat is HTTP-only and does not affect the bus protocol
- Sequence numbers and writer PID (ARC-9001) are preserved in forwarded messages

## 15. Hub Mode (Server-Side Extension)

> *Added 2026-03-23. Outcome of BR-018 (Real-Time Architecture Decision).
> The Hybrid architecture won: Hub for routing + optional P2P tunnels.
> This section specifies Phase A — the Hub as a server-side Agent Node.*

### 15.1 Overview

Sections 1–13 define the Agent Node as a **client daemon** that connects
to a remote gateway. This section extends the specification with a **server
mode** where the Agent Node acts as a Hub — accepting connections from
multiple peer daemons and routing messages between them.

The Hub is the "SMTP server" of HERMES: it relays encrypted envelopes
between clans without reading their contents. ARC-8446 E2E encryption
ensures the Hub operates as a **routing convenience, not a trust boundary**.

**Design principle**: The Hub MUST NOT require changes to the client daemon
protocol defined in §5–§8. Existing daemons connect to the Hub the same way
they connect to any gateway. The Hub adds routing, store-and-forward, and
presence — capabilities that are invisible to clients.

### 15.2 Modes of Operation

An Agent Node deployment declares exactly one mode in its configuration:

| Mode | Role | Transport | Sections |
|------|------|-----------|----------|
| `client` | Local daemon, connects to remote Hub | SSE + POST (§6) or WebSocket (§15.5) | §5–§13 |
| `hub` | Server, accepts peer connections, routes messages | WebSocket listener + legacy SSE/POST | §15 |

A single process MUST NOT run both modes simultaneously. A deployment MAY
run one client node and one Hub node on the same machine if they serve
different clans.

### 15.3 Hub Architecture

```
┌──────────────────────────────────────────────────────────────┐
│                    Hub (server mode)                         │
│                                                              │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────────────┐ │
│  │ WS Listener  │  │ Legacy SSE   │  │ Store-and-Forward  │ │
│  │ /ws          │  │ /events      │  │ TTL queue per peer │ │
│  │ bidirectional│  │ /bus/push    │  │ evict on TTL       │ │
│  └──────┬───────┘  └──────┬───────┘  └─────────┬──────────┘ │
│         └────────┬────────┘                     │            │
│                  ▼                              │            │
│         ┌────────────────┐                      │            │
│         │ Connection     │◄─────────────────────┘            │
│         │ Table          │                                   │
│         │ clan → socket  │                                   │
│         └───────┬────────┘                                   │
│                 ▼                                            │
│         ┌────────────────┐                                   │
│         │ Message Router │                                   │
│         │ E2E passthrough│                                   │
│         │ no decryption  │                                   │
│         └────────────────┘                                   │
│                                                              │
│  ┌──────────────────────────────────────────────────────────┐│
│  │ HubState (hub-state.json)                                ││

│  └──────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────┘
         ↑ WS        ↑ WS        ↑ SSE+POST (legacy)
         │            │            │
    ┌────┴───┐  ┌────┴───┐  ┌────┴───┐
    │Client 1│  │Client 2│  │Client 3│
    │momoshod│  │  jei   │  │ nymyka │
    └────────┘  └────────┘  └────────┘
```

### 15.4 Transport

#### 15.4.1 WebSocket (primary)

The Hub listens for WebSocket connections on a configurable path (default
`/ws`). WebSocket is the preferred transport for Hub mode because it
provides bidirectional messaging over a single TCP connection.

The Hub MUST support standard WebSocket protocol (RFC 6455) over TLS.

#### 15.4.2 Legacy Endpoints (backward compatibility)

The Hub MUST also serve the legacy endpoints defined in §6:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/events` | GET (SSE) | Inbound messages to client (§6.1) |
| `/bus/push` | POST | Outbound messages from client (§6.2) |
| `/healthz` | GET | Heartbeat from client (§6.3) |

Legacy clients that use SSE + POST MUST work with the Hub without
modification. The Hub routes their messages identically to WebSocket peers.

### 15.5 WebSocket Wire Protocol

All WebSocket frames use JSON text encoding. Binary frames are reserved
for future compact format support.

#### 15.5.1 Frame Types

```json
// Client → Hub: Authentication
{"type": "auth", "clan_id": "momoshod",
 "nonce_response": "<Ed25519 signature of challenge nonce>",
 "sign_pub": "<hex Ed25519 public key>"}

// Hub → Client: Authentication challenge
{"type": "challenge", "nonce": "<hex random 32 bytes>"}

// Hub → Client: Authentication result
{"type": "auth_ok", "clan_id": "momoshod", "queue_depth": 3}
{"type": "auth_fail", "reason": "unknown clan_id"}

// Bidirectional: Message relay
{"type": "msg", "payload": <ARC-5322 message object>}

// Client → Hub: Heartbeat
{"type": "ping", "ts": "<ISO-8601>"}

// Hub → Client: Heartbeat response
{"type": "pong", "ts": "<ISO-8601>", "queue_depth": 0}

// Hub → Client: Queued message drain (on reconnect)
{"type": "drain", "messages": [<msg1>, <msg2>, ...], "remaining": 0}

// Hub → Client: Peer presence notification
{"type": "presence", "clan_id": "jei", "status": "online"|"offline"}
```

#### 15.5.2 Message Relay Flow

When the Hub receives a `msg` frame:

1. Extract `dst` from the ARC-5322 payload
2. Look up `dst` in the connection table
3. **If online**: Forward the `msg` frame to the destination's WebSocket
4. **If offline**: Enqueue in the store-and-forward queue for `dst`
5. **If `dst` is `*` (broadcast)**: Forward to all connected peers except sender
6. ACK the message to the sender (implicit — no error frame = success)

The Hub MUST NOT inspect, modify, or decrypt the `msg` field of the
ARC-5322 payload. It routes based solely on the `dst` and `src` header
fields.

### 15.6 Authentication

The Hub uses Ed25519 challenge-response authentication. This reuses the
existing HERMES key infrastructure (ARC-8446) — no new keys required.

#### 15.6.1 Authentication Flow

```
Client                          Hub
  │                              │
  │──── WebSocket connect ──────►│
  │                              │
  │◄─── challenge {nonce} ──────│
  │                              │
  │──── auth {clan_id,          │
  │      sign(nonce, privkey),  │
  │      sign_pub} ────────────►│
  │                              │
  │     Hub: lookup sign_pub     │
  │     in registered peers.     │
  │     Verify signature.        │
  │                              │
  │◄─── auth_ok {queue_depth} ──│
  │                              │
  │     Bidirectional msg flow   │
```

#### 15.6.2 Peer Registration

The Hub maintains a registry of authorized clans and their public keys:

```json
{
  "peers": {
    "momoshod": {
      "sign_pub": "85a940d9b5a2f084...",
      "display_name": "Clan MomoshoD",
      "registered_at": "2026-03-23T00:00:00Z"
    },
    "jei": {
      "sign_pub": "65bf...b893",
      "display_name": "Clan JEI",
      "registered_at": "2026-03-17T00:00:00Z"
    }
  }
}
```

Peer registration is an out-of-band process (manual configuration or
future `amaru peer add` flow). The Hub MUST reject connections from
unregistered clans.

### 15.7 Store-and-Forward

The Hub queues messages for peers that are currently offline.

#### 15.7.1 Queue Semantics

- One queue per registered peer (`dst` clan_id)
- FIFO ordering (oldest message delivered first)
- Maximum queue depth: configurable (default: 1000 messages)
- TTL per message: inherited from the ARC-5322 `ttl` field (in seconds)
- Messages with expired TTL are evicted during periodic sweep

#### 15.7.2 Drain on Reconnect

When a peer authenticates successfully and has queued messages:

1. Hub sends `drain` frame with up to 100 messages per batch
2. If more messages remain, `remaining` field indicates queue depth
3. Client processes drained messages, then Hub sends next batch
4. After drain completes, Hub begins real-time forwarding

#### 15.7.3 Broadcast Messages

Messages with `dst: "*"` are forwarded to all currently connected peers.
They are NOT queued for offline peers — broadcast messages are
ephemeral by nature.

### 15.8 E2E Passthrough

The Hub operates in **E2E passthrough mode**. This is not optional — it
is a fundamental design constraint.

#### 15.8.1 What the Hub Sees

For each routed message, the Hub has access to:

| Field | Visible | Purpose |
|-------|---------|---------|
| `ts` | Yes | Timestamp (routing, TTL calculation) |
| `src` | Yes | Sender clan_id (routing, logging) |
| `dst` | Yes | Destination clan_id (routing) |
| `type` | Yes | Message type (classification) |
| `ttl` | Yes | Time-to-live (queue eviction) |
| `ack` | Yes | Acknowledgment list |
| `msg` | **Opaque** | Encrypted ARC-8446 envelope — the Hub sees ciphertext, nonce, signature, public keys. It cannot decrypt. |
| `seq` | Yes | Sequence number (ARC-9001, if present) |

#### 15.8.2 What the Hub Cannot See

- Plaintext message content
- Shared secrets (ECDHE ephemeral keys are public; private keys never leave peers)
- Message semantics beyond `type` classification

#### 15.8.3 Metadata Exposure

The Hub can observe:
- Who communicates with whom (src/dst pairs)
- Communication frequency and timing patterns
- Message sizes (ciphertext length)
- Online/offline patterns (connection events)

For peers where metadata exposure is unacceptable, Phase B (§16) defines
P2P tunnels that bypass the Hub entirely.

| ID | Threat | Mitigation |
|----|--------|-----------|
| H1 | Unauthorized peer connection | Ed25519 challenge-response auth, peers_file allowlist |
| H2 | Metadata analysis by Hub operator | E2E passthrough (cannot read content). Phase B P2P tunnels for metadata-sensitive peers. |
| H3 | Hub compromise → message injection | Peers verify Ed25519 signatures on messages (ARC-8446). Hub cannot forge signatures. |
| H4 | Denial of service (connection flood) | `max_connections` limit, `auth_timeout`, rate limiting per IP |

| H6 | Man-in-the-middle (Hub position) | TLS for transport. ARC-8446 E2E for content. Hub is already assumed untrusted. |

## 16. P2P Tunnels (Phase B — Specification)

> *This section is a DRAFT specification for Phase B of the Hybrid architecture
> (BR-018 outcome). Phase A (Hub, §15) is IMPLEMENTED. Phase B adds optional
> direct peer-to-peer tunnels. Included for forward compatibility and to
> guide implementation. Updated 2026-03-25.*

### 16.1 Overview

Phase B adds optional direct peer-to-peer tunnels using the **Noise IK**
handshake pattern. Peers that can reach each other directly bypass the Hub
for message delivery, reducing latency and eliminating metadata exposure
at the Hub.

**Key principle**: P2P tunnels are an optimization, never a requirement.
Every message that can travel over a P2P tunnel can also travel via Hub
relay (§15). The Hub is the fallback at every stage.

**Relationship to ARC-8446**: P2P tunnels replace the per-message ECDHE
envelope with a session-level encrypted channel. Within a tunnel, messages
are protected by the Noise session keys — ARC-8446 per-message encryption
is OPTIONAL (the tunnel already provides confidentiality). However, peers
MAY double-encrypt if they want end-to-end guarantees that survive tunnel
compromise.

### 16.2 Noise IK Handshake

The Noise IK pattern provides mutual authentication in **1 round-trip**
using pre-shared static public keys (already distributed via ARC-8446
key infrastructure). Noise IK was chosen over other patterns because:

- **IK** = initiator's static key known to responder (pre-shared via
  `gateway.json` peer registry)
- 1-RTT: handshake completes in a single exchange (lower latency than
  XX which requires 2 RTT)
- Static key identity hiding for the initiator (encrypted under DH)

#### 16.2.1 Handshake Messages

Using Noise Protocol Framework notation
(`Noise_IK_25519_ChaChaPoly_SHA256`):

```
pre-message:
  → s                          (responder has initiator's static pub)
  ← s                          (initiator has responder's static pub)

message 1 (Initiator → Responder):
  → e, es, s, ss               1-RTT handshake initiation

message 2 (Responder → Initiator):
  ← e, ee, se                  Handshake response
```

Detailed token expansion:

| Token | Operation | Key material |
|-------|-----------|--------------|
| `→ e` | Initiator generates ephemeral X25519 keypair `e_i`, sends `e_i.pub` | — |
| `es`  | DH(`e_i.priv`, `s_r.pub`) | Mixes DH result into chaining key |
| `→ s` | Encrypt + send `s_i.pub` under current key | Identity hiding |
| `ss`  | DH(`s_i.priv`, `s_r.pub`) | Mixes static-static DH into key |
| `← e` | Responder generates ephemeral `e_r`, sends `e_r.pub` | — |
| `ee`  | DH(`e_i.priv`, `e_r.pub`) | Forward secrecy (both ephemerals) |
| `se`  | DH(`s_i.priv`, `e_r.pub`) | Binds initiator identity to session |

After message 2, both peers derive **two symmetric keys** (one per
direction) from the final chaining key via HKDF-SHA256.

#### 16.2.2 Key Reuse

HERMES reuses the ARC-8446 key material for Noise:

| ARC-8446 Key | Noise Role | Usage |
|-------------|------------|-------|
| X25519 static keypair (`dh_private` / `x25519_pub`) | `s` (static) | Identity + DH |
| Ed25519 keypair (`sign_private` / `ed25519_pub`) | **Not used in Noise** — used for Hub auth (§15.6) and bus signatures |

> **Note**: Noise IK uses X25519 for all DH operations. Ed25519 keys
> are NOT mixed into the Noise handshake. This is by design: Ed25519
> and X25519 serve different roles (signing vs DH), and Noise IK
> provides its own authentication through the `ss` and `se` DH tokens.

#### 16.2.3 Cipher Suite

```
Noise_IK_25519_ChaChaPoly_SHA256
  DH:     X25519 (RFC 7748)
  Cipher: ChaChaPoly (RFC 7539)
  Hash:   SHA-256
```

ChaChaPoly is chosen over AES-256-GCM (used in ARC-8446 bus encryption)
because:
- Better performance on devices without AES-NI hardware
- Constant-time implementation is simpler (no side-channel concerns)
- Noise Framework reference implementations default to ChaChaPoly

Both cipher suites are 256-bit security level.

#### 16.2.4 Handshake Wire Format

The handshake is carried over the Hub's WebSocket connection as a special
frame type:

```json
// Initiator → Hub → Responder: Tunnel request
{"type": "tunnel_init", "src": "momoshod", "dst": "jei",
 "noise_msg1": "<base64 Noise message 1>"}

// Responder → Hub → Initiator: Tunnel response
{"type": "tunnel_resp", "src": "jei", "dst": "momoshod",
 "noise_msg2": "<base64 Noise message 2>"}

// Hub → Both: Tunnel established
{"type": "tunnel_ok", "peer": "jei", "transport": "udp",
 "endpoint": "203.0.113.5:51234"}
```

The Hub relays handshake messages opaquely — it cannot read the Noise
payload (encrypted). Once both peers have the handshake response, they
attempt direct connectivity.

### 16.3 NAT Traversal

The Hub assists NAT traversal using a STUN-like mechanism integrated
into the Hub protocol. This is NOT full RFC 5389 STUN — it is a
simplified peer-address-exchange tailored for HERMES.

#### 16.3.1 Address Discovery

```
  Peer A                    Hub                     Peer B
    │                        │                        │
    │── STUN_REQUEST ───────►│                        │
    │   (udp_port=51234)     │                        │
    │                        │◄── STUN_REQUEST ───────│
    │                        │   (udp_port=61234)     │
    │                        │                        │
    │◄── STUN_RESPONSE ──────│                        │
    │   reflexive=203.0.113.5│                        │
    │   :51234               │── STUN_RESPONSE ──────►│
    │                        │  reflexive=198.51.100.7│
    │                        │  :61234                │
    │                        │                        │
    │◄── PEER_ADDRESS ───────│                        │
    │   jei=198.51.100.7     │── PEER_ADDRESS ───────►│
    │   :61234               │  momoshod=203.0.113.5  │
    │                        │  :51234                │
    │                        │                        │
    │────────── UDP hole punch ──────────────────────►│
    │◄─────────── UDP hole punch ─────────────────────│
```

#### 16.3.2 Wire Format

```json
// Peer → Hub: Address request
{"type": "stun_request", "clan_id": "momoshod",
 "udp_port": 51234, "target_peer": "jei"}

// Hub → Peer: Reflexive address (Hub sees the source IP)
{"type": "stun_response", "reflexive_addr": "203.0.113.5",
 "reflexive_port": 51234}

// Hub → Both: Exchange addresses (E2E encrypted via §15 WS session)
{"type": "peer_address", "clan_id": "jei",
 "addr": "198.51.100.7", "port": 61234,
 "nat_type": "endpoint_independent"}
```

#### 16.3.3 NAT Type Detection

The Hub performs minimal NAT type detection:

| NAT Type | Behavior | P2P Probability |
|----------|----------|-----------------|
| Full cone (EIM/EIF) | Same external mapping for all destinations | High |
| Address-restricted (EIM/ADF) | External mapping reused, filtered by address | Medium |
| Port-restricted (EIM/PDF) | External mapping reused, filtered by address+port | Medium (with timing) |
| Symmetric (ADM) | Different mapping per destination | Low — fallback to Hub relay |

If both peers are behind symmetric NAT, the Hub returns:

```json
{"type": "tunnel_fail", "reason": "symmetric_nat_both_sides",
 "fallback": "hub_relay"}
```

#### 16.3.4 Hub as TURN Relay (Fallback)

When direct connectivity fails, the Hub MAY act as a TURN-like relay
for the P2P tunnel. In this mode:

- The Noise handshake still completes (via Hub WS frames)
- Post-handshake data frames are relayed by the Hub instead of UDP
- The Hub sees encrypted Noise frames (cannot decrypt — no session keys)
- Latency increases but confidentiality is preserved

This mode is functionally equivalent to §15 Hub relay but with Noise
session encryption instead of per-message ARC-8446 envelopes.

### 16.4 Tunnel Lifecycle

```
                    ┌─────────────────────────────────────────────┐
                    │                                             │
IDLE ──► DISCOVERY ──► STUN_EXCHANGE ──► HANDSHAKE ──► ACTIVE ──►│
  ▲                     │                  │              │       │
  │                     │                  │              ▼       │
  │                     ▼                  ▼           REKEY ─────┘
  │              ┌─────────────┐    ┌────────────┐       │
  │              │ STUN_FAILED │    │ HS_FAILED  │       │
  │              └──────┬──────┘    └─────┬──────┘       │
  │                     │                 │              │
  │                     ▼                 ▼              ▼
  └──────────── FALLBACK_HUB ◄────────────┘         TEARDOWN
                    │                                    │
                    ▼                                    ▼
               (continue via §15 relay)              IDLE
```

#### 16.4.1 State Machine

| State | Entry condition | Exit condition | Timer |
|-------|----------------|----------------|-------|
| `IDLE` | No tunnel requested | Peer sends message → `DISCOVERY` | — |
| `DISCOVERY` | Message to peer; check if online | Hub presence confirmed → `STUN_EXCHANGE` | 5s timeout → `FALLBACK_HUB` |
| `STUN_EXCHANGE` | Both peers online | Reflexive addresses received → `HANDSHAKE` | 10s timeout → `FALLBACK_HUB` |
| `HANDSHAKE` | Addresses exchanged | Noise IK complete → `ACTIVE` | 5s timeout → `FALLBACK_HUB` |
| `ACTIVE` | Session keys derived | Rekey timer fires → `REKEY`; error → `TEARDOWN` | — |
| `REKEY` | 24h since last handshake | New handshake complete → `ACTIVE` | 10s timeout → `TEARDOWN` |
| `TEARDOWN` | Error or explicit close | Cleanup complete → `IDLE` | — |
| `FALLBACK_HUB` | Any failure | Message delivered via §15 | Reset to `IDLE` after message sent |

#### 16.4.2 Tunnel Establishment Policy

A node SHOULD attempt tunnel establishment when:
- First message to a peer that is currently online (via Hub presence)
- A previous tunnel tore down more than 60 seconds ago

A node MUST NOT attempt tunnel establishment when:
- The peer is offline (use Hub store-and-forward instead)
- The previous tunnel attempt failed less than 60 seconds ago (backoff)
- More than 3 tunnel attempts have failed for this peer in 1 hour

#### 16.4.3 Rekeying

Tunnels MUST rekey every **24 hours** by performing a new Noise IK
handshake. During rekeying:

1. New handshake runs in parallel with active tunnel
2. Old session keys are retained until new handshake completes
3. Once new keys are established, old keys are zeroized
4. If rekeying fails, the tunnel enters `TEARDOWN`

> **Rationale**: 24h rekey limits the window of compromise if session
> keys are leaked. The parallel handshake ensures zero downtime.

### 16.5 Data Transport

#### 16.5.1 UDP Framing

Post-handshake data travels over UDP with a minimal framing layer:

```
┌──────────┬──────────┬──────────────────────┐
│ Type (1) │ Len (2)  │ Payload (variable)   │
│  0x01    │  big-end │ Noise-encrypted data │
└──────────┴──────────┴──────────────────────┘
```

| Type | Value | Meaning |
|------|-------|---------|
| `0x01` | Data | ARC-5322 message (encrypted under Noise session) |
| `0x02` | Ping | Keepalive (every 30s) |
| `0x03` | Pong | Keepalive response |
| `0x04` | Close | Graceful tunnel teardown |
| `0x05` | Rekey | Trigger rekeying handshake |

#### 16.5.2 Message Encryption

Within an active tunnel, each data frame is encrypted using the Noise
transport keys (ChaChaPoly with incrementing nonce). The payload is an
ARC-5322 message in either verbose JSON or compact format.

Peers MAY omit the ARC-8446 per-message encryption when sending through
a tunnel (the tunnel provides equivalent confidentiality). However, peers
SHOULD retain ARC-8446 encryption if:
- The message may be forwarded to a third peer
- The message will be persisted to the bus (bus encryption is separate
  from transport encryption)
- Defense-in-depth is desired

#### 16.5.3 Keepalive

Peers exchange Ping/Pong frames every 30 seconds. If 3 consecutive pings
go unanswered (90s), the tunnel enters `TEARDOWN` and messages fall back
to Hub relay.

### 16.6 Security Properties

| Property | Provided by | Notes |
|----------|-------------|-------|
| **Mutual authentication** | Noise IK `ss` + `se` tokens | Both peers prove identity via static DH |
| **Forward secrecy** | Noise IK `ee` token | Ephemeral-ephemeral DH; compromise of static keys does not reveal past sessions |
| **Identity hiding (initiator)** | Noise IK `es` encryption of `s` | Initiator's static key is encrypted; passive eavesdropper cannot identify initiator |
| **Identity hiding (responder)** | **NOT provided** by IK | Responder's identity is visible in pre-message. Acceptable: both peers already know each other via Hub presence |
| **KCI resistance** | Noise IK `se` token | Compromise of initiator's static key does NOT allow impersonating the responder to the initiator |
| **Replay protection** | Noise nonce counter | Incrementing nonce per direction; reuse is fatal (Noise aborts) |
| **Traffic analysis resistance** | P2P (no Hub metadata) | Hub cannot see message timing or size; ISP can see UDP flow exists |

#### 16.6.1 Threat Model

| Threat | Mitigated | How |
|--------|-----------|-----|
| Hub reads message content | Yes (§15 already) | ARC-8446 E2E + Noise session encryption |
| Hub logs metadata (who talks to whom, when, size) | Yes (P2P) | Direct UDP bypasses Hub entirely |
| Passive network eavesdropper | Yes | Noise encryption on UDP datagrams |
| Active MitM on UDP | Yes | Noise IK mutual authentication using pre-shared static keys |
| Compromised Hub injects messages | Yes | Noise session rejects unauthenticated data |
| Compromised static key (past sessions) | Protected | Forward secrecy from ephemeral DH |
| Compromised static key (future sessions) | NOT protected | Requires key rotation via ARC-8446 key infrastructure |

### 16.7 Hub Integration

#### 16.7.1 Presence-Driven Discovery

The Hub's presence system (§15.8) drives tunnel discovery:

```json
// Hub → Client: Peer came online
{"type": "presence", "clan_id": "jei", "status": "online"}

// Client reacts: initiate tunnel if policy allows
→ {"type": "stun_request", ...}
```

#### 16.7.2 Graceful Degradation

The following invariant MUST hold:

> Any message that can be delivered over a P2P tunnel MUST also be
> deliverable via Hub relay (§15), with the only difference being
> latency and metadata exposure.

This means:
- Clients MUST NOT depend on tunnel availability for correctness
- The bus format (ARC-5322) is identical regardless of transport
- Store-and-forward (§15.7) handles offline peers regardless of tunnels

#### 16.7.3 Hub Resource Limits

The Hub imposes limits on tunnel-related operations:

| Resource | Limit | Rationale |
|----------|-------|-----------|
| STUN requests per peer per hour | 20 | Prevent address discovery abuse |
| Concurrent tunnel negotiations per peer | 2 | Prevent resource exhaustion |
| Tunnel relay bandwidth (TURN mode) | 1 MB/s per tunnel | Prevent Hub bandwidth exhaustion |

## §18. Channel-Aware Hub Routing (Amendment) — DRAFT

| Field      | Value                                                                        |
|------------|------------------------------------------------------------------------------|
| **Status** | DRAFT (skeleton)                                                             |
| **Date**   | 2026-04-06                                                                   |
| **Source** | Forward-referenced by ATR-Q.931 §7.4                                         |
| **Scope**  | Hub-side framing, routing, and receipt emission for the signaling plane     |
| **Out of scope** | Peer-side state machines (ATR-Q.931 §5), end-to-end crypto (ARC-8446) |

### 18.1 Motivation

ATR-Q.931 introduces an out-of-band signaling plane that shares the
existing Hub WebSocket transport with the data plane. This amendment
specifies the *minimum* hub-side obligations required for that plane to
operate without breaking pre-signaling clients. The amendment is
intentionally narrow: it does not redefine §15, it extends three points
of §15 (frame parsing, routing, presence advertisement) and adds a new
session-tracking surface that the hub MAY consult.

The amendment is split from ATR-Q.931 because peer obligations and hub
obligations evolve on independent cadences and conformance is reported
separately (peer-side via ARC-1122 L2, hub-side via ARC-1122 L3).

### 18.2 Frame Discrimination

A conformant hub MUST parse the optional `channel` field in every
inbound frame. Two values are defined by ATR-Q.931:

| `channel` | Meaning                                                |
|-----------|--------------------------------------------------------|
| `"sig"`   | Signaling frame (INVITE, 200, ACK, BYE, NOTIFY, …)    |
| `"data"`  | Data plane frame (ARC-5322 message envelope)          |

If the field is absent, the hub MUST treat the frame as `"data"` for
backward compatibility (see §18.6). If the field carries any other
value, the hub MUST reject the frame with a hub error frame
(`{"type":"err","code":"unknown-channel","ref":...}`) and MUST NOT
forward it.

The hub MUST NOT inspect the `payload` of a `"data"` frame beyond
ARC-5322 envelope validation. The hub MAY inspect the `type` field of a
`"sig"` frame for routing and rate-limiting purposes; signaling frames
remain unencrypted at the hub by design (see §18.8).

### 18.3 Session Table

A conformant hub MUST maintain a *session table* mapping
`session_id → {peers, established_at, last_activity_at}`. The table is
populated by observation of ATR-Q.931 signaling primitives:

| Primitive seen | Action on session table                                     |
|----------------|-------------------------------------------------------------|
| `INVITE`       | Insert entry in `INVITING` state                            |
| `200`          | Promote entry to `ESTABLISHED`                              |
| `ACK`          | Refresh `last_activity_at`                                  |
| `BYE`          | Remove entry (or move to `TERMINATING` then GC)             |
| `NOTIFY`       | Refresh `last_activity_at`                                  |

The session table is **advisory** for routing: data-plane frames carry
their `dst` independently and the hub MUST route them via the existing
§15.6 forwarding rules even when no session entry exists. The session
table exists to (a) enforce the signaling rate-limit asymmetry (§18.7),
(b) emit `DELIVERED` receipts (§18.4), and (c) garbage-collect inactive
sessions whose peers disconnected without `BYE`.

Garbage collection: an entry whose `last_activity_at` is older than the
session inactivity timeout (default 1800 seconds / 30 minutes, MAY be
configured) MUST be removed. Hub MAY emit a synthetic `BYE` to
surviving peers when an entry is GC'd.

> **Rationale for the 1800 s default.** This matches the
> `Session-Expires` default in IETF RFC 4028 §4 (SIP Session Timers),
> which solves the same problem for SIP dialogs traversing stateful
> proxies that need to garbage-collect sessions whose endpoints
> disappeared without sending `BYE`. ATR-Q.931 sessions traverse a
> stateful Hub with the same failure mode, so the same default applies.

The session table is per-hub, in-memory, and not persisted across hub
restarts. Federation-side persistence is out of scope and will be
addressed by a future federation-persistence amendment (the federation
plane itself is implemented in `hub.py` `FederationLink`/`FederationTable`
but its formal spec body is not yet incorporated into this document; see
the editorial note above §18 for context).

> **Hub restart semantics (peer-visible behavior).** Because the session
> table is not persisted, a hub restart causes the loss of all active
> session entries. Peers that held an `ESTABLISHED` session through that
> hub MUST NOT assume the session survived. Conformant peers SHOULD
> detect loss of session by:
>
> 1. Observing WebSocket reconnect (the §15.2 auth handshake re-runs),
>    which is an unambiguous signal that any prior session state at the
>    hub is gone; or
> 2. Observing an `err` frame with `code: "unknown-session"` (§18.10) in
>    response to an in-session primitive, which indicates the hub no
>    longer knows the referenced `session_id` (either GC'd or lost on
>    restart).
>
> In both cases, a peer that wishes to continue the conversation MUST
> re-issue `INVITE` (ATR-Q.931 §7.1) to re-establish the session. The
> hub SHOULD treat a `re-INVITE` carrying a previously-known
> `session_id` as a fresh `INVITE` and insert a new entry in `INVITING`
> state rather than erroring. This makes hub restarts transparent at
> the ATR-Q.931 layer after one round-trip of signaling.

### 18.4 Receipt Frame Emission

The four-stage receipt model (`SENT`, `DELIVERED`, `READ`, `PROCESSED`)
defined in ATR-Q.931 §8 is hub-emitted for the first two stages and
peer-emitted for the last two. This subsection defines hub obligations:

**`SENT` receipt** (already implemented in the reference hub at
`reference/python/amaru/hub.py:_emit_sent_receipt`, commit 611b88a).
The hub MUST emit a `SENT` receipt to the originator immediately after
accepting a frame for routing **and** the originator opted in via
`receipt: ["SENT", ...]` in the original message. Receipt format follows
ATR-Q.931 §8.2.

**`QUEUED` receipt** (NOT YET IMPLEMENTED, added in 2026-04-13 revision
as a synthetic hub-emitted stage for offline destinations — see
rationale at the end of this subsection). When the hub accepts a frame
for routing but the destination peer is not currently connected, the
hub MUST emit a `QUEUED` receipt back to the originator iff:

1. The originator opted in via `receipt: [..., "QUEUED", ...]`
   (opt-in, like every other receipt stage).
2. The hub successfully enqueued the frame for store-and-forward per
   §15.7 (i.e., the per-peer queue had capacity; otherwise the hub
   MUST emit an `err` frame with `code:"queue-full"` instead).

`QUEUED` asserts that the hub has accepted responsibility for eventual
delivery — it does **not** assert that the destination will ever read
the frame. The destination's `DELIVERED` receipt (below) is a distinct,
later, stronger signal.

**`DELIVERED` receipt** (NOT YET IMPLEMENTED). `DELIVERED` asserts that
the frame has been written to the destination's live WebSocket session
with no error at the socket layer. The hub MUST emit a `DELIVERED`
receipt back to the originator iff:

1. The originator opted in via `receipt: [..., "DELIVERED", ...]`.
2. The hub successfully forwarded the frame to the destination peer's
   WebSocket and the WebSocket write completed without error. If the
   destination peer was offline at forwarding time but became online
   later and the queued frame was then successfully written on
   reconnect, the hub MUST emit `DELIVERED` at that later point
   (unless the frame expired per §15.7 queue retention, in which case
   the hub MUST emit `err` with `code:"queue-expired"` instead of
   `DELIVERED`).
3. A session entry exists for the message's `session_id`, OR the
   message has no `session_id` (legacy data frames use
   originator+ref tracking).

`READ` and `PROCESSED` receipts MUST NOT be synthesised by the hub; they
are emitted only by the destination Agent Node and the hub forwards them
opaquely as ordinary signaling frames.

> **Rationale — why the receipt lattice is `SENT → QUEUED → DELIVERED
> → READ → PROCESSED` (resolving the 2026-04-06 open question).** The
> original draft considered two alternatives for offline destinations:
> (a) emit a synthetic `QUEUED` stage while the frame sits in store-
> and-forward, then emit `DELIVERED` on the destination's reconnect;
> (b) treat the §15.7 store-and-forward enqueue as the DELIVERED
> checkpoint (weaker semantics, matching SS7 SMSC behaviour).
>
> Alternative (a) is adopted. First-clan feedback (JEI, 2026-04-13)
> argued that JEI-side agents operate in batch modes (e.g. Huitaca's
> scheduled batches, GoGi daemon pauses) where hours of offline time
> between runs is normal and expected. In that regime, a `QUEUED`
> receipt is the meaningful operational signal — it confirms the hub
> accepted custody of the message — whereas collapsing `QUEUED` into
> `DELIVERED` would hide the distinction between "the hub promised to
> try later" and "the destination's socket has actually received the
> bytes." That distinction matters for `alert` and `dispatch` message
> types whose originators need to know whether retry or escalation is
> the right next action.
>
> The SS7 SMSC analogy (alternative b) remains useful for stateless
> `data`-plane frames that do not opt in to receipts, which is already
> the default. Opt-in to `QUEUED` is what promotes a frame into the
> stronger receipt lattice, so operators retain the option to pick the
> weaker semantics on a per-message basis simply by not opting in.

### 18.5 Capability Negotiation (`auth_ok` Extension)

This amendment does **not** introduce a new handshake frame. It extends
the existing `auth_ok` frame defined in §15.5.1 with two OPTIONAL
fields, `caps` and `channels`. Both fields default to absent for
backward compatibility.

```json
{"type":"auth_ok","clan_id":"momoshod","queue_depth":3,
 "caps":["signaling_v1"],
 "channels":["sig","data"]}
```

The peer's `auth` frame (§15.5.1) is similarly extended with the same
two OPTIONAL fields so the hub can learn what the peer supports before
emitting `auth_ok`:

```json
{"type":"auth","clan_id":"momoshod",
 "nonce_response":"<sig>","sign_pub":"<hex>",
 "caps":["signaling_v1"],
 "channels":["sig","data"]}
```

A hub that implements this amendment MUST set `caps` to include
`signaling_v1` in its `auth_ok` frame. A peer that does not include
`signaling_v1` in its `auth` frame MUST be treated by the hub as a
*legacy peer*: the hub MUST NOT route signaling frames *to* that peer,
MUST NOT expect signaling frames *from* it, and MUST NOT include it in
session table entries.

If the hub did not advertise `signaling_v1` in its `auth_ok` and the
peer attempts to send a `"channel":"sig"` frame, the hub MUST reject
the frame with an error frame (see §18.10) carrying
`code:"signaling-not-supported"`.

> **Spec ↔ implementation gap.** The reference Hub at
> `reference/python/amaru/hub.py` exchanges a `HELLO` frame in addition
> to `auth`/`auth_ok` (used for peer invite/accept flows beyond the
> scope of this amendment). That `HELLO` frame is not yet documented in
> §15 and will be incorporated in a follow-up amendment to §15.5.1.
> This subsection deliberately rides on `auth_ok`, which §15 already
> defines, to remain self-contained.
>
> **Cross-amendment consistency (to be honored by the follow-up §15
> amendment).** When `HELLO` is formally added to §15.5.1, the follow-up
> amendment MUST specify how any capability or channel fields it
> carries interact with the `caps`/`channels` advertised in `auth_ok`.
> If both frames advertise overlapping capability state, `auth_ok`
> remains authoritative: `HELLO` occurs earlier in the handshake (per
> the current implementation), but `auth_ok` is the final frame before
> signaling frames become routable per §15.5, so its advertisement
> wins by ordering. The follow-up amendment MAY narrow this rule
> further, but MUST NOT make `HELLO` silently override an inconsistent
> `auth_ok` advertisement without an explicit err-frame signal.

### 18.6 Backward Compatibility

This amendment is fully backward compatible with ARC-4601 §15:

| Pre-amendment behaviour                            | Post-amendment behaviour                                        |
|----------------------------------------------------|-----------------------------------------------------------------|
| Frame without `channel` field                      | Treated as `"data"`, routed identically                          |
| Frame without `session_id`                         | Routed via `dst`, no session table interaction                  |
| Peer that does not negotiate `signaling_v1`        | Receives only data frames, never signaling frames               |
| Hub that does not advertise `signaling_v1`         | Behaves exactly as §15-conformant hub for all peers             |
| ATR-Q.931 receipt opt-in absent                    | No receipt emission, even from `signaling_v1` hub               |

A pre-amendment client connected to a post-amendment hub sees no change
in observable behaviour. A post-amendment client connected to a
pre-amendment hub gracefully degrades: signaling features are disabled,
data plane works as before.

### 18.7 Rate-Limiting Asymmetry

The hub MUST apply rate limits to signaling frames *separately* from
data frames. This is a deliberate architectural choice with prior art:

- **IETF RFC 3261 §26.4** (Session Initiation Protocol — Limitations
  of SIP) explicitly notes that signaling traffic and media traffic
  must be flow-controlled independently because their offered loads,
  burst patterns, and DoS surfaces differ. SIP entities apply
  separate budgets to INVITE storms vs. media RTP streams.
- **3GPP TS 23.501 §5.7.2** (Quality of Service Model — 5G System
  Architecture) defines independent QoS Flow Identifiers (QFIs) for
  signaling (NAS over N1) and user-plane data (PDU sessions over N3),
  precisely because the 5G core network handles both planes in a
  shared transport but with disjoint policy.

ARC-4601 §18 inherits this telecom convention. Suggested defaults (per
peer):

| Class           | Rate                       | Burst |
|-----------------|----------------------------|-------|
| `data` frames   | inherited from §15.7       | §15.7 |
| `sig` frames (non-receipt) | 10 frames/second  | 20    |
| `sig` receipt frames | 50 frames/second      | 100   |

Receipts are exempt from the lower signaling cap because the system is
designed to amplify a single message into up to four receipts and that
amplification is intentional. A hub MUST NOT count a receipt frame
against the data plane budget. Receipts MUST be counted against the
receipt sub-budget defined above.

A hub that drops a frame for rate-limiting reasons SHOULD emit a hub
error frame `{"type":"err","code":"rate-limited","class":"sig|data"}`
to the offender so the originator can back off intelligently.

### 18.8 Security Considerations

This amendment does **not** weaken the E2E confidentiality posture of
ARC-4601 §15. Specifically:

1. **Data plane stays opaque.** The hub never inspects `payload` content
   beyond ARC-5322 envelope validation. End-to-end encryption per
   ARC-8446 continues to apply unchanged.
2. **Signaling plane is intentionally hub-readable.** Signaling frames
   are unencrypted at the hub by design (parallel to SS7 ISUP, SIP over
   TLS where the proxy sees method headers). This is a known and
   accepted trade-off: the hub needs to read `INVITE`, `BYE`, and
   receipt frames to perform its routing role, and signaling content
   carries no payload secrets.
3. **Session table is not a metadata oracle.** The hub already sees
   src→dst pairs by virtue of routing; the session table adds session
   grouping but does not expose new information to a compromised hub.
   Operators that consider signaling metadata sensitive SHOULD use the
   P2P tunnel mode (§16) instead.
4. **Receipt forgery resistance.** Hub-emitted receipts MUST be signed
   under the hub's Ed25519 identity. Receipts forwarded from peers
   (`READ`, `PROCESSED`) MUST carry the peer's Ed25519 signature; the
   hub MUST NOT re-sign them.
5. **Receipt amplification DoS.** A malicious peer could opt every
   message into all four receipts to amplify hub load. The hub MUST
   apply the receipt rate limit of §18.7 even to amplification from a
   single peer.

### 18.10 Error Frames (New `err` Type)

ARC-4601 §15.5.1 defines a closed set of frame types and does not
provide a generic out-of-band error channel. This amendment introduces
the `err` frame type so that the hub can signal recoverable errors
(rate-limit drops, unknown channel, capability mismatch) without
tearing down the WebSocket connection or silently discarding frames.

#### 18.10.1 Frame Format

```json
{"type":"err",
 "code":"<token>",
 "ref":"<message_ref or null>",
 "detail":"<short human-readable phrase>"}
```

| Field    | Required | Description                                                                                     |
|----------|----------|-------------------------------------------------------------------------------------------------|
| `type`   | MUST     | Constant `"err"`                                                                                |
| `code`   | MUST     | Stable token from the registry below                                                            |
| `ref`    | SHOULD   | The `ref` of the offending message; `null` for connection-level                                 |
| `detail` | MAY      | Short phrase for operators; see constraints below                                               |

The `detail` field is intended for operator-facing short phrases (e.g.
`"queue depth exceeded"`, `"unknown session after hub restart"`) and
is subject to the following normative constraints:

- The `detail` field MUST NOT carry message payload contents.
- The `detail` field MUST NOT carry peer identifiers (`clan_id`,
  `sign_pub`, key fingerprints, or any other peer-scoped identifier)
  **beyond the offending peer's own `clan_id`**. In particular, when a
  single hub serves multiple clans (federated or multi-tenant
  deployments), the `detail` returned to peer A MUST NOT disclose
  state about peer B's identity, session, or connectivity.

  This constraint prevents information leakage in multi-clan scenarios
  where the err frame would otherwise become a side-channel for
  enumerating peers, session state, or federation topology of other
  clans sharing the same hub.

The `err` frame MUST be sent on the same WebSocket as the offending
frame. The hub MUST NOT close the WebSocket as a side effect of
emitting an `err` frame; the offender remains responsible for backing
off or unsubscribing.

#### 18.10.2 Code Registry (introduced by §18)

| `code`                       | Meaning                                                              | Section |
|------------------------------|----------------------------------------------------------------------|---------|
| `unknown-channel`            | `channel` field carried a value other than `"sig"` or `"data"`       | §18.2   |
| `signaling-not-supported`    | Peer sent `"channel":"sig"` but hub did not advertise `signaling_v1` | §18.5   |
| `rate-limited`               | Frame dropped because the per-peer budget was exhausted              | §18.7   |
| `unknown-session`            | `session_id` referenced an entry that does not exist (or was GC'd)   | §18.3   |

Future amendments to ARC-4601 MAY add codes to this registry. Codes
are case-sensitive lowercase tokens with hyphens; new codes MUST NOT
shadow existing ones.

#### 18.10.3 Backward Compatibility

A pre-amendment client that receives an `err` frame and does not
recognise the type SHOULD log it and continue. The amendment is safe
because legacy clients ignore unknown top-level keys in JSON frames
(per §15.13 forward-compatibility rule); receiving an `err` frame
will not cause a parser error in any §15-conformant client.

A pre-amendment hub will never emit an `err` frame, so clients that
implement this amendment MUST NOT depend on receiving one as the only
indication of an error. Clients SHOULD continue to detect WebSocket
close as the legacy error signal in addition to handling `err` frames.
