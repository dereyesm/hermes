---

> *"In telecom, the genius of CUPS was separating the brain from the muscle --
> letting control decisions scale independently from data forwarding. In HERMES,
> we go one step further: we separate signaling, orchestration, and execution
> into three planes -- because an agent ecosystem needs not just routing and
> payload, but a conductor who decides what work gets done, by whom, and when."*

## 1. Abstract

This specification defines the **Skill Gateway Plane Architecture** for
HERMES inter-clan operations. It extends the two-plane CUPS model
(ATR-X.200 Section 9) into a **triple-plane architecture** that
separates the operational concerns of the Skill Gateway into:

- **Control Plane (CP)**: Messengers -- signaling, routing, delivery,
  discovery, and session management between clans.
- **Orchestration Plane (OP)**: Dojos -- quest dispatch, backlog
  management, skill assignment, and governance within and across clans.
- **User Plane (UP)**: Skills -- the agents that execute actual work,
  process payloads, and produce deliverables.

Additionally, this specification defines the **HERMES Grand Dojo** --
the standard template that all clan-local Dojos MUST implement, analogous
to how 3GPP TS 23.501 defines the Service-Based Architecture that each
operator implements according to their topology.

---

## 2. Motivation

ATR-X.200 Section 9 established a two-plane CUPS model mapping
`bus.jsonl` to the User Plane and `routes.md` to the Control Plane.
This model is correct for intra-clan message transport but insufficient
for describing the operational architecture of inter-clan collaboration.

When clans connect through the Skill Gateway (ARC-3022), three distinct
operational roles emerge:

1. **Messengers** (e.g., Heraldo, Huitaca) transport control signaling
   between clans -- they route, deliver, and acknowledge, but they do
   not decide what work gets done.

2. **Dojos** manage the operational state -- backlogs, quest assignment,
   skill rosters, XP tracking, and business-mode inter-clan problem
   solving. They decide WHAT gets done and by WHOM, but they do not
   transport messages or execute work.

3. **Skills** execute the actual work -- writing code, auditing security,
   managing projects, designing protocols. They receive quests from the
   Dojo and report results, but they do not route messages or manage
   backlogs.

This triple separation is not merely organizational -- it enables
independent scaling, clear debugging boundaries, and a governance model
where each plane can evolve without breaking the others.

---

## 3. Terminology

| Term | Definition |
|------|-----------|
| **Control Plane (CP)** | The signaling layer responsible for message routing, delivery, discovery, and session management between clans. Operated by Messengers |
| **Orchestration Plane (OP)** | The decision layer responsible for quest dispatch, skill assignment, backlog management, and inter-clan governance. Operated by Dojos |
| **User Plane (UP)** | The execution layer where Skills perform actual work -- code, analysis, audits, designs. Skills process payloads and produce deliverables |
| **Messenger** | A clan's CP agent responsible for inter-clan signaling (e.g., Heraldo for Clan Momosho D, Huitaca for Clan JEI) |
| **Dojo** | A clan's OP agent responsible for orchestration. Each clan has a local Dojo; the Amaru Grand Dojo defines the standard they implement |
| **Skill** | A UP agent that executes work. Skills have profiles (capabilities, experience) and receive quests from the Dojo |
| **Grand Dojo** | The Amaru standard template for Dojo behavior -- defines quest format, dispatch protocol, XP system, and governance rules |
| **Quest** | A unit of work dispatched by the Dojo to one or more Skills |
| **Dojo Business Mode** | An operational mode where Dojos from multiple clans coordinate to solve real-world problems, matching Skills by profile |
| **Skill Roster** | The registry of available Skills within a clan, maintained by the Dojo |

### 4.1 The Three Planes

### 4.2 Why Three Planes, Not Two

ATR-X.200 Section 9 defines a two-plane CUPS model (CP + UP) at the
transport level. This is correct for message forwarding but collapses
two distinct concerns into the Control Plane:

1. **Signaling** (message routing, delivery) -- pure transport control.
2. **Orchestration** (quest dispatch, backlog) -- application-level
   decision making.

In 3GPP's 5G Core (TS 23.501), these ARE separate: the SMF (Session
Management Function) orchestrates sessions, while PFCP handles the
signaling between CP and UP. HERMES follows the same principle:

| Concern | Two-Plane (ATR-X.200 s9) | Three-Plane (ARC-2314) |
|---------|-------------------------|----------------------|
| Message routing | CP | CP (Messenger) |
| Delivery confirmation | CP | CP (Messenger) |
| Quest dispatch | CP (implicit) | OP (Dojo) |
| Backlog management | CP (implicit) | OP (Dojo) |
| Skill assignment | Not addressed | OP (Dojo) |
| Work execution | UP | UP (Skill) |
| Governance | Not addressed | OP (Dojo) |

The Orchestration Plane makes explicit what was previously implicit:
the Dojo is not merely a router -- it is a session manager, a scheduler,
and a governance engine.

## 5. Control Plane: Messengers

### 5.1 Responsibilities

The Messenger is a clan's dedicated CP agent. It handles ALL inter-clan
signaling and MUST NOT execute application-level work. Its responsibilities:

| Function | Description | Protocol Reference |
|----------|-------------|-------------------|
| **Route** | Determine the destination clan for outbound messages | ARC-0791 |
| **Signal** | Exchange hello/hello_ack for clan bootstrap | ARC-3022 s15 |
| **Deliver** | Write messages to the bus with correct addressing | ARC-5322 |
| **Acknowledge** | Manage ACK arrays for delivery confirmation | ARC-0793 |
| **Discover** | Query the Agora directory for clan profiles | ARC-2606 |
| **Translate** | NAT internal identities to external aliases | ARC-3022 s4 |
| **Filter** | Apply outbound/inbound rules at the gateway | ARC-1918 |

### 5.2 What Messengers Do NOT Do

- Messengers MUST NOT dispatch quests or assign Skills.
- Messengers MUST NOT execute application-level work.
- Messengers MUST NOT maintain backlog state.
- Messengers MUST NOT compute XP or Resonance.

A Messenger is a **postman**, not a **manager**. It delivers the mail;
it does not read it or act on it.

### 5.3 Message Types Owned by CP

| Message Type | Direction | Purpose |
|-------------|-----------|---------|
| `hello` | Outbound | Clan bootstrap handshake |
| `hello_ack` | Inbound | Response to hello |
| `state` | Both | Namespace state synchronization |
| `alert` | Both | Urgent signaling |
| `event` | Both | Informational events |

### 5.4 3GPP Mapping

| HERMES CP Component | 3GPP Equivalent | Specification |
|---------------------|-----------------|---------------|
| Messenger routing | NRF service discovery | TS 29.510 |
| hello/hello_ack | NF Registration/Deregistration | TS 29.510 s5 |
| ACK protocol | PFCP Session Report | TS 29.244 s7 |
| NAT translation | PFCP PDR/FAR rules | TS 29.244 s5 |
| Agora discovery | NRF NF Discovery | TS 29.510 s6 |

## 6. Orchestration Plane: Dojos

### 6.1 Responsibilities

The Dojo is a clan's OP agent. It manages the operational state of the
clan's Skill ecosystem. Its responsibilities:

| Function | Description |
|----------|-------------|
| **Dispatch** | Assign quests to Skills based on capability matching |
| **Roster** | Maintain the registry of available Skills and their profiles |
| **Backlog** | Prioritize pending quests by urgency, energy, and dependencies |
| **Track** | Monitor quest progress, completion, and outcomes |
| **Score** | Compute XP, Bounty, and arena metrics for internal Skills |
| **Govern** | Enforce clan rules, permissions, and escalation policies |
| **Coordinate** | In Business Mode, negotiate with other clans' Dojos for cross-clan quests |

### 6.2 Quest Lifecycle

```
  OP (Dojo)                    UP (Skill)
  --------                    ----------
  1. Receive request
     (from CP or human)
  2. Match Skill(s) by
     capability profile
  3. Create Quest
     {id, type, skills,
      priority, deadline}
                    ----dispatch--->
                                   4. Skill accepts quest
                                   5. Skill executes work
                                   6. Skill reports result
                    <---report----
  7. Dojo records outcome
  8. Dojo updates XP/Bounty
  9. Dojo ACKs to CP if
     cross-clan quest
```

### 6.3 Dojo Business Mode

Business Mode is an operational state where Dojos from multiple clans
coordinate to solve real-world problems. In Business Mode:

- Quests are formal: scoped, deadlined, with acceptance criteria.
- Skill matching is profile-based: capabilities, experience, Resonance.
- Results produce attestations (ARC-3022 s12) that affect Resonance.
- Governance rules apply: dispute resolution, escalation paths.

```
Clan A Dojo  <----CP signaling---->  Clan B Dojo
   |                                    |
   | dispatch                           | dispatch
   v                                    v
Skill A1                            Skill B1
(developer)                         (cybersec)
   |                                    |
   +------------ joint quest -----------+
   |                                    |
   v                                    v
Deliverable A                       Deliverable B
   |                                    |
   +----> Attestation A->B + B->A <-----+
```

**Example**: Clan Momosho D (developer profile) and Clan JEI
(cybersec/PM profile) collaborate on a security audit:

1. Clan A's Dojo identifies need: "security review of ARC-8446 draft"
2. Clan A's Messenger signals Clan B's Messenger (via Agora)
3. Clan B's Dojo evaluates: Skill `cybersec-architect` matches
4. Both Dojos create a joint quest with shared acceptance criteria
5. Skills execute independently, exchange deliverables via CP
6. Both Dojos record outcomes, issue cross-clan attestations

| HERMES OP Component | 3GPP Equivalent | Specification |
|---------------------|-----------------|---------------|
| Dojo dispatch | SMF session establishment | TS 23.502 s4.3 |
| Quest lifecycle | PDU Session lifecycle | TS 23.502 s4.3.1-4.3.4 |
| Skill matching | NRF NF profile matching | TS 29.510 s6.2 |

## 7. User Plane: Skills

### 7.1 Responsibilities

Skills are the UP agents that execute actual work. A Skill:

| Function | Description |
|----------|-------------|
| **Execute** | Perform the work assigned by the Dojo quest |
| **Report** | Return results, status, and completion to the Dojo |
| **Declare** | Publish capability profile for Dojo matching |
| **Learn** | Accumulate experience that refines future capability claims |

### 7.2 Skill Profile Schema

Each Skill declares a profile consumed by the Dojo for quest matching:

```json
{
  "skill_id": "protocol-architect",
  "clan_id": "momoshod",
  "capabilities": [
    "eng.protocol-design",
    "eng.telecom.3gpp",
    "research.standards",
    "eng.python"
  ],
  "experience": {
    "quests_completed": 47,
    "domains": ["telecom", "protocol", "open-source"],
    "arena_xp": 259
  },
  "availability": "active"
}
```

Capability taxonomy follows ARC-2606 Section 7 (9 top-level domains:
legal, finance, eng, creative, ops, research, edu, health, community).

### 7.3 What Skills Do NOT Do

- Skills MUST NOT route messages between clans.
- Skills MUST NOT dispatch quests to other Skills.
- Skills MUST NOT modify the backlog or XP system.
- Skills MUST NOT directly communicate with external clans (all
  external communication goes through CP/Messenger).

A Skill is a **worker**, not a **dispatcher** or a **postman**.

### 7.4 3GPP Mapping

| HERMES UP Component | 3GPP Equivalent | Specification |
|---------------------|-----------------|---------------|
| Skill execution | UPF packet processing | TS 29.244 s5.2 |
| Skill profile | NF Profile (capabilities) | TS 29.510 s6.2 |
| Quest result report | Usage Report (to SMF) | TS 29.244 s7.5 |
| Capability declaration | NF Service registration | TS 29.510 s5.2 |

## 8. The Amaru Grand Dojo

### 8.1 Purpose

The Grand Dojo is not a running instance -- it is the **standard
template** that defines what every clan-local Dojo MUST implement.
It is analogous to 3GPP TS 23.501 defining the Service-Based
Architecture: each mobile operator deploys their own 5G Core, but
all conform to the same standard.

### 8.2 What the Grand Dojo Defines

| Component | Specification | Status |
|-----------|--------------|--------|
| Quest format | JSON schema for quest objects | Defined (s8.3) |
| Dispatch protocol | How Dojos assign quests to Skills | Defined (s6.2) |
| XP system | How internal performance is scored | Defined (s8.4) |
| Arena system | Competitive/collaborative Skill evaluation | Defined (s8.5) |
| Governance rules | Permissions, escalation, dispute resolution | Defined (s8.6) |
| Business Mode protocol | How Dojos negotiate cross-clan quests | Defined (s6.3) |
| Roster format | Schema for Skill roster entries | Defined (s7.2) |

### 8.3 Quest Schema

```json
{
  "quest_id": "BR-003",
  "type": "battle_royale",
  "title": "Triple-Plane Architecture Formalization",
  "skills": ["protocol-architect", "telecom-mapper", "spec-drafter", "ref-engineer", "test-architect"],
  "priority": "high",
  "deadline": "2026-03-06",
  "acceptance_criteria": [
    "ARC-2314 spec complete and conformant",
    "Reference implementation in Python",
    "Test suite passing",
    "INDEX.md updated"
  ],
  "status": "in_progress",
  "xp_reward": 50
}
```

### 8.4 XP System

The Grand Dojo defines a standard XP system for internal Skill
evaluation:

| Metric | Formula | Visibility |
|--------|---------|-----------|
| **Quest XP** | `base_xp * difficulty_multiplier` | Clan-internal |
| **Arena XP** | `Sigma(quest_xp * peer_score)` | Clan-internal |
| **Bounty** | `XP * precision * impact` | Clan-internal |
| **Resonance** | `Sigma(attestation_score * recency * diversity)` | Public (via Gateway) |

Bounty and Resonance MUST remain separate (ARC-3022 Section 12):
- Bounty is computed by the Dojo (OP). Never exposed externally.
- Resonance is computed by the Gateway (CP). Based on external attestations.

### 8.5 Arena System

The Arena is a Dojo subsystem for structured Skill evaluation:

| Mode | Description | Participants |
|------|------------|-------------|
| **Solo (TR)** | Single Skill training quest | 1 Skill |
| **Co-Training (TR)** | Two Skills collaborate on training | 2 Skills |
| **Battle Royale (BR)** | Multi-Skill collaboration on real quest | 3+ Skills |
| **PvP** | Two Skills tackle same problem independently | 2 Skills |
| **Cross-Clan BR** | Skills from multiple clans on joint quest | 2+ clans |

Cross-Clan BR is a Business Mode operation requiring Dojo-to-Dojo
negotiation via CP.

### 8.6 Governance Rules

| Rule | Description |
|------|------------|
| **Human authority** | No irreversible action without human approval |
| **Skill sovereignty** | A clan's Dojo MUST NOT dispatch quests to another clan's Skills directly -- it MUST negotiate with the other clan's Dojo |
| **Escalation** | If two clan Dojos cannot agree on quest terms, either side MAY escalate to their human operator |
| **Attestation integrity** | Attestations MUST be issued by the Dojo, not by individual Skills |
| **Profile truthfulness** | Capability claims MUST be backed by quest history |

## 9. Inter-Plane Interfaces

### 9.1 CP-OP Interface (Messenger <-> Dojo)

The Messenger delivers incoming cross-clan messages to the Dojo for
orchestration decisions. The Dojo instructs the Messenger to send
outbound responses.

| Direction | Message Flow | Example |
|-----------|-------------|---------|
| CP -> OP | Inbound quest proposal from external clan | "Clan JEI proposes security review" |
| OP -> CP | Outbound quest acceptance/rejection | "Quest accepted, assigning cybersec-architect" |
| CP -> OP | Delivery confirmation (ACK) | "Clan JEI confirmed receipt" |
| OP -> CP | Attestation for delivery to external clan | "Attestation for Clan JEI's contribution" |

**Analogue**: In 3GPP, this is the N11 interface between AMF and SMF
(TS 23.502 s4.2).

### 9.2 OP-UP Interface (Dojo <-> Skill)

The Dojo dispatches quests to Skills and receives completion reports.

| Direction | Message Flow | Example |
|-----------|-------------|---------|
| OP -> UP | Quest dispatch with acceptance criteria | "Write ARC-2314 spec, English, formal" |
| UP -> OP | Progress report | "Section 1-4 complete, starting section 5" |
| UP -> OP | Completion report with deliverables | "ARC-2314.md written, 250 lines" |
| OP -> UP | Feedback and XP award | "Quest complete, 50 XP awarded" |

**Analogue**: In 3GPP, this is the N4/PFCP interface between SMF and
UPF (TS 29.244 s5).

### 9.3 CP-UP Interface (Messenger <-> Skill)

There is **no direct interface** between Messenger and Skill. All
communication between CP and UP goes through the OP (Dojo). This is
a fundamental architectural constraint:

```
   CP (Messenger)
        |
        | N11-equivalent
        v
   OP (Dojo)           <-- all decisions here
        |
        | N4-equivalent
        v
   UP (Skill)
```

This ensures that the Dojo always has visibility into what work is
being done and can enforce governance rules.

**Exception**: In degenerate deployments (single-agent clans), the
three planes MAY be collapsed into a single agent that performs all
roles. This is analogous to a combined SGW-C/SGW-U node in small
3GPP deployments.

## 10. Relationship to Existing Specs

### 10.1 ATR-X.200 (Reference Model)

ARC-2314 extends ATR-X.200 Section 9 (CUPS) from a two-plane to a
three-plane model. The original CUPS mapping remains valid at the
transport level:

- `bus.jsonl` = User Plane (data forwarding)
- `routes.md` = Control Plane (routing policy)

ARC-2314 operates at a **higher abstraction level** -- the operational
architecture of the Skill Gateway, not the transport of individual
messages. Both models coexist:

```
ARC-2314 (Operational Planes)     ATR-X.200 s9 (Transport Planes)
+----------------------------+    +----------------------------+
|  CP: Messenger             |    |                            |
|  OP: Dojo                  |    |  CP: routes.md + firewall  |
|  UP: Skill                 |    |  UP: bus.jsonl             |
+----------------------------+    +----------------------------+
     ^                                 ^
     |                                 |
 "Who does what"              "How messages flow"
```

### 10.2 ARC-3022 (Gateway Protocol)

The Gateway (ARC-3022) is a CP component -- it is operated by the
Messenger. ARC-2314 clarifies the Gateway's position in the plane
architecture and adds the Dojo as the orchestration counterpart.

### 10.3 ARC-2606 (Agent Profile & Discovery)

Agent profiles (ARC-2606) describe Skills (UP agents) for discovery
by external clan Dojos (OP). The capability ontology defined in
ARC-2606 is consumed by the Dojo for quest matching.

## 11. Deployment Models

### 11.1 Full Deployment (Recommended)

Each plane runs as a distinct agent or skill:

```
Clan Momosho D:
  CP: /heraldo (messenger)
  OP: /dojo (orchestrator)
  UP: /protocol-architect, /sales-engineering-director, ...

Clan JEI:
  CP: /huitaca (messenger)
  OP: /dojo-huitaca (orchestrator)
  UP: /cybersec-architect, /project-commander, ...
```

### 11.2 Minimal Deployment

For small clans (1-3 Skills), planes MAY be collapsed:

```
Clan Minimal:
  CP+OP: /dojo (handles both signaling and orchestration)
  UP: /my-skill
```

### 11.3 Federated Deployment

For large clans with specialized sub-teams:

```
Clan Enterprise:
  CP: /messenger (dedicated signaling)
  OP: /dojo-main (top-level orchestrator)
       +-- /dojo-eng (engineering sub-dojo)
       +-- /dojo-legal (legal sub-dojo)
  UP: [skills organized by sub-dojo]
```

### 11.4 Integrated OP-Strategy Deployment

For clans where the Dojo (OP) and the strategic layer are tightly coupled,
the Orchestration Plane MAY be integrated into the strategy skill:

```
Clan Integrated:
  CP: /messenger (dedicated signaling)
  OP+Strategy: /strategy-skill (orchestrator + deliberation)
  UP: [skills organized by domain or dimension]
```

In this model:
- The strategy skill absorbs Dojo functions: dispatch, quest lifecycle,
  roster management, and governance proposals.
- The `/dojo` command becomes an alias for the strategy skill's dispatch mode.
- Plane separation is preserved: CP (messenger) still cannot directly
  interact with UP (skills) — all traffic flows through the integrated OP.
- Audit functions MAY be distributed architecturally (e.g., firewall rules
  embedded in each skill's configuration) rather than concentrated in a
  single auditor agent.

This deployment model is valid under ARC-2314 because the specification
defines interfaces, not topology. The integrated OP-Strategy skill MUST
still implement:
1. Quest lifecycle (Section 6.2)
2. Skill Profile schema (Section 7.2)
3. Grand Dojo components (Section 8.2)
4. No-direct-CP-UP constraint (Section 9.3)

## 12. Conformance

An implementation conforms to ARC-2314 if it:

1. Separates Messenger (CP), Dojo (OP), and Skill (UP) responsibilities
   as defined in Sections 5-7.
2. Implements the Quest lifecycle defined in Section 6.2.
3. Implements the Skill Profile schema defined in Section 7.2.
4. Enforces the no-direct-CP-UP-interface constraint (Section 9.3).
5. Implements the Grand Dojo components listed in Section 8.2.
6. Maintains Bounty/Resonance separation (Section 8.4).

Partial implementations MUST declare which planes they implement and
which they collapse (Section 11.2).

---

## 13. Security Considerations

- **Plane isolation**: The OP (Dojo) MUST NOT expose Bounty or internal
  XP through the CP (Messenger). Only Resonance (computed from external
  attestations) is publicly visible.
- **Skill sovereignty**: A Dojo MUST NOT directly dispatch quests to
  Skills in another clan. All cross-clan quest coordination MUST go
  through the CP (Messenger-to-Messenger) and then each clan's OP
  (Dojo) independently.
- **Human authority**: Both CP and OP operations that affect external
  clans (sending messages, accepting quests, issuing attestations) MUST
  require human approval in the current phase.
- **Firewall inheritance**: All three planes inherit ARC-1918's
  default-deny model. Internal plane state is never exposed externally
  unless explicitly authorized by the operator.

## 15. Call Flows — Message by Message, Hop by Hop

This section traces complete inter-clan interactions through all three
planes, showing every message, every hop, and every decision point.
These flows serve as the reference for implementors and as onboarding

### 15.1 Call Flow 1: Clan Discovery (First Contact)

**Scenario**: Clan Momosho D discovers Clan JEI in the Agora and
initiates contact.

```
Clan Momosho D                         Agora                          Clan JEI
=================                     ========                       ================

[UP] protocol-architect
  "I need a cybersec review"
         |
         | (quest need)
         v
[OP] /dojo
  "No internal skill matches
   cybersec.audit — search Agora"
         |
         | (discover instruction)
         v
[CP] /heraldo
  "Query Agora for cybersec"
         |                    discover(capability=
         +------------------> "eng.cybersecurity")
                                    |
                                    | (match found:
                                    |  clan=huitaca,
                                    |  skill=cybersec-architect,
                                    |  resonance=4.2)
                                    |

## 16. The Diplomatic Protocol — Clan Sovereignty & Freedom

### 16.1 Philosophy: Sovereign Clans, Open Borders

HERMES treats each clan as a **sovereign nation** in a diplomatic
network. The analogy is precise:

| Diplomatic Concept | HERMES Equivalent |
|-------------------|-------------------|
| Sovereign nation | Clan (own bus, own dojo, own skills) |
| Embassy / Ambassador | Messenger (Heraldo, Huitaca) |
| Diplomatic pouch | Encrypted/filtered bus messages |
| Treaty | Cross-clan quest agreement |
| Diplomatic recognition | Peer registration in gateway.json |
| Passport / Visa | External identity (NAT alias) |
| UN General Assembly | The Agora (public directory) |
| Bilateral agreement | Dojo-to-Dojo quest negotiation |
| Diplomatic immunity | Firewall: internal state never exposed |
| Cultural exchange | Best practices sharing between clans |

### 16.2 Freedom to Fork, Branch, and Experiment

Each clan has **absolute freedom** over its internal architecture:

```
HERMES Standard (Grand Dojo)
         |
         | "implement the interfaces,
         |  build whatever you want inside"
         |
    +----+----+----+----+
    |         |         |
Clan A    Clan B    Clan C
(28 skills, (3 skills, (50 skills,
 6 dims,    1 dim,     custom arena,
 RPG dojo)  minimal)   ML-powered
                       dispatch)
```

**What a clan MUST implement** (the "diplomatic protocol"):
- Messenger interface (CP): hello/hello_ack, message format (ARC-5322)
- Quest proposal/acceptance format (OP): standard schema (s8.3)
- Skill profile schema (UP): capability taxonomy (ARC-2606)

**What a clan is FREE to customize**:
- Internal Dojo logic: quest dispatch algorithm, priority model
- Skill architecture: number, naming, organization, tools
- Arena system: custom game modes, XP formulas, training methods
- Topology: flat, hierarchical, dimensional, matrix
- Internal bus format: extensions beyond base schema
- Tooling: any language, any framework, any MCP configuration
- Governance: consensus, hierarchy, democratic, AI-assisted

#### Organizational Orthogonality

Clans may organize internally along different axes. Two common patterns:

- **Function-based**: Namespaces represent organizational functions
  (strategy, legal, security, operations, gateway, controller). Each
  namespace contains agents specialized in that function across all domains.

- **Dimension-based**: Namespaces represent life/business dimensions
  (corporate, personal, finance, housing, editorial, protocol). Each
  dimension contains all functions relevant to that domain.

These are **orthogonal axes** of the same problem. A function-based clan
and a dimension-based clan can interoperate seamlessly because the Agora
sees only aggregated capabilities — internal organization is invisible to
peers (Section 16.3).

The standard does not prescribe either pattern. Hybrid models (e.g.,
function-based for shared services, dimension-based for domain-specific
work) are equally valid.

### 16.3 Clan as Super Skill

In the Agora, each clan appears as a **single entity** with aggregated
capabilities — a "super skill" that represents the collective power
of all its internal skills:

```

### 16.4 Best Practice Exchange Protocol

Clans MAY exchange operational best practices through a dedicated
message type:

```json
{
  "src": "momoshod",
  "dst": "huitaca",
  "type": "data_cross",
  "msg": "PRACTICE:arena_system. BR+PvP+TR modes. XP formula: base*difficulty. Helps skill growth. Ref:ARC-2314_s8.5",
  "ttl": 14,
  "ack": []
}
```

Best practice exchange is:
- **Voluntary**: no clan is obligated to share or adopt.
- **Non-prescriptive**: the sending clan shares what works for them;
  the receiving clan adapts as they see fit.
- **Bidirectional**: both clans learn. Momosho D might adopt Huitaca's
  threat modeling methodology; Huitaca might adopt Momosho D's arena
  system.

### 16.5 Parallel Environments

A clan MAY create a **parallel environment** — a fork of the Amaru
standard with experimental extensions:

```
HERMES (canonical)
    |
    +-- Clan Momosho D (standard + RPG dojo + dimensional model)
    |
    +-- Clan JEI (standard + security-first extensions)
    |       |
    |       +-- huitaca-crypto-bus (experimental: PQC-signed bus)
    |       +-- huitaca-governance (experimental: DAO-style decisions)
    |
    +-- Clan X (standard + ML dispatch + auto-topology)
```

Rules for parallel environments:
1. The **base protocol** (message format, hello/hello_ack, quest schema)
   MUST remain compatible — or the clan cannot participate in the Agora.
2. Extensions are **opt-in**: a clan advertises experimental capabilities
   in its profile; other clans choose whether to engage with them.
3. Successful experiments MAY be **proposed as new ARCs** — this is how
   the standard evolves. Clan innovation feeds the commons.
4. Failed experiments are **local**: they never affect other clans.

## 17. Onboarding: How a New Clan Joins

### 17.1 Step-by-Step Integration

```
Step 1: Install HERMES
  $ git clone https://github.com/amaru-protocol/amaru
  $ cd hermes && hermes init --clan-id huitaca --display-name "Clan JEI"

Step 2: Define your Messenger
  The Messenger is your clan's ambassador.
  Name it, configure it, give it personality.
  Example: /huitaca — Jeimmy's messenger to the world.

Step 3: Set up your Dojo
  The Dojo is your internal orchestrator.
  Configure it with YOUR rules, YOUR quest types, YOUR XP system.
  The Grand Dojo standard is the minimum — extend freely.

Step 4: Register your Skills
  Each Skill gets a profile with capabilities.
  Use ARC-2606 taxonomy or extend it.
  Example:
    cybersec-architect: eng.cybersecurity, eng.crypto.pqc
    project-commander: ops.project-management.pmp, ops.governance

Step 5: Publish to Agora
  $ hermes publish
  Your clan profile appears in the Agora directory.
  Other clans can now discover you.

Step 6: Peer with another clan
  $ hermes peer --clan-id momoshod
  Exchange hello/hello_ack.
  Add each other as known peers.

Step 7: Collaborate
  Receive quest proposals, evaluate, accept or reject.
  Complete quests, exchange deliverables, issue attestations.
  Build Resonance over time.
```
