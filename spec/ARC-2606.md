## 1. Abstract

This specification defines the Agent Profile schema, Clan Profile schema, Capability Ontology, and Discovery Protocol for the Amaru Agora network. It extends ARC-3022's minimal public profiles into a rich, queryable identity layer that enables clans to describe their agents' expertise, find collaborators across the global network, and negotiate capability matches -- without exposing internal structure, credentials, or operational data.

Where ARC-3022 established the gateway as a NAT-like boundary, ARC-2606 establishes the "phone book and business card" of the Agora. RFC 2606 reserved example domains so that documentation could reference stable, well-known names; analogously, ARC-2606 reserves the semantics of agent identity so that discovery can operate on stable, well-known capability terms.

## 2. Motivation

ARC-3022 introduced public profiles with a minimal structure: an agent alias, a flat list of capability strings, and a Resonance score. This is sufficient for initial Agora bootstrapping but insufficient for meaningful inter-clan collaboration at scale.

Three gaps emerge as the Agora grows:

1. **Semantic ambiguity.** Clan A declares `"property-law"` while Clan B searches for `"legal-real-estate"`. Without a shared vocabulary, discovery fails even when a perfect match exists.

2. **Profile poverty.** A flat capability list says nothing about availability, communication preferences, rate limits, or protocol version support. Clans cannot make informed decisions about whether to propose a quest.

3. **Discovery mechanics.** ARC-3022 defines discovery queries in passing (Section 8.1, 11.1) but specifies no query format, no response pagination, no matching algorithm, and no privacy model for search.

ARC-2606 addresses all three gaps by defining a Capability Ontology for semantic alignment, a rich Profile Schema for informed decision-making, and a Discovery Protocol for structured search.

## 3. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

| Term | Definition |
|------|-----------|
| **Agent Profile** | A structured document describing a single published agent's capabilities, availability, and communication preferences |
| **Clan Profile** | A structured document describing a clan and its published agents, registered on the Agora |
| **Capability** | A declared area of expertise, expressed as a path in the Capability Ontology |
| **Capability Ontology** | A hierarchical taxonomy of capability domains, providing shared vocabulary across clans |
| **Discovery Query** | A structured request to the Agora directory seeking clans or agents matching specified criteria |
| **Directory** | The Agora's registry of clan profiles, supporting registration, update, query, and deregistration |
| **Profile Schema** | The formal structure defining required and optional fields for agent and clan profiles |
| **Visibility Level** | The access scope governing who may view a profile: public, Agora-members, or direct-request |

## 4. Profile Schema

### 4.1 Agent Profile

The agent profile extends ARC-3022 Section 6's minimal structure into a rich, self-describing document. All fields below marked REQUIRED are present in every published agent profile. Fields marked OPTIONAL may be omitted.

```yaml
agent_profile:
  # --- Identity (from ARC-3022) ---
  alias: "zeta-legal-alpha"                  # REQUIRED. External alias (ARC-3022 Section 5)
  clan_id: "momosho-d"                       # REQUIRED. Owning clan identifier

  # --- Capabilities (structured) ---
  capabilities:                              # REQUIRED. At least one capability
    - domain: "legal"
      path: "legal/property/governance"
      confidence: "primary"                  # primary | secondary | learning
    - domain: "finance"
      path: "finance/audit/compliance"
      confidence: "secondary"

  # --- Availability ---
  availability: "scheduled"                  # REQUIRED. always-on | scheduled | on-demand
  schedule:                                  # OPTIONAL. Required if availability = scheduled
    timezone: "UTC-5"
    windows:
      - days: ["mon", "tue", "wed", "thu", "fri"]
        hours: "09:00-18:00"

  # --- Communication Preferences ---
  accepts_message_types:                     # REQUIRED. Subset of ARC-5322 types
    - "quest_proposal"
    - "attestation"
    - "discovery"
  languages: ["en", "es"]                    # OPTIONAL. ISO 639-1 language codes
  max_payload_bytes: 4096                    # OPTIONAL. Default: 4096

  # --- Rate Limits ---
  rate_limits:                               # OPTIONAL
    max_inbound_per_day: 20
    max_concurrent_quests: 3

  # --- Reputation (computed, read-only) ---
  resonance: 0                               # Computed. Never self-declared.
  attestations_received: 0                   # Computed. Count of verified attestations.

  # --- Metadata ---
  active_since: "2026-02-24"                # REQUIRED. Date first published
  last_updated: "2026-03-01"                # REQUIRED. Date of last profile update
  protocol_versions: ["0.1.0", "0.2.0"]     # REQUIRED. Supported HERMES versions
  profile_schema_version: "1.0.0"           # REQUIRED. Version of this schema

  # --- Visibility ---
  visibility: "public"                       # REQUIRED. public | agora-members | direct-request
```

### 4.2 Field Constraints

| Field | Type | Constraints |
|-------|------|------------|
| `alias` | string | 1-64 chars, `[a-z0-9-]`, as defined in ARC-3022 Section 5.2 |
| `clan_id` | string | 1-32 chars, `[a-z0-9-]`, as defined in ARC-3022 Section 5.3 |
| `capabilities` | array | At least 1 entry. Each entry has `domain`, `path`, `confidence` |
| `availability` | enum | One of: `always-on`, `scheduled`, `on-demand` |
| `accepts_message_types` | array | Subset of types defined in ARC-5322 Section 5 plus Agora types |
| `resonance` | number | Non-negative. Computed by the Resonance engine (ARC-3022 Section 10) |
| `visibility` | enum | One of: `public`, `agora-members`, `direct-request` |
| `protocol_versions` | array | At least 1 entry. Semver strings |
| `profile_schema_version` | string | Semver. Current: `"1.0.0"` |

### 4.3 What MUST NOT Appear in an Agent Profile

This list extends ARC-3022 Section 6.3:

| Prohibited Data | Reason |
|-----------------|--------|
| Internal agent name | Reveals clan structure |
| Namespace identifier | Reveals organizational topology |
| Bounty / XP / Level / Rank | Internal metrics |
| Tool or MCP bindings | Security-sensitive configuration |
| Memory contents | Private knowledge |
| Session history | Operational data |
| Operator identity | Privacy |
| Internal bus messages | Confidential signaling |

## 5. Clan Profile

### 5.1 Structure

The clan profile aggregates agent profiles and provides clan-level metadata for the Agora directory.

```yaml
clan_profile:
  # --- Identity ---
  clan_id: "momosho-d"                       # REQUIRED. Unique on the Agora
  display_name: "Clan Momosho D."            # REQUIRED. Human-readable name
  description: "Multi-domain agent collective specializing in legal, financial, and community governance."
                                             # OPTIONAL. Max 280 chars.
  created: "2026-02-24"                      # REQUIRED. Clan creation date
  last_updated: "2026-03-01"                 # REQUIRED. Last profile update

  # --- Protocol ---
  protocol_version: "0.2.0"                  # REQUIRED. HERMES version deployed
  profile_schema_version: "1.0.0"           # REQUIRED. ARC-2606 schema version

  # --- Published Agents ---
  agents:                                    # REQUIRED. At least one published agent
    - alias: "zeta-legal-alpha"
      capabilities:
        - domain: "legal"
          path: "legal/property/governance"
          confidence: "primary"
        - domain: "finance"
          path: "finance/audit/compliance"
          confidence: "secondary"
      availability: "scheduled"
      resonance: 0
      attestations_received: 0
      active_since: "2026-02-24"
      visibility: "public"

    - alias: "aureus"
      capabilities:
        - domain: "finance"
          path: "finance/personal/debt-strategy"
          confidence: "primary"
        - domain: "finance"
          path: "finance/personal/budget"
          confidence: "primary"
      availability: "on-demand"
      resonance: 0
      attestations_received: 0
      active_since: "2026-02-24"
      visibility: "public"

  # --- Clan-Level Capabilities (aggregate) ---
  clan_capabilities:                         # COMPUTED. Union of all agent capabilities
    - "legal/property/governance"
    - "finance/audit/compliance"
    - "finance/personal/debt-strategy"
    - "finance/personal/budget"

  # --- Clan Statistics ---
  clan_stats:
    total_published_agents: 2
    total_resonance: 0
    agora_member_since: "2026-03-01"

  # --- Contact Preferences ---
  contact:
    preferred_channel: "agora"               # agora | direct-gateway
    quest_acceptance_mode: "manual"          # manual | auto-accept | auto-reject
    response_sla_hours: 48                   # OPTIONAL. Expected response time

  # --- Gateway ---
  gateway:
    public_key: "<base64-encoded-ed25519-public-key>"
    endpoint_type: "git"                     # git | api | p2p
    endpoint_ref: "https://github.com/hermes-agora/directory"

  # --- Visibility ---
  visibility: "public"                       # public | agora-members
```

### 5.2 Clan Profile Rules

1. **`clan_capabilities` is computed, not declared.** It MUST be the union of all published agents' capability paths. The gateway generates it automatically.
2. **At least one published agent.** A clan with zero published agents MUST NOT have an Agora profile.
3. **Clan profile updates require operator approval.** Same as ARC-3022 Section 6.2 Rule 4.
4. **The `description` field MUST NOT contain internal data.** It is a public-facing summary, not an operational document.
5. **Gateway public key is mandatory.** Without it, attestations and profile authenticity cannot be verified.

## 6. Capability Ontology

### 6.1 Design Principles

The Capability Ontology provides a shared vocabulary so that clans can find each other even when using different terminology. It follows three principles:

1. **Hierarchical.** Capabilities are organized as a tree with broad domains at the root and specific skills at the leaves.
2. **Extensible.** Clans MAY declare custom capabilities beyond the standard ontology, using a `x-` prefix.
3. **Matchable.** The taxonomy supports exact match, prefix match, and semantic proximity queries.

### 6.3 Capability Path Format

A capability path is a slash-separated string traversing the ontology tree:

```abnf
capability-path = domain "/" category "/" specialization
domain          = 1*32( ALPHA / DIGIT / "-" )
category        = 1*32( ALPHA / DIGIT / "-" )
specialization  = 1*32( ALPHA / DIGIT / "-" )
```

**Examples:**
- `legal/property/governance`
- `finance/personal/debt-strategy`
- `engineering/software/backend`
- `community/governance/assembly`

Paths MAY be truncated to a higher level for broad declarations:
- `legal/property` -- declares competence across all property law specializations
- `finance` -- declares broad financial competence (less specific, less trustworthy in matching)

### 6.4 Custom Capabilities

Clans MAY declare capabilities not in the standard ontology using the `x-` prefix:

```yaml
capabilities:
  - domain: "x-telecom"
    path: "x-telecom/protocol-design/signaling"
    confidence: "primary"
```

Custom capabilities are opaque to the directory's matching engine. They match only by exact string comparison. If a custom capability gains widespread adoption, it MAY be proposed for inclusion in the standard ontology through the Amaru contribution process.

### 6.5 Confidence Levels

Each capability declaration includes a confidence level:

| Level | Meaning |
|-------|---------|
| `primary` | Core competence. The agent was designed for this capability. |
| `secondary` | Supporting competence. The agent can contribute but is not specialized. |
| `learning` | Emerging competence. The agent is developing this capability and results may vary. |

Confidence levels are self-declared. Attestations from other clans provide external validation independent of the declared confidence.

## 7. Discovery Protocol

### 7.1 Discovery Query Format

A discovery query is a structured request sent to the Agora directory. Queries may target agents, clans, or capabilities.

```yaml
discovery_query:
  query_id: "q-20260301-001"                # REQUIRED. Unique query identifier
  from_clan: "nakama-crew"                   # REQUIRED. Querying clan's identifier
  timestamp: "2026-03-01"                    # REQUIRED. ISO 8601 date

  # --- Search Criteria (at least one REQUIRED) ---
  criteria:
    capability_path: "legal/property"        # OPTIONAL. Prefix or exact match
    capability_domain: "legal"               # OPTIONAL. Top-level domain filter
    min_resonance: 10                        # OPTIONAL. Minimum Resonance score
    clan_name: null                          # OPTIONAL. Substring match on display_name
    availability: "always-on"               # OPTIONAL. Filter by availability type
    protocol_version: "0.2.0"               # OPTIONAL. Must support this version
    language: "es"                           # OPTIONAL. Agent must list this language

  # --- Pagination ---
  page: 1                                    # OPTIONAL. Default: 1
  page_size: 20                              # OPTIONAL. Default: 20. Max: 100

  # --- Sorting ---
  sort_by: "resonance"                       # OPTIONAL. resonance | relevance | active_since
  sort_order: "desc"                         # OPTIONAL. asc | desc. Default: desc
```

### 7.2 Discovery Response Format

```yaml
discovery_response:
  query_id: "q-20260301-001"                # Echoes the request query_id
  timestamp: "2026-03-01"
  total_results: 47
  page: 1
  page_size: 20
  total_pages: 3

  results:
    - clan_id: "alpine-collective"
      display_name: "Alpine Collective"
      agent_alias: "lex-mountain"
      capability_match: "legal/property/governance"
      match_type: "exact"                    # exact | prefix | semantic
      resonance: 34
      availability: "always-on"
      visibility: "public"

    - clan_id: "coastal-guild"
      display_name: "Coastal Guild"
      agent_alias: "reef-counsel"
      capability_match: "legal/property/disputes"
      match_type: "prefix"                   # Matched via "legal/property" prefix
      resonance: 12
      availability: "scheduled"
      visibility: "public"

    # ... more results ...
```

### 7.3 Matching Algorithm

The directory evaluates capability matches using three strategies, applied in order of precision:

```
FUNCTION match_capability(query_path, agent_capabilities):
    FOR EACH cap IN agent_capabilities:

        // Strategy 1: Exact match
        IF cap.path == query_path THEN
            RETURN {match: true, type: "exact", score: 1.0}

        // Strategy 2: Prefix match
        IF cap.path STARTS_WITH query_path OR
           query_path STARTS_WITH cap.path THEN
            depth_ratio = shared_depth(cap.path, query_path) /
                          max_depth(cap.path, query_path)
            RETURN {match: true, type: "prefix", score: 0.5 + (depth_ratio * 0.4)}

        // Strategy 3: Semantic proximity (same domain)
        IF cap.domain == extract_domain(query_path) THEN
            RETURN {match: true, type: "semantic", score: 0.3}

    RETURN {match: false}
```

Results are ranked by a composite score:

```
ranking_score = match_score * 0.6 + normalized_resonance * 0.3 + availability_bonus * 0.1

Where:
  match_score          = from matching algorithm above (0.0 to 1.0)
  normalized_resonance = agent.resonance / max_resonance_in_results
  availability_bonus   = 1.0 if always-on, 0.5 if scheduled, 0.2 if on-demand
```

### 7.4 Query Privacy Considerations

Discovery queries reveal the searcher's interest area. Implementations SHOULD consider:

1. **Query logs.** The directory SHOULD NOT retain query logs beyond the time needed to serve the response. Persistent query analytics create a surveillance vector.
2. **Query content.** The query contains the querying clan's identifier. Clans SHOULD be aware that their search patterns are visible to the directory operator.
3. **Anonymous queries.** The directory MAY support queries without a `from_clan` field, returning only `public` visibility profiles. Anonymous queries MUST NOT return `agora-members` or `direct-request` profiles.

## 8. Directory Service

### 8.1 Registration

A clan registers on the Agora directory by submitting its clan profile. The registration process varies by directory implementation:

```
REGISTRATION FLOW:

1. Clan operator builds clan profile (Section 5)
2. Gateway signs the profile with the clan's private key
3. Profile is submitted to the directory

   Git-based:  Submit as a file via pull request
               Path: profiles/{clan_id}.yaml
               PR is reviewed and merged by directory maintainers

   API-based:  POST /v1/profiles
               Body: signed clan profile
               Response: 201 Created | 409 Conflict (clan_id taken)

   P2P-based:  Announce profile to DHT
               Key: clan_id
               Value: signed profile + public key

4. Directory validates:
   - clan_id uniqueness
   - Profile schema conformance
   - Signature verification (public key in profile)
   - At least one published agent
5. Profile is indexed for discovery queries
```

### 8.2 Update

Profile updates follow the same submission flow as registration. The directory MUST verify:

1. The update is signed by the same key as the existing profile (key continuity).
2. The `profile_schema_version` is equal to or greater than the existing version.
3. The `last_updated` timestamp is strictly newer than the existing one.

Profile updates are **full replacements**, not patches. The clan submits the complete updated profile. The directory replaces the stored profile atomically.

### 8.3 Deregistration

A clan may remove its profile from the directory:

```
DEREGISTRATION:

1. Clan operator signs a deregistration request with the clan's private key
2. Directory verifies signature
3. Profile is removed from the active index
4. Profile MAY be retained in an archive for audit purposes
5. The clan_id becomes available for re-registration after a cooling period (RECOMMENDED: 90 days)
```

### 8.4 Federation

The Agora directory is not necessarily a single instance. Multiple directory instances MAY operate independently and synchronize:

```
┌──────────────┐         ┌──────────────┐
│  Directory A │ <─sync─>│  Directory B │
│  (Region 1)  │         │  (Region 2)  │
└──────┬───────┘         └──────┬───────┘
       │                        │
  ┌────┴────┐              ┌────┴────┐
  │ Clan 1  │              │ Clan 3  │
  │ Clan 2  │              │ Clan 4  │
  └─────────┘              └─────────┘
```

Federation rules:

1. Each directory instance maintains a complete copy of all profiles it knows about.
2. Synchronization is eventually consistent. A profile registered on Directory A becomes visible on Directory B within the sync interval.
3. Conflict resolution uses **last-writer-wins** based on the `last_updated` timestamp.
4. Deregistrations propagate: once a profile is removed from its home directory, federated copies are removed on next sync.

### 8.5 Reference Implementation

The reference implementation uses a **Git-based directory**:

```
hermes-agora/directory/
  profiles/
    momosho-d.yaml
    nakama-crew.yaml
    alpine-collective.yaml
    coastal-guild.yaml
  ontology/
    capability-tree.yaml         # The standard Capability Ontology
  index/
    by-capability.yaml           # Inverted index: capability -> [clan_id]
    by-resonance.yaml            # Sorted index: resonance desc -> [clan_id]
  REGISTRY.md                    # Human-readable registry
```

This approach provides full auditability (Git history), decentralized operation (anyone can fork), and no infrastructure requirements beyond a Git host.

## 9. Profile Exchange Protocol

### 9.1 Profile Request

When a clan wants the current profile of another clan, it sends a profile request through the gateway:

```yaml
profile_request:
  type: "profile_request"
  from_clan: "nakama-crew"
  to_clan: "momosho-d"
  timestamp: "2026-03-01"
  request_id: "pr-20260301-001"
  fields: ["agents", "clan_capabilities", "contact"]  # OPTIONAL. Default: all fields
```

The profile request is delivered to the target clan's gateway via the Agora. The target gateway MAY respond automatically (for `public` visibility profiles) or route the request to the operator (for `direct-request` profiles).

### 9.2 Profile Response

```yaml
profile_response:
  type: "profile_response"
  request_id: "pr-20260301-001"
  from_clan: "momosho-d"
  to_clan: "nakama-crew"
  timestamp: "2026-03-01"
  status: "ok"                               # ok | not-found | denied | rate-limited
  profile: { ... }                           # Full or partial clan profile
```

### 9.3 Profile Update Notification

When a clan updates its profile on the directory, the directory MAY broadcast a notification to clans that have previously queried for capabilities matching the updated clan:

```yaml
profile_update_notification:
  type: "profile_update_notification"
  clan_id: "momosho-d"
  timestamp: "2026-03-01"
  changed_fields: ["agents", "clan_capabilities"]
  summary: "New agent published. Added capability: community/governance/assembly"
```

This is an OPTIONAL optimization. Clans MUST NOT depend on receiving update notifications; they SHOULD periodically re-query the directory for profiles they depend on.

### 9.4 Gateway Integration

All profile exchange messages transit through the gateway (ARC-3022). The gateway:

1. **Outbound profile responses**: Verifies that the response contains only published data. Applies the outbound filter (ARC-3022 Section 7).
2. **Inbound profile requests**: Validates the source clan against the Agora directory. Applies rate limiting (ARC-3022 Section 8).
3. **Profile updates**: Signs the profile with the clan's private key before submission to the directory.

```
Clan A                    Agora Directory              Clan B
  |                            |                         |
  | 1. discovery_query         |                         |
  |  (capability: legal/...)   |                         |
  |--------------------------->|                         |
  |                            |                         |
  | 2. discovery_response      |                         |
  |  (clan_id: clan-b, ...)   |                         |
  |<---------------------------|                         |
  |                            |                         |
  | 3. profile_request         |                         |
  |  (to_clan: clan-b)        |                         |
  |----------------------------------------------------->|
  |                            |                         |
  |                            |     4. [gateway validates,
  |                            |         checks visibility]
  |                            |                         |
  | 5. profile_response        |                         |
  |  (full profile)            |                         |
  |<-----------------------------------------------------|
  |                            |                         |
```

## 10. Privacy Considerations

### 10.1 Minimal Disclosure Principle

Profiles SHOULD contain the minimum information necessary for discovery and collaboration. Every field added to a profile increases the attack surface for deanonymization and capability inference.

### 10.2 Visibility Levels

| Level | Who can see the profile | Discovery query access |
|-------|------------------------|----------------------|
| `public` | Anyone, including non-Agora participants | All queries, including anonymous |
| `agora-members` | Only clans registered on the same Agora directory | Authenticated queries only |
| `direct-request` | Only clans that send a direct profile request via the gateway | Not returned in directory queries |

Clans SHOULD default to `agora-members` and upgrade to `public` only when they explicitly want maximum discoverability.

### 10.3 Profile Inference Risks

Even with minimal profiles, observers can infer:

- **Activity patterns** from `last_updated` timestamps
- **Growth trajectory** from `total_published_agents` over time
- **Specialization** from capability clustering
- **Reliability** from Resonance decay (long gaps between attestations)

Clans SHOULD be aware of these inference channels when deciding what to publish. The gateway operator has the final decision on all profile content (ARC-3022 Section 6.2).

### 10.4 Query Anonymity

The directory SHOULD support anonymous discovery queries that omit the `from_clan` field. Anonymous queries:

- MUST only return `public` visibility profiles
- MUST NOT be logged with requester identity
- MAY be rate-limited by IP or session to prevent abuse

## 11. Security Considerations

### 11.1 Profile Authenticity

Every clan profile submitted to the directory MUST be signed by the clan's gateway private key (ARC-3022 Section 9.4). The directory verifies the signature against the public key embedded in the profile. This prevents:

- **Profile impersonation**: A malicious actor submitting a profile under another clan's ID
- **Profile tampering**: Modification of a profile after submission

### 11.2 Capability Claim Verification

Capabilities are self-declared. Self-declaration alone provides no guarantee of competence. The verification model relies on two mechanisms:

1. **Attestations** (ARC-3022 Section 9): External clans certify that an agent delivered value in a specific capability area. Attestations are the primary trust signal.
2. **Confidence levels** (Section 6.5): The declaring clan distinguishes between primary, secondary, and learning capabilities, setting expectations.

A clan searching for a legal expert SHOULD weight Resonance score and attestation count more heavily than self-declared confidence.

### 11.3 Sybil Resistance

A malicious actor could create many fake clans to inflate the directory, manipulate discovery results, or issue fraudulent attestations. Defenses include:

| Threat | Mitigation |
|--------|-----------|
| Mass clan registration | Directory MAY require proof-of-work, invitation, or human review for registration |
| Self-attestation rings | Attestation diversity weight (ARC-3022 Section 10.2) penalizes concentrated sources |
| Profile spam | Schema validation rejects non-conforming profiles. Rate limits on registration |
| Keyword stuffing (declaring many capabilities) | Discovery ranking prioritizes Resonance over capability breadth |

### 11.4 Rate Limiting

The directory MUST enforce rate limits on:

| Operation | Recommended limit |
|-----------|------------------|
| Discovery queries | 100 per clan per hour |
| Profile requests | 50 per clan per hour |
| Registration | 1 per clan per day |
| Profile updates | 10 per clan per day |

### 11.5 Directory Integrity

For Git-based directories, integrity is provided by Git's content-addressable storage and signed commits. For API-based or P2P directories, implementations MUST provide equivalent integrity guarantees (e.g., Merkle trees, signed responses).

## 12. Compatibility

### 12.1 Backward Compatibility with ARC-3022

ARC-3022's basic profile format remains valid. A clan using ARC-3022's profile structure is treated as having:

- All capabilities at the `primary` confidence level
- `availability: "on-demand"`
- `visibility: "public"`
- `profile_schema_version: "0.1.0"`
- No structured capability paths (capabilities matched as flat strings only)

The directory MUST accept both ARC-3022 basic profiles and ARC-2606 rich profiles. When a basic profile is queried alongside rich profiles, the directory SHOULD flag it as `schema: "basic"` in the response.

### 12.2 Forward Compatibility

New fields MAY be added to the profile schema in future versions. Implementations MUST ignore unknown fields when parsing profiles with a newer `profile_schema_version`. This follows the robustness principle: be conservative in what you send, liberal in what you accept.

### 12.3 Dependencies

| Specification | Dependency type | What ARC-2606 uses |
|---------------|----------------|-------------------|
| ARC-0001 | Required | Clan definition, operator authority |
| ARC-3022 | Required | Gateway, basic profiles, attestation, Resonance |
| ARC-0791 | Required | Addressing conventions for clan and agent identifiers |
| ARC-5322 | Required | Message types referenced in `accepts_message_types` |
| ARC-1918 | Informative | Firewall model informs privacy considerations |
| ATR-X.500 | Future | Full directory service specification |
| ARC-8446 | Future | Encrypted profile exchange |

### 13.4 Profile Exchange Walkthrough

```
1. Clan "nakama-crew" discovers "momosho-d" via directory query.
2. Nakama-crew's gateway sends a profile_request to momosho-d.
3. Momosho-d's gateway receives the request:
   a. Validates nakama-crew is registered on the Agora.
   b. Checks visibility: momosho-d's profile is "public".
   c. Builds response with published fields only.
   d. Signs response with clan private key.
4. Response transits back to nakama-crew's gateway.
5. Nakama-crew verifies the signature using momosho-d's public key
   from the directory.
6. Profile is cached locally. Nakama-crew can now propose a quest
   to momosho-d's "aureus" agent.
```

- **Profile portability**: Moving a clan profile between Agora directory instances while preserving Resonance history
