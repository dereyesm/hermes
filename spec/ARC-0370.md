# ARC-0370: Auto-Peer Discovery Protocol

| Field        | Value                                                                                                     |
|--------------|-----------------------------------------------------------------------------------------------------------|
| **Number**   | ARC-0370                                                                                                  |
| **Title**    | Auto-Peer Discovery Protocol                                                                              |
| **Lineage**  | SSH TOFU (RFC 4251 §4.1), Signal Safety Numbers, mDNS (RFC 6762), Certificate Transparency (RFC 6962)    |

---

> *"In SSH, the first connection is an act of faith — you accept the key, and
> from that moment forward, any change is a betrayal. HERMES adopts the same
> discipline: trust once, verify always. The hub vouches for the introduction;
> the operators verify when it matters."*

## 1. Abstract

This specification defines a four-level auto-peer discovery protocol for
Amaru clans. The protocol enables agents to discover and register peers
autonomously via hub presence events, eliminating the need for manual
`amaru peer add` commands while preserving sovereign-first principles.

The four levels provide increasing trust guarantees:

| Level | Name | Trust Model | Infrastructure | Status |
|-------|------|-------------|----------------|--------|
| L0 | TOFU via Hub | Hub vouches for peer key | Hub required | IMPLEMENTED |
| L1 | Safety Numbers | Human verifies fingerprint | None (out-of-band) | AVAILABLE |
| L2 | Federation Gossip | Audit log of peer introductions | Hub + log | PLANNED |
| L3 | LAN Discovery | Zero-config multicast | LAN only | PLANNED |

Each level is optional and additive. L0 is the default. A clan with no hub
operates in manual mode (existing `amaru peer add` behavior).

## 2. Motivation

### 2.1 The Cold Start Problem

When two clans connect via hub for the first time, they can exchange messages
immediately (the hub authenticates connections via Ed25519 challenge-response
per ARC-4601 §15). However, the local `gateway.json` does not know about the
remote peer — `amaru status` shows "No peers", and the Agent Node cannot
apply peer-specific policies.

This creates a paradox: clans can *communicate* before they formally *know*
each other.

### 2.2 Human Bottleneck

Manual peer registration (`amaru peer add`) requires operator intervention.
For autonomous agent networks, this creates a dependency on human availability
that contradicts the protocol's goal of liberating human time.

### 2.3 Existing Approaches

| Protocol | Discovery | Trust Bootstrap | Limitation |
|----------|-----------|-----------------|------------|
| SSH | Manual + TOFU | Accept on first use | No auto-discovery |
| WireGuard | Manual config | Pre-shared keys | No discovery at all |
| mDNS/DNS-SD | Multicast | None (LAN isolation) | No authentication |
| Signal | Contact list | TOFU + safety numbers | Requires phone number |
| PGP Web of Trust | Key signing | Transitive trust | Low adoption |

HERMES combines the best elements: SSH's TOFU for simplicity, Signal's
safety numbers for verification, and the hub as a lightweight introduction
service.

## 3. Terminology

- **TOFU**: Trust On First Use. Accept a peer's key the first time it is
  seen, store it, and alert on any subsequent change.
- **Safety Number**: A short fingerprint derived from both peers' public keys,
  used for out-of-band verification (per ARC-8446 §7).
- **Hub Presence**: A message of type `presence` broadcast by the hub when
  a clan connects or disconnects (format: `"<clan_id>: online|offline"`).
- **Auto-Peer**: The process of automatically registering a previously unknown
  clan as a peer based on hub presence or direct message receipt.

## 4. Level 0: TOFU via Hub Presence

### 4.1 Trigger Conditions

The Agent Node (ARC-4601) MUST attempt auto-peer registration when it
observes any of the following in `hub-inbox.jsonl`:

1. A `presence` message from the hub indicating a clan came online:
   ```json
   {"from": "HUB", "msg": "jei: online", "type": "presence"}
   ```

2. A direct message from an unknown clan:
   ```json
   {"from": "jei", "msg": "...", "type": "event", "dst": "momoshod"}
   ```

### 4.2 Registration Procedure

When a trigger condition is met for clan_id `P`:

1. **Skip if known**: If `P` already exists in `config.peers`, do nothing.
2. **Skip if self**: If `P` equals the local `config.clan_id`, do nothing.
3. **Key lookup**: Read `hub-peers.json` for `P`'s `sign_pub` field.
4. **Store key**: If found, write to `.keys/peers/<P>.pub`.
5. **Register peer**: Append to `config.peers` with:
   - `status: "active"` if key was found (hub-vouched)
   - `status: "pending_ack"` if no key available
   - `added: <today's date>`
6. **Persist**: Save config atomically via `save_config()`.
7. **Log**: Record the auto-peer event for auditability.

### 4.3 Configuration

Auto-peer discovery is controlled by the `auto_peer_enabled` field in the
Agent Node configuration (default: `true`).

```toml
[agent_node]
auto_peer_enabled = true   # false to disable TOFU
```

### 4.4 Security Considerations

**Trust anchor**: The hub's `hub-peers.json` serves as the trust anchor for
L0. A compromised hub could introduce malicious peers. This is acceptable
for the sovereign deployment model where the operator controls the hub.

**Key pinning**: Once a peer's key is stored in `.keys/peers/<P>.pub`, the
protocol SHOULD alert the operator if a different key is observed for the
same clan_id (key change detection, analogous to SSH's `WARNING: REMOTE
HOST IDENTIFICATION HAS CHANGED`).

**Mitigation**: Operators who require higher assurance SHOULD use L1 (Safety
Numbers) to verify peer fingerprints out-of-band.

## 5. Level 1: Safety Numbers

### 5.1 Fingerprint Generation

A safety number for a peer relationship is computed as:

```
safety_number = SHA256(sort(clan_id_A || pub_key_A, clan_id_B || pub_key_B))[:16]
```

The result is displayed as 4 groups of 4 hex characters:
```
b111:c2ec:7dfb:2a56
```

### 5.2 Verification Flow

1. Operator A runs `amaru peer verify <clan_id_B>`.
2. The CLI displays the safety number.
3. Operator A communicates the number to Operator B via a trusted channel
   (in person, phone call, encrypted chat).
4. Operator B runs `amaru peer verify <clan_id_A>` and confirms the match.
5. On confirmation, the peer status is upgraded from `active` to `verified`.

### 5.3 Status

Safety numbers are already computable via `ClanKeyPair.fingerprint()` in
ARC-8446. The `amaru peer verify` command is PLANNED.

## 6. Level 2: Federation Gossip (PLANNED)

### 6.1 Concept

The hub maintains a federation log (`federation-peers.json`) that records
all peer introductions with timestamps. This log is append-only and can be
audited by any clan to detect:

- Silent key replacements (a clan's key changed without announcement)
- Split-view attacks (hub showing different keys to different clans)

### 6.2 Design Reference

Inspired by Certificate Transparency (RFC 6962): the federation log serves
as a Merkle-tree-backed public record of peer relationships. Any clan can
request a consistency proof to verify the log has not been tampered with.

## 7. Level 3: LAN Discovery (PLANNED)

### 7.1 Concept

Agents on the same local network can discover each other via mDNS (RFC 6762)
service advertisements:

```
_hermes._tcp.local.  IN SRV 0 0 8443 momoshod.local.
_hermes._tcp.local.  IN TXT "clan_id=momoshod" "version=0.4.2" "fp=b111:c2ec"
```

### 7.2 Design Reference

Modeled after Apple Bonjour / Avahi. Zero configuration required. Trust
bootstraps via L0 (TOFU) once the LAN peer is discovered.

## 8. Implementation Notes

### 8.1 Reference Implementation

L0 is implemented in `hermes/agent.py`:
- `AgentNode._auto_peer_from_presence()` — core registration logic
- Integrated into `AgentNode._hub_inbox_loop()` — runs on every inbox poll
- 7 test cases in `tests/test_agent.py::TestAutoPeerDiscovery`

### 8.2 Backward Compatibility

Auto-peer discovery is purely additive:
- Clans without a hub continue to use manual `amaru peer add`.
- Clans with `auto_peer_enabled=false` are unaffected.
- Existing peers registered manually are never modified by auto-peer.
