## 1. Abstract

This specification defines the interpretation of requirement-level keywords used
across all HERMES standards. It adapts the semantics of RFC 2119 to the context
of file-based, inter-agent communication protocols, and provides guidance for
spec authors on choosing the appropriate keyword.

## 2. Motivation

HERMES standards use the keywords MUST, SHOULD, MAY (and their negatives) to
express requirement levels. Across the 10 existing specs, these keywords appear
in over 280 normative statements.

Each spec currently includes a boilerplate reference to RFC 2119 in its
Terminology section. While RFC 2119 provides the foundational semantics, it was
written for network protocol RFCs — not for agent communication protocols where:

- Implementations are file-based, not socket-based.
- Agents are stateless and ephemeral (session-scoped).
- A human operator holds final authority over all actions.
- Trust boundaries are enforced by namespace isolation, not TLS.

ARC-2119 serves as the HERMES-native canonical reference for requirement
keywords, supplementing (not replacing) each spec's RFC 2119 boilerplate.

## 3. Keyword Definitions

The following keywords, when they appear in UPPERCASE in a HERMES specification,
carry the normative meaning defined below.

### 3.1 MUST / REQUIRED / SHALL

The specification is an absolute requirement. An implementation that omits a MUST
requirement is non-conformant.

*Example: A message MUST contain a `ts`, `src`, `dst`, `type`, and `msg` field
(ARC-5322 Section 3).*

### 3.2 MUST NOT / SHALL NOT

The specification is an absolute prohibition. An implementation that performs a
MUST NOT action is non-conformant.

*Example: An agent MUST NOT access resources outside its own namespace unless
explicitly permitted by the routing table (ARC-0001 Section 2.1).*

### 3.3 SHOULD / RECOMMENDED

The specification describes a best practice. There may exist valid reasons in
particular circumstances to ignore it, but the full implications must be
understood before deviating.

*Example: A gateway SHOULD include a correlation ID (CID) in REL messages to
enable end-to-end tracking (ARC-3022 Section 5).*

### 3.4 SHOULD NOT / NOT RECOMMENDED

The specification describes a practice that is discouraged. There may exist valid
reasons in particular circumstances to adopt it, but the full implications must
be understood before doing so.

*Example: An agent SHOULD NOT broadcast messages with TTL greater than 7 unless
targeting a specific recipient (ARC-0793 Section 4).*

### 3.5 MAY / OPTIONAL

The specification describes a truly optional capability. An implementation may
include it for additional functionality or omit it without affecting
interoperability.

*Example: A gateway MAY suppress escalation for DGM messages that expire
unacknowledged (ARC-0768 Section 5).*

## 4. Usage Guidelines for Spec Authors

When writing HERMES specifications, choose requirement keywords based on the
following criteria:

| Keyword | Use when... | Typical domain |
|---------|-------------|----------------|
| **MUST** | Protocol breaks or security is violated without it | Message format fields, firewall rules, SYN/FIN procedures |
| **SHOULD** | Best practice with understood trade-offs for deviation | CID tracking, SYNC HEADER updates, TTL conventions |
| **MAY** | Truly optional; omission does not affect interoperability | Escalation suppression, additional message types, UI hints |

**Principles:**

1. **Prefer SHOULD over MUST** when the protocol remains functional without the
   requirement. Over-constraining implementations raises the barrier to adoption
   without proportional benefit.

2. **Reserve MUST for invariants** — conditions whose violation would cause
   message loss, security breach, or protocol incompatibility.

3. **Use MAY sparingly.** If a feature is described but marked MAY, ensure the
   spec defines behavior for implementations that omit it.

4. **MUST NOT and SHOULD NOT are not interchangeable.** MUST NOT means the action
   is a protocol violation. SHOULD NOT means the action is ill-advised but not
   a violation.

## 5. Conformance Implications

Keywords map directly to the conformance levels defined in ARC-0001 Section 6:

| Conformance Level | Keyword Coverage |
|-------------------|-----------------|
| **Level 1 — Minimal** | All MUST and MUST NOT requirements |
| **Level 2 — Standard** | All MUST, MUST NOT, SHOULD, and SHOULD NOT requirements |
| **Level 3 — Full** | All requirements including MAY and OPTIONAL features |

An implementation that violates a MUST or MUST NOT requirement cannot claim any
conformance level. An implementation that satisfies all MUST requirements but
deviates from a SHOULD requirement can claim Level 1 (Minimal) conformance if
the deviation is documented.

## 6. Formatting Conventions

In all HERMES specifications:

1. Keywords carry normative meaning ONLY when written in UPPERCASE
   (e.g., "agents MUST execute SYN").

2. The same words in lowercase carry no normative weight
   (e.g., "agents must have a purpose" is descriptive, not a requirement).

3. No additional typographic treatment (bold, italic) is required — UPPERCASE
   provides sufficient visual distinction.

4. Each spec SHOULD include the following boilerplate in its Terminology section:

   > The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   > "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this
   > document are to be interpreted as described in RFC 2119 and ARC-2119.
