> *"In SS7, every signaling message carries a sequence number. Not for ordering —
> the network handles that. For integrity: so the receiver can detect gaps, replays,
> and failures. The bus is HERMES's signaling link. It deserves the same discipline."*

## 1. Abstract

This specification defines the **Bus Integrity Protocol** for HERMES deployments.
It addresses six related concerns that emerge when the bus transitions from a
single-writer development tool to a multi-agent production system:

1. **Message Sequencing** (F1) — monotonic sequence numbers per source namespace
2. **Write Ownership** (F2) — authorization of namespace writers
3. **Multi-Version Concurrency Control** (F3) — causal ordering across namespaces
4. **Conflict Log** (F4) — detection and recording of integrity violations
5. **Recovery** (F5) — replay and merge after conflicts
6. **Garbage Collection** (F6) — archival and compaction

F1-F6 are all implemented.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHOULD", "SHOULD NOT",
"RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as
described in ARC-2119.

- **seq**: A monotonically increasing integer assigned to each message written
  by a particular source namespace. Starts at 1.
- **Write vector**: A dictionary `{src: seq}` capturing the causal state of
  the bus at the time of a write (F3, PLANNED).
- **Ownership claim**: A binding between a namespace and its authorized writer.
- **Anomaly**: A detected integrity violation (gap, duplicate, ownership breach).
- **Conflict**: Two or more messages with incompatible write vectors (F3, PLANNED).

## 3. Motivation

The Amaru bus (`bus.jsonl`) is an append-only JSONL file. In Phase 0, a single
session writes to the bus. As the Agent Node daemon (ARC-4601) and Agent Service
Platform (ARC-0369) introduce autonomous agents, multiple writers may coexist:

- The daemon writes on behalf of its namespace
- ASP agents write dispatch results and acknowledgments
- Relay delivers inbound messages from peer clans

Without sequencing, the reader cannot detect:
- **Gaps**: a message was lost or never written
- **Duplicates**: a message was written twice (replay)
- **Ordering violations**: messages arrived out of causal order

Without ownership, the reader cannot detect:
- **Source spoofing**: an agent writes as a namespace it doesn't own
- **Unauthorized writes**: an external relay injects messages into local namespaces

## 5. F1: Message Sequencing

### 5.1 Sequence Field

Each bus message MAY carry an optional `seq` field:

```json
{
  "ts": "2026-03-20",
  "src": "engineering",
  "dst": "*",
  "type": "state",
  "msg": "API deployed",
  "ttl": 7,
  "ack": [],
  "seq": 42
}
```

- `seq` MUST be a positive integer (>= 1).
- `seq` is monotonically increasing per `src` namespace.
- Messages without `seq` (legacy) MUST be accepted by all readers.
- The compact wire format (ARC-5322 §14) does NOT include `seq`. Sequence
  numbers are verbose-only for this version.

### 5.2 Sequence Tracking

A **SequenceTracker** maintains per-source state:

```
For each src:
  last_seq: int    — highest observed seq
  gap_count: int   — cumulative gaps detected
```

On write:
1. `next_seq(src)` returns `last_seq + 1` (or 1 if unknown)
2. Writer creates message with `seq = next_seq(src)`
3. `record(src, seq)` updates `last_seq`

On read:
1. For each message with `seq`: validate against expected `last_seq + 1`
2. If `seq > expected`: gap detected (missing messages)
3. If `seq <= last_seq`: duplicate detected (replay)
4. Messages without `seq`: skip validation (backward compat)

### 5.3 Gap Detection

A gap occurs when `seq > last_seq + 1` for a source. This indicates:
- Message loss (writer crashed between write and flush)
- Relay failure (inbound message not delivered)
- Bus file corruption (line truncated)

Implementations SHOULD log gaps as anomalies. Implementations MUST NOT
reject messages with gaps — the gap may be filled later (out-of-order delivery).

### 5.4 Duplicate Detection

A duplicate occurs when `seq <= last_seq` for a source. This indicates:
- Replay attack (re-appending old messages)
- Writer restart without state recovery
- Bus file concatenation error

Implementations SHOULD log duplicates. Implementations MAY reject duplicates
at the application layer (see F4: Conflict Log).

### 5.5 Persistence

The SequenceTracker state MUST be persisted in the Agent Node state file
(`agent-node.state.json`) as `seq_state: {src: last_seq}`. On restart, the
tracker restores from saved state. If no saved state exists, the tracker
bootstraps by scanning the bus.

## 6. F2: Write Ownership

### 6.1 Ownership Claims

An **OwnershipClaim** binds a namespace to its authorized writer:

```json
{
  "namespace": "engineering",
  "owner_id": "daemon",
  "granted_at": "2026-03-20"
}
```

### 6.2 Authorization Rules

When verifying whether a writer may write `src=namespace`:

1. **Explicit claim exists**: writer_id MUST match the claim's owner_id.
2. **No claim + daemon**: authorized (implicit daemon ownership).
3. **No claim + non-daemon**: denied.

### 6.3 Default Ownership

On daemon startup:
1. The daemon claims all local namespaces (from gateway.json).
2. For each enabled ASP agent, the daemon transfers ownership of the
   agent's namespace to the agent (revokes daemon claim, grants agent claim).

### 6.4 Ownership Transfer

Transfer is a two-step operation:
1. Revoke existing claim (daemon or previous owner)
2. Create new claim for the target writer

An OwnershipViolation MUST be raised if a writer attempts to claim a
namespace already owned by a different writer without first revoking.

### 6.5 Integration with MessageClassifier

The `MessageClassifier.verify_source()` method (ARC-0369 §6.3) is extended
with optional ownership-based authorization:

- When `ownership` and `writer_id` are provided: stricter ownership check
- When omitted: existing namespace membership check (backward compatible)

### 6.6 Persistence

The OwnershipRegistry MUST be persisted in `agent-node.state.json` as
`ownership_claims: {namespace: {owner_id, granted_at}}`. On restart, claims
are restored. ASP agent grants are re-applied (idempotent).

## 7. F3: Multi-Version Concurrency Control (PLANNED)

### 7.1 Write Vector

Each message carries a write vector `w: {src: seq}` representing the
writer's view of the bus at write time:

```json
{
  "ts": "2026-03-20",
  "src": "engineering",
  "dst": "ops",
  "type": "dispatch",
  "msg": "Deploy v2.1",
  "ttl": 3,
  "ack": [],
  "seq": 42,
  "w": {"engineering": 41, "ops": 15}
}
```

The write vector captures **causality**: message 42 from engineering was
written having seen engineering:41 and ops:15.

### 7.2 Conflict Detection

Two messages conflict when their write vectors are **concurrent** (neither
dominates the other). In a single-writer scenario, this cannot happen. In
multi-writer scenarios (relay, multi-daemon), conflicts indicate:

- Concurrent writes without coordination
- Relay delivering before local write is visible

### 7.3 Conflict Resolution

Resolution strategies (configured per clan):
- **Last-writer-wins** (default): timestamp breaks ties
- **Manual**: conflict logged, operator resolves
- **Merge**: domain-specific merge function (future)

*Implementation: IMPLEMENTED in integrity.py (WriteVector, WriteVectorTracker).*

## 8. F4: Conflict Log

### 8.1 Log Format

Conflicts are recorded in `bus-conflicts.jsonl`:

```json
{
  "detected_at": "2026-03-20T14:30:00",
  "type": "gap|duplicate|ownership|concurrent",
  "src": "engineering",
  "seq": 42,
  "expected": 43,
  "resolution": "logged|rejected|merged",
  "messages": ["<msg_hash_1>", "<msg_hash_2>"]
}
```

### 8.2 Retention

The conflict log SHOULD be retained independently of bus archival. Conflicts
provide forensic data for security audits and debugging.

*Implementation: IMPLEMENTED in integrity.py (ConflictLog, ConflictRecord).*

## 9. F5: Recovery (PLANNED)

### 9.1 Replay Protocol

When a gap is detected, the daemon MAY request replay from the source:
1. Send `REPLAY_REQUEST` dispatch with `{src, from_seq, to_seq}`
2. Source replays missing messages
3. Receiver validates and inserts in order

### 9.2 Snapshot

Periodic state snapshots (`bus-snapshot.json`) enable fast recovery:
- Contains: `{seq_state, ownership_claims, bus_hash, message_count}`
- Created: every N messages or on clean shutdown
- Used: bootstrap SequenceTracker without full bus scan

### 9.3 Merge Strategy

When two bus segments diverge (e.g., relay and local):
- **Last-writer-wins**: sort by timestamp, keep latest per src+seq
- **Manual**: present both to operator for resolution

*Implementation: IMPLEMENTED in integrity.py (SnapshotManager, BusSnapshot, ReplayRequest).*

## 10. F6: Garbage Collection

### 10.1 Archive Threshold

Messages below a sequence threshold MAY be archived:
1. Determine minimum `seq` still referenced by any pending dispatch or approval
2. Archive all messages with `seq < threshold` per source
3. Update `bus-archive.jsonl`

### 10.2 Compaction

After archival, the bus file MAY be compacted:
1. Read remaining messages
2. Write to temporary file
3. Atomic rename

### 10.3 Conflict Log Independence

The conflict log (`bus-conflicts.jsonl`) MUST NOT be affected by bus
garbage collection. Conflicts provide long-term forensic data.

*Implementation: IMPLEMENTED in integrity.py (BusGC). Atomic compaction via temp file + os.replace().*

## 11. Security Considerations

### 11.1 Sequence Manipulation

An attacker who can append to the bus file can forge sequence numbers.
Mitigation: sealed envelopes (ARC-8446) sign the message including `seq`,
preventing post-write tampering.

### 11.2 Ownership Spoofing

The OwnershipRegistry is enforced at the daemon level. An attacker with
file-system write access can bypass it by writing directly to `bus.jsonl`.
Mitigation: sealed envelopes verify sender identity via Ed25519 signatures.

### 11.3 Replay Protection

The NonceTracker (ARC-8446 §9.5) provides nonce-based replay protection
for encrypted messages. The SequenceTracker provides seq-based duplicate
detection for cleartext messages. Together, they cover both modes.

### 11.4 Trust Model

- **Single-clan**: daemon is trusted. SequenceTracker validates its own writes.
- **Multi-clan (relay)**: inbound messages carry sender signatures. The
  receiver validates signature + seq before accepting.
- **Multi-daemon (future)**: requires F3 MVCC for conflict-free concurrent writes.

## 12. Wire Format

### 12.1 Verbose JSON

The `seq` field is added as an optional field to the verbose JSON format:

```json
{"ts":"2026-03-20","src":"eng","dst":"*","type":"state","msg":"test","ttl":7,"ack":[],"seq":1}
```

Omitted when `seq` is null (backward compatible).

### 12.2 Compact Format

The compact array format (ARC-5322 §14) is NOT modified. Sequence numbers
are verbose-only. Rationale: the compact format optimizes for throughput
(872K msg/sec); adding a field changes the positional encoding and breaks
existing implementations.

A future amendment to ARC-5322 MAY add `seq` at position 8 or 9 of the
compact array, with auto-detection by array length.

## 13. Migration

### 13.1 Backward Compatibility

- Old writers produce messages without `seq`. These are valid.
- Old readers ignore unknown fields (including `seq`). No breakage.
- The `_parse_bus_message_permissive()` function in agent.py already
  tolerates `seq` and `w` fields.

### 13.2 Forward Compatibility

- New writers SHOULD include `seq` when a SequenceTracker is available.
- New readers SHOULD validate `seq` when present but MUST NOT reject
  messages without it.

### 13.3 Phase-In

| Phase | Writer | Reader | SequenceTracker |
|-------|--------|--------|-----------------|
| 0 (current) | No seq | Ignores | Not instantiated |
| 1 (ARC-9001 F1-F2) | Adds seq when tracker available | Validates when present | Bootstraps from bus |
| 2 (ARC-9001 F3-F6) | Adds seq + w | Validates causality, logs conflicts, recovery, GC | Full integrity stack |
