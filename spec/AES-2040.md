# AES-2040: Agent Visualization Standard

| Field       | Value                                                     |
|-------------|-----------------------------------------------------------|
| **Number**  | AES-2040                                                  |
| **Title**   | Agent Visualization Standard                              |
| **Lineage** | Original (no direct IEEE lineage)                         |
| **Status**  | DRAFT                                                     |
| **Authors** | HERMES Contributors                                       |
| **Created** | 2026-03-14                                                |
| **Requires** | ARC-0001 (Architecture), ARC-4601 (Agent Node), ARC-5322 (Message Format) |

---

> *"No tool exists today for tracing agent-to-agent protocol flows interactively.
> The networking world has Wireshark, tcpdump, and Packet Tracer. The agent world
> has nothing. AES-2040 fills that gap — a Protocol Explorer that makes HERMES
> as inspectable as TCP/IP."*

## 1. Abstract

This specification defines the **Agent Visualization Standard** for HERMES. It
establishes a 5-layer visualization stack for protocol documentation, and
specifies the **Protocol Explorer** — an interactive, browser-based tool for
tracing agent-to-agent communication flows in real time.

The Protocol Explorer connects to a HERMES Agent Node (ARC-4601) via SSE or
reads `bus.jsonl` directly, providing 6 visualization modes: Message Flow,
Session Timeline, Cross-Clan Path, Crypto Envelope, Dispatch Tree, and Bus
Health.

Research across the agent protocol landscape (A2A, MCP, NLIP, ANP) confirmed
that no interactive tracing tool exists for agent-to-agent protocols. Tools
like Wireshark, Cisco Packet Tracer, and GNS3 serve TCP/IP. AES-2040 brings
the same capability to agent communication.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
"SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be
interpreted as described in ARC-2119.

| Term | Definition |
|------|-----------|
| **Visualization Layer** | A tool or format in the 5-layer stack, each serving a distinct purpose and audience |
| **Protocol Explorer** | An interactive browser-based SPA for tracing HERMES protocol flows in real time |
| **Diagram Taxonomy** | The naming and classification system for visual documentation (SEQ, UC, ARCH, STATE) |
| **Connected Mode** | Protocol Explorer receives events via SSE from an Agent Node |
| **Local Mode** | Protocol Explorer reads `bus.jsonl` directly from the local filesystem |

## 3. Visualization Layer Model

HERMES uses a layered approach to visual documentation. Each layer serves a
different audience and rendering context. Higher layers provide richer
interactivity but require more tooling.

### 3.1 Layer 1: Inline ASCII Art

**Purpose**: Small, inline examples within spec text.

- **Audience**: Spec readers, terminal users, code reviewers
- **Rendering**: Universal — any text editor, terminal, or browser
- **Version control**: Perfect (plain text, git-diffable)
- **When to use**: Field layouts, small state machines, quick architecture sketches
- **Identity**: ASCII art is part of HERMES identity and MUST be preserved in specs
- **Limitations**: Hard to maintain at scale, poor aesthetics for complex flows

### 3.2 Layer 2: Mermaid Diagrams

**Purpose**: Structured protocol flow diagrams rendered natively by GitHub.

- **Audience**: Contributors, builders, students browsing the repo on GitHub
- **Rendering**: GitHub-native (zero setup), embeddable via `mermaid.js` npm package
- **Version control**: Excellent (text-based Markdown)
- **Diagram types**: `sequenceDiagram`, `flowchart`, `stateDiagram-v2`
- **When to use**: Sequence diagrams (message-by-message), use case flows, architecture views
- **Limitations**: Limited layout control, no animation, no interactivity
- **Location**: `docs/diagrams/` directory with naming convention `seq-NNNN-*.md`, `uc-NN-*.md`, `arch-*.md`

### 3.3 Layer 3: D2 Animated Diagrams

**Purpose**: Animated protocol flow demonstrations for presentations and desktop apps.

- **Audience**: Conference attendees, potential adopters, desktop app users
- **Rendering**: D2 CLI generates SVG; embeddable in any browser or Electron/Tauri app
- **Version control**: Good (text-based `.d2` source files)
- **Unique capabilities**: Animation (protocol flows), sketch mode (hand-drawn aesthetic), modular imports
- **When to use**: Live demos, presentations, desktop app protocol views
- **Limitations**: No native GitHub rendering (requires SVG export)
- **Location**: `docs/diagrams/d2/` directory

### 3.4 Layer 4: Excalidraw Community Templates

**Purpose**: Informal, collaborative architecture sketches for the community.

- **Audience**: Contributors, community members, brainstorming sessions
- **Rendering**: excalidraw.com (browser) or embedded React component
- **Version control**: Good (`.excalidraw` files are JSON)
- **When to use**: Architecture brainstorming, contribution-friendly visual guides
- **Limitations**: Not structured (freeform canvas), no protocol-specific semantics
- **Location**: `docs/diagrams/excalidraw/` directory

### 3.5 Layer 5: Protocol Explorer (Interactive)

**Purpose**: Real-time, interactive visualization of HERMES protocol flows.

- **Audience**: Operators, builders debugging deployments, students learning the protocol
- **Rendering**: Browser-based SPA, zero server required (static files)
- **Version control**: Source code in `explorer/` directory
- **When to use**: Live bus inspection, protocol learning, deployment debugging
- **Capabilities**: See Section 4 (full specification)

### 3.6 Layer Selection Guide

| Scenario | Recommended Layer |
|----------|------------------|
| Spec author documenting a field layout | L1 (ASCII) |
| Contributor reading a protocol flow on GitHub | L2 (Mermaid) |
| Presenter showing a protocol demo to investors | L3 (D2 animated) |
| Community member sketching a new topology | L4 (Excalidraw) |
| Operator debugging why a message isn't arriving | L5 (Protocol Explorer) |
| Student learning how SYN/FIN works | L2 (Mermaid) or L5 (Protocol Explorer) |

## 4. Protocol Explorer Specification

### 4.1 Functional Requirements

The Protocol Explorer MUST provide:

1. **Real-time bus visualization** — messages appear as they are written to `bus.jsonl`
2. **Message inspection** — click any message to see all 7 fields, encoding, and metadata
3. **Namespace filtering** — show/hide messages by source or destination namespace
4. **Type filtering** — show/hide messages by type (state, alert, event, etc.)
5. **Timeline view** — messages plotted on a time axis with TTL countdown
6. **Session boundaries** — SYN/FIN phases visually demarcated
7. **Zero-install mode** — works as a static SPA served from local files or GitHub Pages

The Protocol Explorer SHOULD provide:

1. **Connected mode** — SSE connection to Agent Node for real-time updates
2. **Cross-clan path tracing** — visualize message hops through Gateway NAT
3. **Crypto envelope inspection** — step-by-step seal/open visualization
4. **Dispatch tree** — decision tree showing quest → skill match → execution
5. **Bus health dashboard** — TTL countdown bars, throughput, ACK rates
6. **Dark mode** — default appearance

### 4.2 Data Sources

| Source | Access Method | Data Provided |
|--------|--------------|---------------|
| `bus.jsonl` | File API (local mode) | All bus messages |
| Agent Node SSE | `GET /events` (connected mode) | Real-time new messages |
| Agent Node state | `GET /state` | Node status, dispatch slots, heartbeat |
| SYNC HEADERs | Parsed from namespace configs | Session boundaries, pending counts |
| Gateway logs | Future integration | Cross-clan message paths |

### 4.3 Visualization Modes

#### 4.3.1 Message Flow

The primary view. Messages flow between namespace swimlanes, color-coded by type.

- **X-axis**: Namespace swimlanes (one column per namespace)
- **Y-axis**: Time (newest at bottom)
- **Arrows**: Directed from `src` to `dst`, colored by `type`
- **Broadcast**: `dst="*"` messages fan out to all namespace columns
- **ACK indicators**: Checkmark overlay when namespace appears in `ack[]`
- **Stale indicators**: Warning icon for messages unACKed > 3 days

**Color scheme** (RECOMMENDED):

| Type | Color | Rationale |
|------|-------|-----------|
| `state` | Blue | Calm, informational |
| `alert` | Red | Urgency |
| `event` | Green | Something happened |
| `request` | Orange | Needs attention |
| `data_cross` | Purple | Cross-boundary data |
| `dispatch` | Yellow | Action assignment |
| `dojo_event` | Teal | System evolution |

#### 4.3.2 Session Timeline

A Gantt-like view showing session boundaries over time.

- **Rows**: One per namespace
- **Bars**: SYN → ACTIVE → FIN phases, colored by state
- **Gaps**: Idle periods between sessions
- **Annotations**: SYNC HEADER version numbers at FIN boundaries
- **Stale detection**: Namespaces with `last_sync` > 7 days highlighted

#### 4.3.3 Cross-Clan Path

Visualizes how a message travels through the Gateway NAT boundary.

- **Nodes**: Source namespace → Gateway (NAT) → Agora → Remote Gateway → Destination
- **Annotations at each hop**: Identity translation, filter decisions, validation results
- **Latency**: Time between hops (if available from gateway logs)
- **Error paths**: Blocked by firewall, rate limited, validation failed

#### 4.3.4 Crypto Envelope

Step-by-step visualization of ARC-8446 seal and open procedures.

- **Seal flow**: Plaintext → DH derivation → HKDF → Nonce → AAD → AES-GCM encrypt → Ed25519 sign → Envelope
- **Open flow**: Envelope → Ed25519 verify → DH derivation → HKDF → AAD reconstruct → AES-GCM decrypt → Plaintext
- **Side-by-side**: Plaintext vs ciphertext comparison
- **ECDHE variant**: Ephemeral key generation and zeroization highlighted
- **Clickable steps**: Each cryptographic operation expandable with input/output details

#### 4.3.5 Dispatch Tree

Decision tree showing how the Dojo evaluates and dispatches quests.

- **Root**: Incoming dispatch or request message
- **Branches**: Capability match → XP threshold → Availability → Load check
- **Leaf nodes**: Selected skill with assignment details
- **Metrics**: Match score, skill XP, concurrent dispatch slots

#### 4.3.6 Bus Health

Dashboard view of bus operational metrics.

- **TTL countdown bars**: Each active message with remaining days visualized
- **Throughput histogram**: Messages per day, by type
- **ACK completion rate**: Percentage of messages fully ACKed before expiration
- **Queue depth**: Number of unACKed messages trending over time (Little's Law: L = λW)
- **Stability indicator**: λ < μ (green) vs λ ≥ μ (red) per ARC-0793 Section 11.3

### 4.4 Technology Stack

| Component | Recommended Technology | Rationale |
|-----------|----------------------|-----------|
| **Framework** | Svelte | Lighter than React, excellent for real-time data viz |
| **Visualization** | D3.js | Industry standard for custom data visualization |
| **Standard diagrams** | Mermaid.js | For rendering Mermaid within the explorer |
| **Build** | Vite | Fast builds, zero-config for Svelte |
| **Deployment** | Static SPA | Zero server required — aligns with HERMES philosophy |
| **Hosting** | GitHub Pages / local file:// | Accessible to anyone, no infrastructure needed |

**Alternative**: For maximum simplicity, the initial prototype MAY use plain
HTML + D3.js with no build step. This enables opening `index.html` directly
from the filesystem.

### 4.5 Integration with Agent Node (ARC-4601)

The Agent Node (ARC-4601) already runs as a persistent daemon with bus observation
and gateway link capabilities. The Protocol Explorer integrates via a lightweight
HTTP server embedded in the Agent Node.

#### 4.5.1 Explorer Server Component

The Agent Node MAY expose an ExplorerServer on a configurable port (default: 8472):

```
Agent Node daemon
  |
  |-- BusObserver (watches bus.jsonl)
  |-- GatewayLink (SSE to/from gateway)
  |-- Dispatcher (sub-agent processes)
  |-- ExplorerServer (port 8472)
       |
       |-- GET /              Serve Protocol Explorer SPA
       |-- GET /bus.jsonl     Serve raw bus file
       |-- GET /events        SSE stream of new bus messages
       |-- GET /state         Current node state (JSON)
       |-- GET /health        Health check
```

#### 4.5.2 SSE Event Format

The `/events` endpoint streams new bus messages as SSE events:

```
event: message
data: {"ts":"2026-03-14","src":"engineering","dst":"ops","type":"state","msg":"Deploy complete.","ttl":7,"ack":[]}

event: heartbeat
data: {"ts":"2026-03-14T17:30:00","uptime":3600,"bus_lines":42}
```

#### 4.5.3 Configuration

The ExplorerServer is configured in the `agent_node` section of `gateway.json`:

```json
{
  "agent_node": {
    "explorer": {
      "enabled": false,
      "port": 8472,
      "bind": "127.0.0.1"
    }
  }
}
```

- `enabled`: Default `false` — opt-in only
- `bind`: Default `127.0.0.1` — local access only for security
- The explorer MUST NOT be exposed to public networks without authentication

## 5. Diagram Taxonomy

All visual documentation in HERMES follows a consistent naming and classification system.

### 5.1 Sequence Diagrams (SEQ-*)

- **Format**: `seq-NNNN-description.md` where NNNN is the ARC/ATR/AES number
- **Purpose**: Message-by-message, hop-by-hop protocol flows
- **Mermaid type**: `sequenceDiagram`
- **Example**: `seq-5322-message-lifecycle.md`

### 5.2 Use Case Diagrams (UC-*)

- **Format**: `uc-NN-description.md`
- **Purpose**: Customer journeys showing end-to-end scenarios
- **Mermaid type**: `flowchart TD`
- **Example**: `uc-01-sovereign-clan-setup.md`

### 5.3 Architecture Diagrams (ARCH-*)

- **Format**: `arch-description.md`
- **Purpose**: System-level views of components and their relationships
- **Mermaid type**: `flowchart TB` or `flowchart LR`
- **Example**: `arch-5-layer-stack.md`

### 5.4 State Diagrams (STATE-*)

- **Format**: `state-NNNN-description.md`
- **Purpose**: State machines for protocol lifecycle
- **Mermaid type**: `stateDiagram-v2`
- **Note**: Currently embedded within SEQ diagrams; MAY be extracted to standalone files

## 6. Rendering Requirements

### 6.1 Mermaid Compatibility

All Mermaid diagrams MUST render correctly on:

1. GitHub.com (primary target — native rendering)
2. GitLab.com (supports Mermaid natively)
3. VS Code with Mermaid extension
4. `mermaid.live` online editor

Diagrams SHOULD use only `sequenceDiagram`, `flowchart`, and `stateDiagram-v2`
types, which have the most consistent cross-platform rendering.

### 6.2 D2 Compatibility

D2 diagrams MUST include pre-rendered SVG files alongside `.d2` source files.
This ensures readers without the D2 CLI can view the diagrams.

### 6.3 Protocol Explorer Compatibility

The Protocol Explorer MUST work in:

1. Modern browsers (Chrome, Firefox, Safari, Edge — latest 2 versions)
2. Electron/Tauri desktop apps (for the HERMES desktop agent)
3. Local `file://` protocol (for offline use)

## 7. Accessibility

### 7.1 Text Alternatives

All diagrams MUST include a prose description (step-by-step explanation) below
the visual diagram. This serves users who cannot view the diagram (screen
readers, text-only terminals, low-bandwidth connections).

### 7.2 Color Independence

The Protocol Explorer MUST NOT rely on color alone to convey information.
All color-coded elements (message types, states) MUST also include text labels
or shape differentiation.

### 7.3 Keyboard Navigation

The Protocol Explorer SHOULD support keyboard navigation for message
inspection and filtering.

## 8. Security Considerations

### 8.1 ExplorerServer Access

The ExplorerServer component MUST bind to `127.0.0.1` by default. Exposing it
to non-local interfaces requires explicit configuration and SHOULD include
authentication (e.g., bearer token).

### 8.2 Bus File Contents

The Protocol Explorer displays raw bus contents. Operators MUST ensure that
bus files do not contain sensitive data (per ARC-5322 Section 13.1: no
credentials in payloads).

### 8.3 No Write Access

The Protocol Explorer is a **read-only** tool. It MUST NOT modify `bus.jsonl`,
inject messages, or alter any protocol state.

## 9. Reference Implementation

The initial implementation will follow a phased approach:

1. **Phase C** (current): This specification document (DRAFT status)
2. **Phase D** (Jul-Aug 2026): D2 animated diagrams in `docs/diagrams/d2/`
3. **Phase E** (Sep-Oct 2026): Excalidraw community templates
4. **Protocol Explorer prototype**: Target Phase 4-5 (Sep-Dec 2026)

The Protocol Explorer source code will reside in `explorer/` at the
repository root.
