# ARC-8446: Encrypted Bus Protocol

| Field     | Value                                                        |
|-----------|--------------------------------------------------------------|
| Number    | ARC-8446                                                     |
| Title     | Encrypted Bus Protocol                                       |
| Lineage   | RFC 8446 (TLS 1.3) key schedule and verify-before-decrypt    |
| Status    | IMPLEMENTED                                                  |
| Author    | HERMES Contributors, Clan JEI (Security Review)              |
| Created   | 2026-03-07                                                   |

## 1. Abstract

ARC-8446 specifies the cryptographic protocol for authenticated and encrypted
communication between HERMES clans over the inter-clan relay. It defines key
generation, key exchange, message encryption, digital signatures, and the
verify-before-decrypt pattern that enables secure inter-agent messaging without
shared pre-existing secrets.

The protocol uses three algorithms as a unified stack:

- **Ed25519** for digital signatures (authenticity + integrity)
- **X25519** for Diffie-Hellman key agreement (shared secret derivation)
- **AES-256-GCM** for authenticated encryption (confidentiality)

This combination provides forward-compatible security while remaining
lightweight enough for file-based relay transport.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
"SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be
interpreted as described in [ARC-2119](ARC-2119.md).

### 2.1 Definitions

**Clan Key Pair** -- A pair of asymmetric key pairs owned by a clan: one
Ed25519 key pair for signing and one X25519 key pair for Diffie-Hellman key
agreement. Together they form the clan's cryptographic identity.

**Shared Secret** -- A 256-bit symmetric key derived from X25519 key agreement
between two clans. Used as the AES-256-GCM encryption key.

**Sealed Message** -- A message that has been encrypted with AES-256-GCM and
signed with Ed25519 over the ciphertext.

**Fingerprint** -- A human-readable representation of a public key, used for
in-person verification. Format: 8 groups of 4 hex characters separated by
colons (e.g., `a1b2:c3d4:e5f6:7890:1234:5678:9abc:def0`).

**Relay** -- A shared storage location (e.g., private Git repository) where
clans exchange sealed messages via per-clan outbox files.

**Associated Authenticated Data (AAD)** -- Additional data bound to the
ciphertext via AES-256-GCM that is authenticated but not encrypted. Prevents
metadata tampering without revealing sensitive content.

## 3. Cryptographic Stack

### 3.1 Algorithm Selection

| Layer           | Algorithm    | Purpose                          | Key Size |
|-----------------|-------------|----------------------------------|----------|
| Signature       | Ed25519     | Authenticity + integrity         | 256 bits |
| Key Agreement   | X25519      | Shared secret derivation         | 256 bits |
| Encryption      | AES-256-GCM | Authenticated encryption         | 256 bits |
| Key Derivation  | SHA-256     | Uniform key from DH raw output   | 256 bits |

### 3.2 Rationale

- **Ed25519 over ECDSA**: Deterministic signatures, faster verification, no
  nonce reuse vulnerability, constant-time implementation.
- **X25519 over ECDH-P256**: Simpler API, resistant to implementation errors,
  designed for key agreement specifically.
- **AES-256-GCM over AES-256-CBC**: Provides both confidentiality and integrity
  in a single operation (AEAD). No need for separate HMAC.
- **SHA-256 over HKDF**: Sufficient for single-purpose key derivation from a
  uniform random DH output. HKDF adds complexity without security benefit in
  this context since X25519 output is already uniform.

### 3.3 Why Not PSK (Pre-Shared Key)?

A symmetric-only approach (e.g., AES-256 with pre-shared keys) was considered
and rejected for the following reasons:

| Criterion           | PSK (AES-only)           | Ed25519 + X25519 + AES-256-GCM |
|---------------------|--------------------------|--------------------------------|
| Key distribution    | O(n²) — unique key/pair  | O(n) — one key pair per clan   |
| Key rotation        | Manual coordination      | New DH per session (possible)  |
| Non-repudiation     | No (shared key)          | Yes (unique signature)         |
| Implementation      | ~50 lines                | ~80 lines                      |
| Migration cost      | Requires rekey to scale  | None — scales natively         |

The marginal implementation cost (~30 lines) is far outweighed by the
architectural benefits of asymmetric cryptography.

## 4. Key Generation

### 4.1 Clan Key Pair

A conforming implementation MUST generate two independent key pairs:

1. **Signing key pair**: Ed25519 (32-byte private key, 32-byte public key)
2. **Key agreement key pair**: X25519 (32-byte private key, 32-byte public key)

The two key pairs MUST be generated independently. An implementation MUST NOT
derive one from the other.

### 4.2 Key Storage

Private keys MUST be stored locally with restricted file permissions
(e.g., `0600` on UNIX systems). Private keys MUST NOT be transmitted over any
channel.

Public keys SHOULD be stored in the relay's `keys/` directory as JSON files
with the following format:

```json
{
  "clan": "<clan_id>",
  "ed25519_pub": "<hex-encoded 32-byte Ed25519 public key>",
  "x25519_pub": "<hex-encoded 32-byte X25519 public key>",
  "fingerprint_sign": "<fingerprint of Ed25519 public key>",
  "fingerprint_dh": "<fingerprint of X25519 public key>",
  "generated": "<ISO 8601 timestamp>"
}
```

### 4.3 Fingerprint Computation

Each public key has its own fingerprint. The fingerprint MUST be computed as:

```
fingerprint = SHA-256(raw_public_key_bytes)
            → take first 16 bytes (128 bits)
            → encode as hex
            → format as 8 groups of 4 hex chars separated by ':'
```

Example: `2a37:fb25:a054:93ee:a608:41fc:4ad9:687e`

Fingerprints enable in-person verification of public key authenticity,
following the Trust On First Use (TOFU) model established by Signal's
safety numbers.

## 5. Key Agreement

### 5.1 Shared Secret Derivation

Two clans derive a shared secret using X25519 Diffie-Hellman:

```
raw_shared = X25519(my_dh_private, peer_dh_public)
shared_secret = SHA-256(raw_shared)
```

The raw DH output MUST be hashed with SHA-256 to produce a uniformly
distributed 256-bit key suitable for AES-256-GCM.

### 5.2 Symmetry Property

Both parties derive the same shared secret:

```
SHA-256(X25519(A_private, B_public)) == SHA-256(X25519(B_private, A_public))
```

This property is fundamental to Diffie-Hellman and MUST be verified in any
conformance test suite.

## 6. Message Sealing (Encrypt + Sign)

### 6.1 Sealing Procedure

To seal a message for transmission:

1. **Derive shared secret** per Section 5.1
2. **Generate nonce**: 12 random bytes (96-bit IV for AES-256-GCM)
3. **Construct AAD**: Build the Associated Authenticated Data per Section 6.1.1
4. **Encrypt**: `ciphertext = AES-256-GCM.encrypt(shared_secret, nonce, plaintext_utf8, aad=aad_bytes)`
5. **Sign**: `signature = Ed25519.sign(sender_sign_private, ciphertext_bytes)`
6. **Assemble** the sealed message envelope

#### 6.1.1 AAD Construction

The AAD binds envelope metadata to the ciphertext cryptographically, preventing
header manipulation attacks. The AAD MUST be constructed as a canonical JSON
object containing the envelope fields, encoded as UTF-8 bytes:

```
aad_bytes = canonical_json({"dst": dst, "src": src, "ts": ts, "type": type})
```

Canonical JSON is defined as: keys sorted alphabetically, no whitespace
(`separators=(',', ':')` in Python's `json.dumps`). This ensures both parties
produce identical byte sequences regardless of insertion order.

The AAD object MUST include the following fields from the ARC-5322 envelope:
- `src` — sender clan identifier
- `dst` — destination clan identifier
- `type` — message type
- `ts` — timestamp

The sealed message envelope SHOULD include the AAD as a hex-encoded `aad` field
for diagnostic purposes. The receiver MUST reconstruct the AAD from the outer
ARC-5322 envelope fields and provide it during decryption. If the AAD does not
match, AES-256-GCM decryption will fail with an authentication error, indicating
that the envelope metadata was tampered with.

For backward compatibility with pre-AAD messages: if the sealed envelope does
not contain an `aad` field, the receiver SHOULD decrypt without AAD (`aad=None`).
This allows graceful migration from unbound to bound messages.

### 6.2 Sealed Message Format

```json
{
  "ciphertext": "<hex-encoded ciphertext + GCM tag>",
  "nonce": "<hex-encoded 12-byte nonce>",
  "signature": "<hex-encoded 64-byte Ed25519 signature>",
  "sender_sign_pub": "<hex-encoded 32-byte Ed25519 public key>"
}
```

All binary values MUST be hex-encoded for JSON compatibility.

**Security note on `sender_sign_pub`**: The `sender_sign_pub` field is included
for message routing and debugging purposes only. Receivers MUST verify
signatures using the pre-established public key from `keys/<clan_id>.pub`, NOT
the `sender_sign_pub` value from the sealed envelope. Using the envelope's
`sender_sign_pub` for verification enables identity substitution attacks where
an attacker replaces both the signature and public key to impersonate a
legitimate sender.

### 6.3 Signature Scope

The signature MUST be computed over the **ciphertext bytes** (the raw bytes of
the encrypted output, not the hex string). This is the verify-before-decrypt
pattern from TLS 1.3:

```
signature = Ed25519.sign(private_key, bytes.fromhex(ciphertext_hex))
```

**Rationale**: Signing the ciphertext allows the recipient to verify message
authenticity *before* decryption. This prevents processing of unauthenticated
data and reduces the attack surface.

An implementation MUST NOT sign the plaintext, the nonce, or any combination
other than the ciphertext bytes alone.

## 7. Message Opening (Verify + Decrypt)

### 7.1 Opening Procedure

To open a sealed message:

1. **Verify signature**: `Ed25519.verify(peer_sign_public, ciphertext_bytes, signature)`
2. If verification fails → REJECT message, return `None`
3. **Derive shared secret** per Section 5.1
4. **Reconstruct AAD**: Build the AAD from the outer envelope fields per Section 6.1.1
5. **Decrypt**: `plaintext = AES-256-GCM.decrypt(shared_secret, nonce, ciphertext, aad=aad_bytes)`
6. **Decode**: UTF-8 decode the plaintext bytes

The `peer_sign_public` used in step 1 MUST be the key loaded from
`keys/<clan_id>.pub`, NOT the `sender_sign_pub` from the sealed envelope
(see Section 6.2 security note).

### 7.2 Verify-Before-Decrypt

An implementation MUST verify the Ed25519 signature before attempting
decryption. This ordering is REQUIRED because:

- It prevents wasting computational resources on unauthenticated data
- It provides defense-in-depth against oracle attacks
- It aligns with TLS 1.3 record processing (RFC 8446, Section 5.2)

If signature verification fails, the implementation MUST NOT attempt
decryption and SHOULD log the failure for auditing.

## 8. Relay Transport

### 8.1 Relay Structure

The relay is a shared storage location with the following structure:

```
relay/
├── keys/
│   ├── <clan_a>.pub       # Clan A public keys
│   └── <clan_b>.pub       # Clan B public keys
├── relay/
│   ├── <clan_a>_outbox.jsonl   # Messages FROM Clan A
│   └── <clan_b>_outbox.jsonl   # Messages FROM Clan B
└── relay/payloads/
    └── .gitkeep                # Large payloads (encoding: "ref")
```

### 8.2 Relay Message Format

Each line in an outbox file is a JSON object conforming to ARC-5322 with the
sealed message in the `msg` field:

```json
{
  "ts": "2026-03-07",
  "src": "<sender_clan>",
  "dst": "<receiver_clan>",
  "type": "hello_ack",
  "msg": "<sealed message JSON as string>",
  "ttl": 7,
  "ack": []
}
```

The `msg` field contains the sealed message envelope (Section 6.2) serialized
as a JSON string.

### 8.3 First Contact (Bootstrap)

The first inter-clan exchange follows this sequence:

1. Both clans generate key pairs (Section 4.1)
2. Both clans publish public keys to `keys/<clan_id>.pub`
3. Fingerprints are verified in person (Section 4.3)
4. Clan A seals a `hello` message and writes to `relay/<clan_a>_outbox.jsonl`
5. Clan B reads, verifies, decrypts → seals a `hello_ack` response
6. Clan A reads and verifies the `hello_ack`
7. Bidirectional encrypted channel is established

## 9. Security Considerations

### 9.1 Nonce Reuse

AES-256-GCM is catastrophically broken if a nonce is reused with the same key.
Implementations MUST generate nonces using a cryptographically secure random
number generator (`os.urandom` or equivalent). With 96-bit random nonces, the
collision probability remains negligible for up to 2^32 messages per key pair.

### 9.2 Private Key Protection

Private keys MUST be stored with filesystem-level access control. On UNIX
systems, private key files MUST have permissions `0600` (owner read/write
only). Private keys MUST NOT be committed to version control, logged, or
transmitted.

### 9.3 Trust Model

ARC-8446 uses Trust On First Use (TOFU) with in-person fingerprint
verification. This is suitable for the current small-clan model. Future
extensions (see Section 11) MAY introduce certificate authorities or web-of-
trust models for larger deployments.

### 9.4 Forward Secrecy

The current specification uses static DH keys, which means compromise of a
private key allows decryption of all past messages. A future revision SHOULD
introduce ephemeral DH keys (one per message or per session) to provide
forward secrecy, following the TLS 1.3 pattern.

### 9.5 Replay Protection

Implementations MUST track received nonces to detect and reject replayed
messages. The nonce-set MUST be maintained per sender clan, since nonces are
only required to be unique within a single sender's message stream.

The nonce-set SHOULD be bounded by the TTL window: nonces from messages whose
TTL has expired MAY be evicted from the set. This bounds memory usage while
maintaining replay protection for all messages that are still considered valid.

The nonce-set MUST be persisted across sessions (e.g., to a local file) to
prevent replay attacks that exploit application restarts. On startup, an
implementation MUST load the persisted nonce-set before processing any
incoming messages.

If nonce-set persistence is not feasible (e.g., embedded environments), the
implementation MUST document this limitation and SHOULD employ an alternative
mechanism such as monotonic timestamps validated against a local clock.

### 9.6 Key Revocation

A clan whose private key is compromised MUST initiate key revocation
immediately. The revocation procedure is as follows:

1. **Generate new key pair**: The compromised clan generates a new Ed25519 +
   X25519 key pair per Section 4.1.
2. **Sign a REVOKE message**: Using the OLD (compromised) signing key, the clan
   signs a `REVOKE` message containing the new public keys. Signing with the
   old key proves that the revocation originates from the legitimate key holder
   rather than an attacker.
3. **Publish to relay**: The `REVOKE` message is written to the clan's outbox
   on the relay.
4. **Notify peers out-of-band**: The clan MUST notify all peer clans through an
   out-of-band channel (e.g., in-person, secure messaging) that key revocation
   has occurred.

The `REVOKE` message format:

```json
{
  "ts": "<ISO 8601 timestamp>",
  "src": "<clan_id>",
  "dst": "*",
  "type": "revoke",
  "msg": "<sealed envelope containing new public keys JSON>",
  "ttl": 30,
  "ack": []
}
```

The sealed payload MUST contain the new public keys in the same format as
`keys/<clan_id>.pub` (Section 4.2).

Peer clans MUST stop accepting messages signed with the revoked key
immediately upon receiving and validating the `REVOKE` message. Peer clans
MUST NOT trust the new public keys from the `REVOKE` message alone -- re-keying
REQUIRES the same in-person fingerprint verification as first contact
(Section 8.3).

Revocation is effective immediately upon relay propagation. Messages signed
with the revoked key that arrive after the `REVOKE` message has been processed
MUST be rejected.

## 10. Reference Implementation

The reference implementation is in `reference/python/hermes/crypto.py`:

- `ClanKeyPair` — Key generation, storage, fingerprinting
- `derive_shared_secret()` — X25519 + SHA-256
- `encrypt_message(key, plaintext, aad=None)` / `decrypt_message(key, nonce, ciphertext, aad=None)` — AES-256-GCM with optional AAD
- `sign_message()` / `verify_signature()` — Ed25519
- `seal_bus_message(plaintext, sender, receiver, aad=None)` / `open_bus_message(sealed, sender, receiver, aad=None)` — High-level seal/open with optional AAD

The `aad` parameter accepts bytes and is passed directly to AES-256-GCM as
Associated Authenticated Data. Callers SHOULD construct AAD per Section 6.1.1.
When `aad` is `None`, no associated data is bound (backward-compatible with
pre-AAD sealed messages).

Test suite: `reference/python/tests/test_crypto.py` (36 tests)

Dependencies: `cryptography>=41.0` (via pyproject.toml)

## 11. Future Work

### 11.1 Post-Quantum Cryptography

When NIST PQC standards (FIPS 203 Kyber, FIPS 204 Dilithium) achieve wider
library support, ARC-8446 SHOULD offer a hybrid mode:

- Key agreement: X25519 + ML-KEM (Kyber) combined
- Signatures: Ed25519 + ML-DSA (Dilithium) combined

This provides quantum resistance while maintaining backward compatibility
with classical-only implementations.

**Minimum library version**: Implementations targeting PQC hybrid mode MUST
use `cryptography>=42.0`, which provides ML-KEM support via OpenSSL 3.2+.

**Migration trigger**: Migration to hybrid mode SHALL be initiated when:
(a) the `cryptography` library ships stable ML-KEM/ML-DSA support, AND
(b) all participating clans agree to upgrade via bilateral explicit agreement.
Unilateral migration MUST NOT be performed, as it would break interoperability.

**Harvest-now-decrypt-later threat**: Adversaries with access to relay traffic
MAY store encrypted messages today for decryption once quantum computers become
viable. Clans exchanging high-sensitivity messages SHOULD evaluate the timeline
for PQC migration against the sensitivity lifetime of their data and consider
accelerating adoption when stable library support becomes available.

### 11.2 Ephemeral Key Exchange

Introduce per-session or per-message ephemeral DH keys to achieve forward
secrecy. The sealed message format would include an ephemeral X25519 public
key alongside the ciphertext.

### 11.3 Extended Associated Data

Basic AAD binding (sender, destination, message type) is specified in
Section 6.1.1. Future work MAY extend the AAD scope to include additional
fields such as timestamps, sequence numbers, or protocol version identifiers.
Extended AAD would provide stronger cryptographic binding between the envelope
metadata and the encrypted payload, further limiting the attack surface for
metadata manipulation.
