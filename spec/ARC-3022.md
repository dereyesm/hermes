> *"No protocol is an island. TCP/IP did not succeed because it was technically
> superior — it succeeded because it let heterogeneous networks interconnect
> without surrendering their autonomy. The gateway is the membrane that makes
> this possible: it preserves sovereignty inward and enables collaboration
> outward. Through it, independent clans form a hive — a colmena mundial —
> where every node serves its community while strengthening the whole."*

## 1. Abstract

This specification defines the Agent Gateway Protocol for HERMES deployments. The gateway is the boundary component between a clan's private namespace topology and the public inter-clan network (the **Agora**). It performs identity translation, outbound filtering, inbound validation, and profile publication -- enabling clans to collaborate without exposing their internal structure, credentials, or operational data.

The design mirrors RFC 3022's Network Address Translation: internal addresses are invisible to the outside world, translated at the boundary to public identifiers. A clan's private bus, routing table, agent configurations, and metrics remain sovereign. Only what the operator explicitly authorizes leaves the gateway.

## 2. Motivation

HERMES Phase 0 established a robust intra-clan protocol: namespaces communicate through a shared bus with firewall isolation (ARC-1918), structured addressing (ARC-0791), and typed messages (ARC-5322). However, Phase 0 assumes a single clan operating on a shared filesystem.

The next evolutionary step is **inter-clan communication** -- multiple independent HERMES deployments discovering each other, verifying capabilities, exchanging attestations, and collaborating on cross-clan tasks. This requires solving three problems:

1. **Privacy**: A clan's internal topology, agent names, bus messages, and metrics MUST NOT leak to external clans.
2. **Identity**: Agents need public identifiers that don't reveal their internal names or namespace structure.
3. **Trust**: Clans need a mechanism to verify each other's claims about agent capabilities without accessing each other's private data.

The Agent Gateway solves all three by acting as a NAT, firewall, and identity broker at the clan boundary.

## 3. Terminology

| Term | Definition |
|------|-----------|
| **Gateway** | The boundary component that mediates all communication between a clan's private space and the Agora |
| **Agora** | The public inter-clan network where clans discover each other and exchange attestations |
| **Clan** | A complete HERMES deployment with its own bus, routing table, namespaces, and operator (as defined in ARC-0001) |
| **Public Profile** | A structured document that a clan publishes to the Agora, declaring its agents' capabilities |
| **Internal Identity** | An agent's real name and namespace within the clan (e.g., `admin-ph` in namespace `community`) |
| **External Identity** | The public alias assigned to an agent for Agora interactions (e.g., `zeta-legal-alpha`) |
| **Translation Table** | The mapping from internal identities to external identities, maintained by the gateway |
| **Attestation** | A signed statement from one clan certifying that another clan's agent delivered value |
| **Resonance** | The public reputation metric for an agent, computed from verified attestations |
| **Bounty** | The private reputation metric for an agent, computed from internal XP, precision, and impact |
| **Outbound Filter** | The set of rules governing what information may leave the clan through the gateway |
| **Inbound Validator** | The set of rules governing what information from external clans is accepted into the clan |

## 4. Gateway Architecture

### 4.1 Position in the Stack

The gateway operates at the boundary between HERMES Layer 4 (Application/Orchestration) and the external Agora network:

```
┌─────────────────────────────────────────┐
│              CLAN (private)              │
│                                         │
│  L4: Orchestration (Dojo/Controller)    │
│  L3: Transport (bus.jsonl)              │
│  L2: Addressing (ARC-0791)             │
│  L1: Message Format (ARC-5322)          │
│  L0: Storage (filesystem)               │
│                                         │
│         ┌───────────────┐               │
│         │   GATEWAY     │ ← ARC-3022    │
│         │  ┌─────────┐  │               │
│         │  │ NAT     │  │               │
│         │  │ Filter  │  │               │
│         │  │ Profile │  │               │
│         │  └─────────┘  │               │
│         └───────┬───────┘               │
└─────────────────┼───────────────────────┘
                  │
    ══════════════╪════════════════════════
                  │
           ┌──────┴──────┐
           │   AGORA     │
           │  (public)   │
           └─────────────┘
```

### 4.2 Component Model

The gateway consists of four logical components:

| Component | Function | Analogy |
|-----------|----------|---------|
| **NAT (Identity Translator)** | Maps internal agent identities to external aliases | RFC 3022 NAT |
| **Outbound Filter** | Enforces what information leaves the clan | Egress firewall |
| **Inbound Validator** | Validates and routes external messages into the clan | Ingress firewall |
| **Profile Publisher** | Maintains the clan's public profile on the Agora | DNS/WHOIS |

### 4.3 Single Gateway Rule

An Amaru clan MUST have exactly one gateway. Multiple gateways create ambiguous identity translation and split-brain routing. The gateway is a singleton component, not a per-namespace construct.

### 4.4 Gateway is Not an Agent

The gateway is an **infrastructure component**, not an agent/skill. It does not have XP, Bounty, or a rank. It does not appear in the Dojo roster. It is a protocol-level mechanism that the operator configures and the runtime enforces.

However, an agent MAY be designated as the **gateway operator** -- responsible for proposing profile updates and reviewing inbound requests. This agent acts through the gateway but is not the gateway itself. The distinction is analogous to a network administrator versus the NAT device.

## 5. Identity Translation

### 5.1 Translation Table

The gateway maintains a translation table that maps internal identities to external aliases:

```yaml
gateway:
  clan_id: "momosho-d"  # Unique clan identifier on the Agora
  translation_table:
    - internal: {namespace: "community", agent: "admin-ph"}
      external: "zeta-legal-alpha"
      published: true
      capabilities: ["property-law", "financial-audit", "community-governance"]

    - internal: {namespace: "finance", agent: "plutus"}
      external: "aureus"
      published: true
      capabilities: ["personal-finance", "debt-strategy", "budget-optimization"]

    - internal: {namespace: "engineering", agent: "eng-lead"}
      external: null  # Not published -- internal only
      published: false
```

### 5.2 Translation Rules

1. **Every external identity MUST map to exactly one internal identity.** No aliasing to multiple agents.
2. **Multiple internal agents MAY share a composite external identity** if the operator explicitly configures it. This allows a "team" of agents to appear as a single entity externally.
3. **External aliases MUST NOT reveal internal names, namespaces, or clan structure.**
4. **The operator chooses all external aliases.** The gateway MUST NOT auto-generate aliases from internal names.
5. **Unpublished agents are invisible.** If `published: false`, the agent does not exist from the Agora's perspective.

### 5.3 Clan Identifier

Each clan on the Agora has a unique identifier:

```abnf
clan-id = 1*32( ALPHA / DIGIT / "-" )
```

The clan identifier is chosen by the operator and registered on the Agora. It MUST be unique across the Agora network. Resolution of uniqueness conflicts is defined by the Agora's directory service (future ATR-X.500).

## 6. Public Profile

### 6.1 Profile Structure

A clan's public profile is the only information visible to other clans on the Agora. It is maintained by the gateway and published to the Agora directory.

```yaml
profile:
  clan_id: "momosho-d"
  display_name: "Clan Momosho D."
  created: "2026-02-24"
  protocol_version: "0.1.0"

  agents:
    - alias: "zeta-legal-alpha"
      capabilities: ["property-law", "financial-audit", "community-governance"]
      resonance: 0
      attestations_received: 0
      active_since: "2026-02-24"

    - alias: "aureus"
      capabilities: ["personal-finance", "debt-strategy", "budget-optimization"]
      resonance: 0
      attestations_received: 0
      active_since: "2026-02-24"

  clan_stats:
    total_published_agents: 2
    total_resonance: 0
    agora_member_since: "2026-03-01"
```

### 6.2 Profile Rules

1. **No internal data.** The profile MUST NOT contain namespace names, internal agent names, bus messages, Bounty scores, XP values, medallas, or any Dojo/evolution data.
2. **Capabilities are self-declared.** A clan declares what its agents can do. Other clans verify through attestations (Section 8).
3. **Resonance is computed, not declared.** The Resonance score is calculated from attestations received, not self-reported. The gateway updates it based on verified attestations.
4. **Profile updates require operator approval.** The gateway MUST NOT auto-publish profile changes.
5. **Minimal exposure principle.** Publish the minimum necessary. Capabilities SHOULD be described in generic terms (e.g., "financial-audit") rather than implementation details (e.g., "reads-nequi-api-and-computes-plutus-score").

### 6.3 What MUST NOT Appear in a Profile

| Data | Reason |
|------|--------|
| Internal agent names | Reveals clan structure |
| Namespace identifiers | Reveals organizational topology |
| Bus messages | Private operational data |
| Bounty / XP / Level / Rank | Internal gamification metrics |
| Medallas (badges) | Internal progression system |
| Tool/MCP bindings | Security-sensitive configuration |
| Account emails | Identity theft risk |
| Operator name | Privacy |
| Session logs | Operational history |
| Memory contents | Private knowledge base |

## 7. Outbound Rules

### 7.1 Default Deny

The gateway follows a **default-deny outbound policy**, inheriting the principle from ARC-1918. Nothing leaves the clan unless explicitly listed in the outbound rules.

### 7.2 Outbound Message Types

The gateway recognizes three categories of outbound communication:

| Category | What it is | Permitted by default |
|----------|-----------|---------------------|
| **Profile publication** | Updated public profile pushed to the Agora directory | YES (with operator approval) |
| **Attestation response** | A signed attestation certifying another clan's agent | NO (requires operator approval per instance) |
| **Quest response** | Result of a cross-clan quest/collaboration | NO (requires operator approval per instance) |

### 7.3 Outbound Filter Rules

```yaml
outbound_rules:
  # Profile updates: allowed with operator approval
  - type: profile_update
    action: allow
    approval: operator

  # Attestation issuance: case-by-case
  - type: attestation
    action: allow
    approval: operator_per_instance

  # Quest results: case-by-case
  - type: quest_response
    action: allow
    approval: operator_per_instance

  # Everything else: denied
  - type: "*"
    action: deny
```

### 7.4 Prohibited Outbound Data

The following MUST NEVER leave the gateway, regardless of rules:

1. Internal bus messages (any `msg` from `bus.jsonl`)
2. Routing table contents (`routes.md`)
3. Agent configurations (SKILL.md, registry.json)
4. Credentials, tokens, API keys
5. Memory files
6. Session logs
7. Evolution/Dojo data (XP, Bounty, level, medallas)

## 8. Inbound Rules

### 8.1 Inbound Message Types

The gateway accepts three categories of inbound communication from the Agora:

| Category | What it is | Action |
|----------|-----------|--------|
| **Discovery query** | Another clan asks for the public profile | Auto-respond with cached profile |
| **Attestation** | Another clan certifies one of our agents | Validate, store, update Resonance |
| **Quest proposal** | Another clan proposes a collaboration | Route to operator for review |
| **Hello** | A new clan introduces itself (see Section 15) | Auto-accept if valid Agora profile; otherwise quarantine |
| **Hello_ack** | Response to a hello we sent | Update peer status to `established` |

### 8.2 Inbound Validation

Every inbound message MUST pass through validation before reaching the clan's internal bus:

```
FUNCTION validate_inbound(message, source_clan):
    // Step 1: Source verification
    IF source_clan NOT IN agora_directory THEN
        RETURN reject("unknown source clan")

    // Step 2: Message type check
    IF message.type NOT IN {discovery, attestation, quest_proposal, hello, hello_ack} THEN
        RETURN reject("unsupported inbound type")

    // Step 3: Target resolution
    IF message.target_agent NOT IN published_agents THEN
        RETURN reject("unknown agent")

    // Step 4: Payload inspection
    IF contains_executable_content(message.payload) THEN
        RETURN reject("executable content not permitted")
    IF payload_size(message) > MAX_INBOUND_SIZE THEN
        RETURN reject("payload too large")

    // Step 5: Rate limiting
    IF rate_limit_exceeded(source_clan) THEN
        RETURN reject("rate limit exceeded")

    // Step 6: Route to operator
    internal_msg = translate_to_internal(message)
    RETURN route_to_operator(internal_msg)
```

### 8.3 Inbound-to-Internal Translation

When the gateway accepts an inbound message, it translates it to an internal bus message:

```
External (from Agora):
{
  "source_clan": "nakama-crew",
  "target_agent": "aureus",
  "type": "attestation",
  "payload": "quest-debt-strategy completed. quality: excellent."
}

Internal (to bus.jsonl):
{
  "ts": "2026-03-15",
  "src": "gateway",
  "dst": "finance",
  "type": "event",
  "msg": "AGORA:attestation from nakama-crew for aureus. quality:excellent",
  "ttl": 7,
  "ack": []
}
```

Note the `AGORA:` prefix, which marks messages that originated from outside the clan. This allows internal agents and the operator to distinguish external signals from internal ones.

### 8.4 Quarantine

Inbound messages that pass validation but seem anomalous (unusual source, first-time clan, high volume) SHOULD be quarantined for operator review rather than auto-routed to the internal bus.

## 9. Attestation Protocol

### 9.1 What is an Attestation

An attestation is a statement from Clan A certifying that an agent from Clan B delivered measurable value during a cross-clan interaction. Attestations are the building blocks of the Resonance metric.

### 9.2 Attestation Format

```yaml
attestation:
  id: "att-20260315-001"
  from_clan: "nakama-crew"
  to_clan: "momosho-d"
  to_agent: "aureus"          # External alias only
  quest_id: "quest-2026-042"  # Optional: reference to the collaboration
  timestamp: "2026-03-15"
  rating:
    quality: 5       # 1-5 scale
    reliability: 5   # 1-5: did the agent deliver on time?
    collaboration: 4 # 1-5: how well did it work with our agents?
  summary: "Provided debt restructuring strategy that reduced projected costs by 15%."
  signature: "<clan-signature>"  # Cryptographic signature (see Section 9.4)
```

### 9.3 Attestation Rules

1. **Attestations are asymmetric.** Clan A attests for Clan B's agent. The reverse requires a separate attestation.
2. **Self-attestation is prohibited.** A clan MUST NOT issue attestations for its own agents.
3. **One attestation per quest per agent pair.** Duplicate attestations for the same interaction are rejected.
4. **Attestations are append-only.** Once issued, an attestation cannot be modified or retracted. However, a subsequent attestation CAN express a different rating.
5. **Attestations reference external aliases only.** The attesting clan never knows the internal identity of the agent it is certifying.
6. **The operator reviews all incoming attestations** before they affect the Resonance score.

### 9.4 Signature and Verification

Each clan generates a key pair at gateway initialization:

- **Private key**: Stored in the clan's private space, never leaves the gateway
- **Public key**: Published in the clan's Agora profile

Attestations are signed with the issuing clan's private key. Any clan can verify the signature using the public key from the Agora directory.

The specific cryptographic algorithm is out of scope for this specification. Implementations SHOULD use Ed25519 for initial deployments and MAY upgrade to post-quantum schemes as defined by ARC-8446 when available.

## 10. Resonance Metric

### 10.1 Definition

**Resonance** is the public reputation score of an agent on the Agora. It is computed from verified attestations received from other clans. Unlike Bounty (which is internal and self-computed), Resonance is externally validated and cryptographically verifiable.

### 10.2 Computation

```
Resonance(agent) = Σ attestation_score(a) × recency_weight(a) × diversity_weight(a)
                   for each verified attestation a

Where:
  attestation_score(a) = (quality + reliability + collaboration) / 3
                         # Normalized to 1-5 scale

  recency_weight(a)    = max(0, 1 - (days_since(a) / 365))
                         # Linear decay over 1 year

  diversity_weight(a)  = 1 + (0.1 × unique_clans_attesting)
                         # Bonus for attestations from different clans
```

### 10.3 Properties

| Property | Description |
|----------|-----------|
| **Externally computed** | Resonance is calculated from attestations, not self-reported |
| **Decays over time** | Old attestations contribute less, encouraging continuous value delivery |
| **Diversity bonus** | Attestations from many different clans are worth more than many from one clan |
| **Cannot be gamed** | Requires cryptographically signed attestations from real clans |
| **Starts at zero** | Every agent begins with Resonance 0 regardless of internal Bounty |
| **Public** | Visible to all clans on the Agora |

### 10.4 Resonance vs. Bounty

| Aspect | Bounty (internal) | Resonance (external) |
|--------|-------------------|---------------------|
| **Computed by** | Dojo (clan's evolution system) | Attestation protocol |
| **Visible to** | Clan only | All clans on the Agora |
| **Based on** | XP, precision, impact, medallas | External attestations |
| **Can be gamed** | No (operator controls evaluation) | No (requires external signatures) |
| **Decays** | No (XP is permanent) | Yes (recency weight) |
| **Meaning** | "How good is this agent internally?" | "How much value has this agent proven to the world?" |

### 10.5 Bounty-to-Resonance Relationship

Bounty and Resonance are complementary metrics that measure different dimensions of value:

- **High Bounty, low Resonance**: A strong agent that hasn't yet proven itself externally. Common for new agents or clans that haven't joined the Agora.
- **Low Bounty, high Resonance**: An agent that is more valued externally than internally. Rare but possible if the agent excels at cross-clan collaboration.
- **High both**: A proven agent, valued internally and externally.
- **Low both**: A new or underperforming agent.

The gateway MUST NOT expose Bounty values through the public profile. Resonance is the only reputation metric visible externally.

## 11. Inter-Instance Peering

### 11.1 Agora Discovery

Clans discover each other through the Agora directory. The directory is a shared registry of clan profiles, accessible to all participants.

The Agora directory MAY be implemented as:

- A Git repository (profiles as files, PRs for registration)
- A distributed hash table (DHT)
- A centralized registry (for early bootstrapping)
- A federated model (regional registries with cross-references)

This specification does not mandate a specific implementation. The reference implementation uses a Git-based directory for simplicity, auditability, and alignment with HERMES's file-based philosophy.

### 11.2 Peering Handshake

When two clans wish to collaborate, the following handshake occurs:

```
Clan A Gateway                  Agora Directory                  Clan B Gateway
     |                               |                               |
     | 1. discover(capability)       |                               |
     |------------------------------>|                               |
     |                               |                               |
     | 2. profile(clan-b)            |                               |
     |<------------------------------|                               |
     |                               |                               |
     | 3. quest_proposal(target)     |                               |
     |-------------------------------------------------------------->|
     |                               |                               |
     |                               |          4. validate + route  |
     |                               |          to operator          |
     |                               |                               |
     | 5. quest_accepted / rejected  |                               |
     |<--------------------------------------------------------------|
     |                               |                               |
     | [collaboration occurs]        |                               |
     |                               |                               |
     | 6. attestation(agent, rating) |                               |
     |<--------------------------------------------------------------|
     |                               |                               |
     | 7. attestation(agent, rating) |                               |
     |-------------------------------------------------------------->|
     |                               |                               |
```

### 11.3 Trust Model

Inter-clan trust follows the **Trust-On-First-Use (TOFU)** model, similar to SSH:

1. The first time Clan A interacts with Clan B, it records Clan B's public key.
2. Subsequent interactions verify that the key has not changed.
3. If the key changes, the gateway alerts the operator.

Future specifications (ARC-8446, ATR-X.509) MAY upgrade this to certificate-based trust with a PKI model.

### 11.4 Hive Topology

When multiple clans join the Agora, their gateways form a decentralized mesh — the **hive** (colmena). Each clan is an autonomous node; gateways are the edges; the Agora is the shared discovery layer.

```
                        ┌─────────────────────────┐
                        │        A G O R A         │
                        │   (Directory / Registry) │
                        └──┬──────┬──────┬──────┬──┘
                           │      │      │      │
                    ┌──────┴──┐ ┌─┴──────┴─┐ ┌──┴──────┐
                    │ Gateway │ │ Gateway   │ │ Gateway │
                    │  Clan A │ │  Clan B   │ │  Clan C │
                    ├─────────┤ ├───────────┤ ├─────────┤
                    │ Private │ │  Private  │ │ Private │
                    │  Bus A  │ │   Bus B   │ │  Bus C  │
                    │ ns₁ ns₂ │ │ ns₃ ns₄  │ │ ns₅     │
                    └─────────┘ └───────────┘ └─────────┘
                         │                          │
                         │    Quest / Attestation    │
                         └──────────────────────────┘
```

**Hive properties:**

1. **No central authority.** The Agora is a directory, not a controller. No clan has authority over another.
2. **Sovereignty preserved.** Each clan's internal topology, bus, and credentials remain invisible to the network.
3. **Organic growth.** New clans join by publishing a profile. No approval process beyond key registration.
4. **Resilient through decentralization.** If a clan goes offline, the rest of the hive continues. Attestations persist.
5. **Emergent trust.** Reputation emerges from attestations, not from central assignment. The hive rewards value creation.

This topology mirrors the internet's Autonomous System (AS) model: each clan is an AS, each gateway is a border router, and the Agora is the DNS/BGP of the agent world.

### 11.5 Gateway as Protocol Bridge

The gateway is the natural integration point between HERMES's file-based world and the broader internet-based protocol ecosystem (MCP, A2A, ACP, ANP). A gateway implementation MAY translate between formats:

| Direction | From | To | Method |
|-----------|------|----|--------|
| **Outbound** | ARC-5322 bus message | External protocol format | Gateway serializer |
| **Inbound** | External protocol message | ARC-5322 bus message with `AGORA:` prefix | Gateway deserializer |

The security model is invariant regardless of transport protocol: default-deny outbound, validated inbound, operator approval for all cross-boundary actions. Detailed protocol mappings are out of scope for this specification and will be addressed in future extension specs.

## 12. Gateway Configuration

### 12.1 Configuration File

The gateway is configured through a YAML file in the clan's infrastructure directory:

```yaml
# gateway.yaml — Clan boundary configuration
clan_id: "momosho-d"
display_name: "Clan Momosho D."
protocol_version: "0.1.0"

# Keypair for signing attestations
keys:
  private: ".keys/gateway.key"     # Relative to clan root, NEVER published
  public: ".keys/gateway.pub"      # Published to Agora directory

# Identity translation table
agents:
  - internal: {namespace: "community", agent: "admin-ph"}
    external: "zeta-legal-alpha"
    published: true
    capabilities: ["property-law", "financial-audit", "community-governance"]

  - internal: {namespace: "finance", agent: "plutus"}
    external: "aureus"
    published: true
    capabilities: ["personal-finance", "debt-strategy", "budget-optimization"]

# Outbound rules
outbound:
  profile_update: {action: allow, approval: operator}
  attestation: {action: allow, approval: operator_per_instance}
  quest_response: {action: allow, approval: operator_per_instance}
  default: deny

# Inbound limits
inbound:
  max_payload_bytes: 4096
  rate_limit_per_clan: 10  # messages per day
  quarantine_first_contact: true

# Agora directory
agora:
  type: git  # git | dht | registry
  url: "https://github.com/hermes-agora/directory"
```

### 12.2 Key Management

1. The private key MUST be stored in the clan's private space, protected by file system permissions.
2. The private key MUST NOT be committed to any repository.
3. Key rotation SHOULD be supported. The old public key remains in the Agora directory with a `deprecated_since` date.
4. If the private key is compromised, the operator MUST rotate keys and all previous attestations issued by this clan become unverifiable until re-signed.

## 13. Internal Bus Integration

### 13.1 Gateway as Bus Participant

The gateway appears on the internal bus as a special source: `gateway`. It is NOT a namespace -- it is an infrastructure source.

```json
{"ts":"2026-03-15","src":"gateway","dst":"controller","type":"event","msg":"AGORA:quest_proposal from nakama-crew targeting aureus. review_required","ttl":5,"ack":[]}
```

### 13.2 AGORA: Prefix Convention

All messages originating from outside the clan carry the `AGORA:` prefix in the `msg` field. This makes external signals visually and programmatically distinguishable from internal ones.

Agents and the operator SHOULD treat `AGORA:` messages with appropriate scrutiny -- they represent claims from external, potentially untrusted sources.

### 13.3 Gateway Message Types

The gateway emits the following message types on the internal bus:

| Internal type | Trigger | Example |
|---------------|---------|---------|
| `event` | Quest proposal received | `AGORA:quest_proposal from clan-x targeting alias-y` |
| `event` | Attestation received | `AGORA:attestation from clan-x for alias-y. quality:4` |
| `alert` | Key mismatch detected | `AGORA:key_changed for clan-x. possible impersonation` |
| `state` | Profile published | `AGORA:profile_updated. 2 agents published. resonance:47` |

## 14. Security Considerations

### 14.1 Threat Model

| Threat | Mitigation |
|--------|-----------|
| External clan reads internal bus | Gateway is the only boundary. Internal bus is never exposed. |
| Attestation forgery | Cryptographic signatures on all attestations |
| Identity probing (guessing internal names from external aliases) | Aliases are operator-chosen, unrelated to internal names |
| Resonance manipulation (colluding clans) | Diversity weight penalizes attestations from a single source |
| Denial of service (attestation flood) | Rate limiting per source clan |
| Key compromise | Key rotation + deprecation mechanism |
| Profile inflation (false capabilities) | Capabilities are self-declared but only Resonance (from external attestations) carries weight |
| MITM on Agora directory | Git-based directory provides audit trail + signed commits |

### 14.2 Privacy Guarantees

The gateway provides the following privacy guarantees:

1. **Topology hiding**: External clans cannot determine how many namespaces exist, what they are called, or how they are structured.
2. **Agent privacy**: Internal agent names, configurations, and metrics are never exposed.
3. **Operational privacy**: Bus messages, session logs, and memory are never exposed.
4. **Selective disclosure**: The operator chooses exactly which agents to publish and what capabilities to declare.

### 14.3 Non-Goals

This specification does NOT address:

- End-to-end encryption of inter-clan messages (see future ARC-8446)
- Reputation portability (moving Resonance between Agora implementations)
- Dispute resolution (when clans disagree on attestation accuracy)
- Automated quest matching (future Agora application layer)
- Visual representation of agents (future AES-2040 specification)

## 15. Multi-Clan Bootstrap Protocol

This section defines how two fresh clans discover each other and establish their first connection through the Agora. It builds on the peering handshake (Section 11.2) and trust model (Section 11.3) by specifying the concrete identity schema, ceremony, and message types required for a cold start between clans that have never interacted.

### 15.1 Clan Identity Schema

Every clan on the Agora MUST publish a structured identity document. This schema extends the public profile (Section 6.1) with the fields necessary for trust bootstrap:

```yaml
clan_identity:
  clan_id: string           # 1-32 chars, [a-z0-9-], unique on Agora
  display_name: string      # Human-readable name
  operator: string          # Operator alias (MUST NOT be a real name — see 14.2)
  created: date             # ISO 8601 date (YYYY-MM-DD)
  protocol_version: string  # Semver (e.g., "0.3.0")
  gateway:
    public_key: string      # Ed25519 public key (hex or base64)
    agora_url: string       # URL of the Agora directory this clan uses
  heraldo:                  # Inter-clan messenger configuration
    alias: string           # External alias for the clan's messenger agent
    capabilities: ["inter-clan-messaging"]
```

**Schema rules:**

| Field | Constraint | Rationale |
|-------|-----------|-----------|
| `clan_id` | `1*32( ALPHA / DIGIT / "-" )` per ARC-0791 | Matches addressing grammar; prevents namespace collisions |
| `display_name` | UTF-8, max 64 chars | Human readability on directory listings |
| `operator` | Alias only, never real identity | Privacy (Section 14.2) |
| `public_key` | Ed25519 (SHOULD), Dilithium (MAY, per ARC-8446) | TOFU verification anchor |
| `agora_url` | HTTPS URL | Identifies which Agora instance this clan is registered on |
| `heraldo.alias` | Same grammar as external agent aliases | The inter-clan messenger is a published agent (Section 5.2) |

The `heraldo` field designates the agent responsible for receiving and sending inter-clan handshake messages. Every clan that participates in the Agora MUST declare a heraldo. The heraldo is a published agent (Section 5.1) with at minimum the `inter-clan-messaging` capability.

### 15.2 Trust Bootstrap Ceremony

The Trust Bootstrap Ceremony defines the TOFU handshake between two clans that have never interacted. It extends the trust model (Section 11.3) with concrete message types and state transitions.

**Ceremony steps:**

```
Step  Action       Actor    Description
────  ──────────   ──────   ────────────────────────────────────────────────
 1    ANNOUNCE     Clan A   Publishes clan_identity to Agora directory
                            (Git PR or direct push, per Agora policy)
 2    DISCOVER     Clan B   Finds Clan A via capability query on Agora
                            (ARC-2606 discovery queries)
 3    VERIFY       Clan B   Reads Clan A's public key from Agora profile
                            (TOFU: first seen = trusted, per Section 11.3)
 4    PEER         Clan B   Adds Clan A to its known_clans set in gateway
                            config (Section 16, peers array)
 5    HELLO        Clan B   Sends a hello message to Clan A's heraldo
                            via the Agora inbox
 6    ACK_HELLO    Clan A   Validates hello, adds Clan B to known_clans,
                            responds with hello_ack
 7    ESTABLISHED  Both     Bidirectional channel open — clans may exchange
                            discovery, attestation, and quest_proposal messages
```

**State machine per peer:**

```
                  ┌──────────┐
                  │ UNKNOWN  │
                  └────┬─────┘
                       │ peer add (Step 4)
                       ▼
                  ┌──────────────┐
                  │ PENDING_ACK  │──── timeout (7d) ──→ EXPIRED
                  └────┬─────────┘
                       │ hello_ack received (Step 6)
                       ▼
                  ┌──────────────┐
                  │ ESTABLISHED  │
                  └────┬─────────┘
                       │ operator suspend
                       ▼
                  ┌──────────────┐
                  │  SUSPENDED   │
                  └──────────────┘
```

If a `hello` is received from a clan NOT in known_clans, the gateway SHOULD auto-accept if the source has a valid profile on the Agora (Section 15.6). This enables organic discovery without requiring both sides to manually add each other.

### 15.3 Peer Registration Flow

The `amaru peer add` command provides the operator interface for Step 4 of the bootstrap ceremony:

```
hermes peer add <clan-id>

  1. Fetch profile from Agora:     GET profiles/<clan-id>.yaml
  2. Validate profile structure:   Verify required fields per Section 15.1
  3. Store public key locally:     Write to .keys/peers/<clan-id>.pub
  4. Update gateway config:        Add clan-id to peers[] in gateway.yaml
  5. Send hello message:           Deposit hello to Agora inbox/<clan-id>/
  6. Wait for hello_ack:           Poll or async (timeout: 7 days TTL)
  7. Log result:                   "Peer <clan-id> registered. Status: pending_ack"

On hello_ack received:
  8. Update peer status:           pending_ack → established
  9. Emit internal bus event:      src=gateway, type=event,
                                   msg="AGORA:peer_established <clan-id>"
```

**CLI output example:**

```
$ hermes peer add jeimmy-clan
Fetching profile from agora... OK
  clan_id:          jeimmy-clan
  display_name:     Clan Jeimmy
  protocol_version: 0.3.0
  public_key:       Ed25519:a3f8...c7e2
Storing public key... .keys/peers/jeimmy-clan.pub
Sending hello... deposited to inbox/jeimmy-clan/msg-20260306-001.json
Status: pending_ack (TTL: 7 days)
```

### 15.4 First Message Protocol

After the bootstrap ceremony reaches ESTABLISHED state, the first operational message flow follows the quest lifecycle:

```
Clan A                                Clan B
  │                                     │
  │ 1. quest_proposal(target, brief)    │
  │────────────────────────────────────→│
  │                                     │
  │      [Clan B operator reviews]      │
  │      [Routes to internal agent]     │
  │                                     │
  │ 2. quest_accepted(cid)              │
  │←────────────────────────────────────│
  │                                     │
  │ 3. quest_response(cid, payload)     │
  │←────────────────────────────────────│
  │                                     │
  │ 4. attestation(agent, rating)       │
  │────────────────────────────────────→│
  │                                     │
```

**Message correlation:** All messages in a quest sequence share a Correlation ID (`cid`) as defined in ARC-0768 Section 4.1. The `quest_proposal` generates the initial CID; subsequent messages reference it via the `RE` (References) field per ARC-0768 Section 4.2.

**Outbound filtering:** The `quest_response` payload passes through the outbound filter (Section 7). The gateway MUST strip any data that violates Section 7.4 (prohibited outbound data) before transmission.

**Attestation timing:** The attestation (Step 4) SHOULD be issued within 30 days of quest completion. Attestations without a corresponding `quest_id` are valid but carry lower weight in Resonance computation (Section 10.2) since they cannot be correlated to a specific interaction.

### 15.5 Agora Directory Structure

The reference Agora implementation uses a Git repository, consistent with HERMES's file-based philosophy (Section 11.1). The directory structure:

```
hermes-agora/                    # Git repository (public)
├── profiles/                    # Clan identity documents (Section 15.1)
│   ├── momosho-d.yaml           # One file per clan
│   └── jeimmy-clan.yaml
├── inbox/                       # Per-clan message drops
│   ├── momosho-d/               # Messages addressed to momosho-d
│   │   └── <msg-id>.json        # Individual inbound messages
│   └── jeimmy-clan/
│       └── <msg-id>.json
├── attestations/                # Published attestations (append-only)
│   └── <attestation-id>.yaml    # Follows format in Section 9.2
├── quest_log/                   # Quest metadata (no payloads)
│   └── <quest-id>.yaml          # Status tracking: proposed → accepted → completed
└── README.md                    # Agora usage instructions
```

**Directory rules:**

| Rule | Rationale |
|------|-----------|
| Profiles are YAML, one per clan | Consistent with gateway.yaml format; human-readable |
| Inbox messages are JSON (ARC-5322 format) | Machine-parseable; consistent with bus.jsonl |
| Attestations are append-only | Immutability per Section 9.3 rule 4 |
| Quest log contains metadata only, never payloads | Payloads are exchanged directly between gateways |
| `msg-id` format: `msg-YYYYMMDD-NNN` | Sortable, collision-resistant within a day |
| All files MUST be signed (Git signed commits) | Tamper evidence per Section 14.1 |

**Inbox polling:** A clan's gateway periodically checks its inbox directory for new messages. The sync interval is configured in `gateway.yaml` (Section 16, `agora.sync_interval`). For the Git-based implementation, this is a `git pull` on the Agora repository.

### 15.6 Hello / Hello_Ack Message Types

The `hello` and `hello_ack` message types are added to the set of supported inbound types (Section 8.1):

| Type | Direction | Purpose | Approval |
|------|-----------|---------|----------|
| `hello` | Inbound | New clan introduces itself to this clan | Auto-accept if valid Agora profile |
| `hello_ack` | Inbound (response) | Peer confirms our hello was received | Auto-process (updates peer status) |

**Hello message format** (deposited in target clan's Agora inbox):

```json
{
  "type": "hello",
  "source_clan": "jeimmy-clan",
  "source_key": "Ed25519:a3f8...c7e2",
  "display_name": "Clan Jeimmy",
  "capabilities_summary": "Creative writing, editorial review, community engagement",
  "protocol_version": "0.3.0",
  "timestamp": "2026-03-06",
  "signature": "<signed-with-source-clan-private-key>"
}
```

**Hello_ack message format:**

```json
{
  "type": "hello_ack",
  "source_clan": "momosho-d",
  "target_clan": "jeimmy-clan",
  "status": "accepted",
  "timestamp": "2026-03-06",
  "signature": "<signed-with-source-clan-private-key>"
}
```

**Processing rules:**

1. **Hello auto-accept:** A `hello` is auto-accepted (no operator approval needed) if and only if the `source_clan` has a valid profile on the Agora AND the `source_key` matches the public key in that profile. The hello is an introduction, not a quest — it carries no obligation.
2. **Hello from unknown clan:** If the source clan has no Agora profile, the hello MUST be quarantined (Section 8.4) for operator review.
3. **Duplicate hello:** If a `hello` is received from a clan already in `known_clans`, it is idempotent — the gateway responds with `hello_ack` but does not change state.
4. **Hello_ack processing:** On receiving `hello_ack`, the gateway updates the peer's status from `pending_ack` to `established` and emits an internal bus event.
5. **Signature verification:** Both `hello` and `hello_ack` MUST be signed with the source clan's private key. The receiving gateway verifies against the public key from the Agora profile.

**Internal bus notification on hello received:**

```json
{
  "ts": "2026-03-06",
  "src": "gateway",
  "dst": "controller",
  "type": "event",
  "msg": "AGORA:hello from jeimmy-clan. auto-accepted. peer status: established",
  "ttl": 5,
  "ack": []
}
```

## 16. Gateway Configuration Reference

This section provides a comprehensive reference for `gateway.yaml`, expanding on the configuration introduced in Section 12.1. Implementations MUST support all fields marked REQUIRED; fields marked OPTIONAL have the indicated defaults.

```yaml
# gateway.yaml — Complete configuration reference
# ================================================

# ── Clan Identity (REQUIRED) ──────────────────────────────────────────────
clan_id: "momosho-d"              # REQUIRED. 1-32 chars, [a-z0-9-].
                                  # Must match Agora profile clan_id.
display_name: "Clan Momosho D."   # REQUIRED. Human-readable name (max 64 chars).
protocol_version: "0.3.0"        # REQUIRED. Semver. Peers with incompatible
                                  # major versions MUST NOT exchange messages.

# ── Cryptographic Identity (REQUIRED) ─────────────────────────────────────
keys:
  private: ".keys/gateway.key"    # REQUIRED. Path relative to clan root.
                                  # MUST NOT be committed to any repository.
                                  # MUST be protected by filesystem permissions.
  public: ".keys/gateway.pub"     # REQUIRED. Published to Agora profile.
  algorithm: "ed25519"            # OPTIONAL. Default: "ed25519".
                                  # Future: "dilithium" per ARC-8446.

# ── Identity Translation Table (REQUIRED) ─────────────────────────────────
agents:
  - internal: {namespace: "community", agent: "admin-ph"}
    external: "zeta-legal-alpha"  # Operator-chosen alias (Section 5.2)
    published: true               # Visible on Agora profile
    capabilities:                 # Self-declared (Section 6.2 rule 2)
      - "legal/property/governance"
      - "finance/audit/community"

  - internal: {namespace: "finance", agent: "plutus"}
    external: "aureus"
    published: true
    capabilities:
      - "finance/personal/strategy"
      - "finance/debt/optimization"

  - internal: {namespace: "engineering", agent: "eng-lead"}
    external: null                # Not published — internal only
    published: false

# ── Heraldo — Inter-Clan Messenger (REQUIRED for Agora participation) ─────
heraldo:
  internal: {namespace: "global", agent: "heraldo"}
  external: "herald"             # External alias for the heraldo
  capabilities: ["inter-clan-messaging"]
                                  # The heraldo handles hello/hello_ack and
                                  # routes inter-clan messages to the operator.

# ── Outbound Rules (REQUIRED) ─────────────────────────────────────────────
outbound:
  profile_update:                 # Publishing clan profile to Agora
    action: allow
    approval: operator            # Requires explicit operator approval
  attestation:                    # Issuing attestations for external agents
    action: allow
    approval: operator_per_instance  # Case-by-case review
  quest_response:                 # Sending quest results to external clans
    action: allow
    approval: operator_per_instance
  hello_ack:                      # Responding to hello messages
    action: allow
    approval: auto                # Auto-approved for valid Agora profiles
  default: deny                   # Everything not listed is blocked (Section 7.1)

# ── Inbound Limits (REQUIRED) ─────────────────────────────────────────────
inbound:
  max_payload_bytes: 4096         # REQUIRED. Maximum payload size per message.
                                  # Messages exceeding this are rejected.
  rate_limit_per_clan: 10         # REQUIRED. Max messages per clan per day.
                                  # Prevents attestation/quest flooding.
  quarantine_first_contact: true  # OPTIONAL. Default: true.
                                  # First message from unknown clan goes to
                                  # operator review (Section 8.4).
  auto_accept_hello: true         # OPTIONAL. Default: true.
                                  # Auto-accept hello from valid Agora profiles
                                  # (Section 15.6 rule 1).

# ── Known Peers (OPTIONAL, populated by hermes peer add) ──────────────────
peers:
  - clan_id: "jeimmy-clan"
    public_key_file: ".keys/peers/jeimmy-clan.pub"
    status: "established"         # pending_ack | established | suspended
    added: "2026-03-06"           # ISO date when peer was added
    last_hello_ack: "2026-03-06"  # ISO date of last hello_ack exchange

# ── Agora Directory (REQUIRED for inter-clan communication) ───────────────
agora:
  type: "git"                     # REQUIRED. Implementation type.
                                  # Supported: "git" (reference implementation).
                                  # Future: "dht", "registry", "federated".
  url: "https://github.com/hermes-agora/directory"
                                  # REQUIRED. URL of the Agora directory.
  local_cache: ".agora/"          # OPTIONAL. Default: ".agora/".
                                  # Local clone/cache of the Agora directory.
  sync_interval: "manual"         # OPTIONAL. Default: "manual".
                                  # Supported: "manual", "hourly", "daily".
                                  # "manual" = operator triggers sync explicitly.
```

### 16.1 Configuration Validation

The gateway MUST validate `gateway.yaml` at startup. Validation rules:

| Check | Severity | Description |
|-------|----------|-------------|
| `clan_id` matches ABNF grammar | MUST | Reject startup if malformed |
| `keys.private` file exists and is readable | MUST | Reject startup if missing |
| `keys.public` file exists | MUST | Reject startup if missing |
| All `external` aliases are unique | MUST | No two agents share an alias |
| `heraldo` entry exists with `inter-clan-messaging` capability | MUST | Required for Agora participation |
| `protocol_version` is valid semver | MUST | Reject startup if malformed |
| `agents[].capabilities` use ontology domains (ARC-2606) | SHOULD | Warn if free-form |
| `peers[].public_key_file` exists for each peer | SHOULD | Warn if missing |
| `agora.url` is reachable | SHOULD | Warn if unreachable (offline mode allowed) |

### 16.2 Relationship to Section 12.1

Section 12.1 introduced the basic `gateway.yaml` structure. This section (16) serves as the normative reference with:

- All fields documented with REQUIRED/OPTIONAL status
- Default values specified for all OPTIONAL fields
- The `heraldo` block (required for bootstrap, Section 15)
- The `peers` array (populated by `amaru peer add`, Section 15.3)
- The `auto_accept_hello` inbound flag (Section 15.6)
- The `hello_ack` outbound rule (auto-approved)
- The `agora.local_cache` and `agora.sync_interval` fields

Implementations SHOULD treat Section 16 as authoritative when Section 12.1 and Section 16 diverge.

## 17. Compatibility

### 17.1 Backward Compatibility

ARC-3022 is an **extension** specification. An Amaru deployment that does not implement ARC-3022 remains fully functional -- it simply cannot participate in the Agora. The gateway is optional.

Sections 15-16 (Multi-Clan Bootstrap Protocol and Gateway Configuration Reference) are additive extensions. Existing gateway deployments that do not implement `hello`/`hello_ack` or the heraldo role continue to function for intra-clan operations. Full Agora participation requires the fields defined in Section 16.

### 17.2 Dependencies

| Specification | Dependency type | What ARC-3022 uses |
|---------------|----------------|-------------------|
| ARC-0001 | Required | Clan definition, bus model, operator authority |
| ARC-1918 | Required | Firewall model extended to the clan boundary |
| ARC-0791 | Required | Internal addressing (for identity translation) |
| ARC-0793 | Required | Session lifecycle (SYN/FIN for gateway bus integration) |
| ARC-5322 | Required | Message format (for internal bus integration) |
| ARC-8446 | Optional | Cryptographic primitives (TOFU used if unavailable) |
| ATR-X.500 | Future | Agora directory service specification |
| ATR-X.509 | Future | PKI-based trust (upgrade from TOFU) |

## 19. Examples

### 19.1 Clan Publishes Profile to Agora

```
Operator: "Publish our profile with 2 agents."
Gateway: Reads gateway.yaml, builds profile, signs with private key.
Agora: Receives profile, verifies signature, stores in directory.
Result: Clan "momosho-d" is now discoverable with agents "zeta-legal-alpha" and "aureus".
```

### 19.2 Cross-Clan Quest

```
Clan "nakama-crew" discovers "momosho-d" on the Agora.
Nakama-crew proposes a quest: "Need debt restructuring strategy for a client scenario."
Momosho-d gateway receives the proposal, routes to operator.
Operator approves. Internal bus dispatches to "finance" namespace, targeting "plutus".
Plutus works on the quest (internally). Results go through gateway outbound filter.
Gateway sends filtered response to nakama-crew.
Nakama-crew issues attestation: quality 5, reliability 5, collaboration 4.
Momosho-d gateway receives attestation, validates signature, updates Resonance.
Aureus now has Resonance > 0.
```

### 19.3 Inbound Rejection

```
Unknown clan "shadow-corp" sends a quest proposal.
Gateway checks: shadow-corp NOT in Agora directory.
Result: REJECT("unknown source clan"). Message never reaches internal bus.
```
