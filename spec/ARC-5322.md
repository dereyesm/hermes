## 1. Abstract

This specification defines the wire format for HERMES inter-namespace messages. Each message is a single JSON object occupying exactly one line in a JSONL bus file. The format is designed for simplicity, human readability, file-based transport, and zero external dependencies.

The design philosophy follows Shannon's information theory: every field is a deliberate signal. The 120-character payload constraint forces precision over verbosity, treating bandwidth as a discipline rather than a limitation.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

**Namespace**: An isolated domain of responsibility within a HERMES deployment (e.g., "engineering", "finance", "operations"). Namespaces are the addressing unit.

**Bus file**: A JSONL file where each line is exactly one HERMES message. The bus file is the transport medium.

**Archive file**: A JSONL file where expired messages are moved for historical reference.

## 3. Transport Layer

### 3.1. Encoding

Messages MUST be encoded as UTF-8 JSON. Each message MUST occupy exactly one line (no embedded newlines). The bus file MUST use `\n` (LF, U+000A) as the line terminator.

### 3.2. File Format

The bus file is a valid JSONL file (JSON Lines, one JSON object per line). Implementations MUST be able to parse the file by reading it line by line and deserializing each line independently.

### 3.3. Atomicity

Each line is an atomic message. Implementations MUST NOT split a single message across multiple lines, nor combine multiple messages into a single line.

## 4. Message Structure

A HERMES message is a JSON object with exactly 7 fields:

```json
{"ts":"YYYY-MM-DD","src":"namespace","dst":"namespace|*","type":"type","msg":"payload","ttl":7,"ack":[]}
```

### 4.1. Field Specification

| Field | JSON Type | Required | Description |
|-------|-----------|----------|-------------|
| `ts`  | string    | YES      | Emission timestamp in ISO 8601 date format |
| `src` | string    | YES      | Source namespace identifier |
| `dst` | string    | YES      | Destination namespace identifier, or `"*"` for broadcast |
| `type`| string    | YES      | Message type (enumerated) |
| `msg` | string    | YES      | Compact payload |
| `ttl` | integer   | YES      | Time-to-live in days from `ts` |
| `ack` | array     | YES      | List of namespace identifiers that have consumed the message |

### 4.2. Field Constraints

#### 4.2.1. `ts` (Timestamp)

- Format: `YYYY-MM-DD` (ISO 8601 calendar date)
- MUST represent the date the message was emitted
- MUST NOT be a future date at time of creation
- Timezone interpretation is deployment-specific; UTC is RECOMMENDED

```abnf
ts = 4DIGIT "-" 2DIGIT "-" 2DIGIT
     ; YYYY-MM-DD, valid calendar date
```

#### 4.2.2. `src` (Source Namespace)

- MUST be a non-empty string
- MUST match the pattern: lowercase alphanumeric characters and hyphens
- MUST NOT be `"*"`
- Maximum length: 64 characters

```abnf
src = 1*64( ALPHA / DIGIT / "-" )
```

#### 4.2.3. `dst` (Destination Namespace)

- MUST be a non-empty string
- If `"*"`, the message is a broadcast to all namespaces
- If not `"*"`, MUST follow the same format as `src`
- A message MUST NOT have `dst` equal to `src` (a namespace does not message itself)

```abnf
dst = "*" / 1*64( ALPHA / DIGIT / "-" )
```

#### 4.2.4. `type` (Message Type)

- MUST be one of the enumerated values defined in Section 5
- Case-sensitive, lowercase with underscores

```abnf
type = "state" / "alert" / "event" / "request"
     / "data_cross" / "dispatch" / "dojo_event"
```

#### 4.2.5. `msg` (Payload)

- MUST be a non-empty string
- Maximum length: 120 characters (see Section 6), unless an `encoding` mode is specified (see Section 12)
- MUST NOT contain newline characters (`\n`, `\r`)
- SHOULD be a single human-readable sentence or structured token sequence
- Payload conventions are defined in Section 7

```abnf
msg = 1*120(VCHAR / SP)
      ; Printable ASCII + space, no control characters
```

#### 4.2.6. `ttl` (Time-to-Live)

- MUST be a positive integer (minimum 1)
- Represents the number of days from `ts` that the message remains active
- A message is expired when `current_date > ts + ttl days`
- Expired messages SHOULD be moved to the archive file
- RECOMMENDED defaults by type:

| Type        | Recommended TTL |
|-------------|-----------------|
| state       | 7               |
| alert       | 5               |
| event       | 3               |
| request     | 5               |
| data_cross  | 7               |
| dispatch    | 3               |
| dojo_event  | 3               |

#### 4.2.7. `ack` (Acknowledgments)

- MUST be a JSON array
- Each element MUST be a string matching the `src` format
- An empty array `[]` means no namespace has consumed the message
- When a namespace processes a message, it MUST append its own identifier to this array
- A namespace MUST NOT appear more than once in the `ack` array
- For broadcast messages (`dst = "*"`), the `ack` array accumulates all namespaces that have consumed the message

```abnf
ack = "[" [ ns-id *( "," ns-id ) ] "]"
ns-id = DQUOTE 1*64( ALPHA / DIGIT / "-" ) DQUOTE
```

### 4.3. Field Ordering

Field ordering in the JSON object is NOT significant per the JSON specification. However, implementations SHOULD emit fields in the canonical order: `ts`, `src`, `dst`, `type`, `msg`, `ttl`, `ack`. This aids human readability when inspecting bus files directly.

### 4.4. No Extension Fields

A conforming HERMES message MUST contain exactly the 7 specified fields. Additional fields MUST NOT be present. If future extensions require new fields, a new ARC specification MUST be published.

## 5. Message Types

### 5.1. `state` -- State Change

Announces that a namespace's internal state has changed in a way that other namespaces may need to know about.

- **When to emit**: A significant milestone, status transition, or configuration change occurs
- **Who consumes**: Any namespace that depends on or monitors the source namespace
- **Example**: A project moves from "planning" to "active"

### 5.2. `alert` -- Urgent Information

Carries time-sensitive information that another namespace needs to act on promptly.

- **When to emit**: A deadline is approaching, a threshold is breached, or an anomaly is detected
- **Who consumes**: The namespace(s) best positioned to respond
- **Urgency**: Consumers SHOULD process alerts before other message types
- **Example**: A budget threshold has been exceeded

### 5.3. `event` -- Informational Occurrence

Records that something happened. Purely informational; no action is expected from consumers.

- **When to emit**: A notable event occurs that other namespaces may find relevant
- **Who consumes**: Any namespace interested in the event for context or logging
- **Example**: A deployment completed successfully

### 5.4. `request` -- Inter-Namespace Request

One namespace explicitly asks another namespace for something (information, action, decision).

- **When to emit**: A namespace cannot proceed without input from another namespace
- **Who consumes**: The namespace identified in `dst`
- **Response**: The consuming namespace SHOULD emit a `state` or `event` message in reply
- **Example**: Operations requests a cost estimate from finance

### 5.5. `data_cross` -- Permitted Data Crossing

Transfers a data point between namespaces, subject to deployment-specific firewall rules.

- **When to emit**: Verified or projected data needs to flow from one namespace to another
- **Who consumes**: The destination namespace
- **Firewall**: Implementations MUST enforce a namespace crossing policy (see Section 9)
- **Payload conventions**: MUST use the `PROJ:` prefix for projections (see Section 7.2)
- **Example**: Finance reports a verified monthly cost figure to operations

### 5.6. `dispatch` -- Task Assignment

A controller namespace assigns one or more agents to a task.

- **When to emit**: A controller or orchestrator determines that work needs to be routed
- **Who consumes**: The agent(s) being dispatched
- **Authority**: Only namespaces designated as controllers SHOULD emit dispatch messages
- **Example**: A controller assigns the security agent to review a pull request

### 5.7. `dojo_event` -- Evolution System Event

Records events in an agent evolution or progression system (level-ups, badges, quests).

- **When to emit**: An agent achieves a milestone in a training or progression framework
- **Who consumes**: The evolution system controller and optionally the affected agent's namespace
- **Example**: An agent earns a certification badge

## 6. Shannon Capacity Constraint

The `msg` field is limited to 120 characters. This is a deliberate design constraint inspired by Shannon's information theory, not a technical limitation.

### 6.1. Rationale

- **Signal over noise**: A 120-character limit forces the sender to distill the message to its essential signal, eliminating filler and ambiguity.
- **Human scanability**: Operators can scan a bus file and understand each message at a glance, similar to a log stream.
- **Atomicity enforcement**: The constraint naturally discourages packing multiple concerns into a single message. If a payload does not fit in 120 characters, it is likely violating the atomicity principle (Section 7.1) and should be split.

### 6.2. What the Limit Covers

The 120-character limit applies to the decoded string value of the `msg` field, measured in Unicode scalar values (not bytes). JSON escape sequences (e.g., `\u00e9`) count as their decoded character (1 character, not 6).

### 6.3. When 120 Characters Are Insufficient

If a message genuinely requires more detail, the payload SHOULD contain a summary within the 120-character limit and reference an external resource (e.g., a file path, issue ID, or URL) where the full details can be found.

## 7. Payload Encoding Modes

### 7.1. The `encoding` Field

Messages MAY include an OPTIONAL `encoding` field that specifies how the `msg` payload is encoded. This field enables payloads beyond the default 120-character signaling constraint while maintaining full backward compatibility with Phase 0 deployments.

```abnf
encoding = "raw" / "cbor" / "ref"
```

When the `encoding` field is absent, implementations MUST treat the message as `"raw"` encoding. This ensures that existing agents, which do not recognize the `encoding` field, continue to operate correctly.

**Note on Section 4.4 (No Extension Fields):** The `encoding` field is a formally specified OPTIONAL extension per this section. Conforming implementations MUST accept messages both with and without this field.

### 7.2. Encoding Modes

#### 7.2.1. `raw` (Default)

The current behavior. The `msg` field contains a human-readable text payload.

- The `msg` field MUST NOT exceed 120 characters (Section 6)
- RECOMMENDED for signaling, coordination, and control messages
- This is the default when `encoding` is absent

```json
{"ts":"2026-03-15","src":"engineering","dst":"*","type":"state","msg":"API v2.1 deployed to production","ttl":7,"ack":[]}
```

#### 7.2.2. `cbor`

CBOR-encoded payload per RFC 8949. The `msg` field contains a base64-encoded (RFC 4648) CBOR byte string. No size limit applies to the decoded content.

- The `msg` field MUST contain valid base64-encoded data
- The decoded data MUST be valid CBOR
- RECOMMENDED for structured data exchange, bulk transfers, and binary payloads
- Implementations that do not support CBOR SHOULD log a warning and skip the message

```json
{"ts":"2026-03-15","src":"engineering","dst":"ops","type":"data_cross","msg":"o2Rjb3N0GQl4ZG1vbnRoZjIwMjYtMDM=","ttl":7,"ack":[],"encoding":"cbor"}
```

#### 7.2.3. `ref`

Reference payload. The `msg` field contains a file path or URI pointing to the actual content. No size limit on the referenced content.

- The `msg` field MUST contain a valid file path or URI
- The referenced content is outside the bus — the bus carries only the reference
- RECOMMENDED for large payloads, binary data, or content that exceeds atomic signaling
- Implementations MUST NOT automatically fetch or execute referenced content without explicit agent logic

```json
{"ts":"2026-03-15","src":"engineering","dst":"ops","type":"data_cross","msg":"/shared/reports/q1-analysis.json","ttl":7,"ack":[],"encoding":"ref"}
```

### 7.3. Backward Compatibility

The encoding extension is designed for zero-disruption deployment:

1. Messages without an `encoding` field are valid and MUST be treated as `raw`
2. Agents that do not implement encoding modes SHOULD ignore the `encoding` field and process the `msg` field as-is
3. The 120-character limit (Section 6) applies ONLY to `raw` encoding. Messages with `cbor` or `ref` encoding are exempt from this limit
4. When in doubt, senders SHOULD use `raw` encoding for maximum compatibility

### 7.4. Design Rationale

The IETF draft-chang-agent-token-efficient proposes schema deduplication and adaptive fields to reduce token overhead in agent communication. HERMES takes a complementary approach: the `encoding` field preserves the lightweight default (raw signaling) while enabling structured and binary payloads when needed.

## 8. Payload Conventions

### 8.1. Atomicity

**One message, one topic.** A single HERMES message MUST address exactly one concern.

- MUST NOT: `"Budget approved and deployment scheduled and new hire onboarded"`
- MUST: Three separate messages, one for each topic

Violations of atomicity make acknowledgment semantics ambiguous (a consumer may need to ack one concern but not another).

### 8.2. `PROJ:` Prefix for Projections

In `data_cross` messages, the `PROJ:` prefix distinguishes projected or estimated values from verified data.

- `PROJ:` present: The value is a projection, estimate, or forecast. Consumers MUST NOT treat it as confirmed.
- `PROJ:` absent: The value is verified and factual.

```
msg: "PROJ: Q3 hosting cost ~$2,400 based on current trajectory"  -- projection
msg: "Q3 hosting cost $2,187.43 @billing-system"                  -- verified
```

### 8.3. `@source` Suffix for Data Provenance

When a payload includes a quantitative value, the `@source` suffix indicates where the data originated. This enables consumers to assess reliability and trace data lineage.

```
msg: "Monthly infrastructure cost $3,200 @cloud-dashboard"
msg: "Sprint velocity 42 points @project-tracker"
```

- The `@source` token MUST appear after the value it annotates
- The source identifier SHOULD be a recognizable system or namespace name
- The `@source` suffix is RECOMMENDED for `data_cross` messages and OPTIONAL for other types

### 8.4. Combining Conventions

The `PROJ:` prefix and `@source` suffix MAY be used together:

```
msg: "PROJ: Next quarter infra cost ~$4,000 @cloud-dashboard"
```

## 9. Message Lifecycle

### 9.1. Creation

A namespace creates a message by appending a new JSON line to the bus file. The `ack` array MUST be empty at creation time.

### 9.2. Consumption

When a namespace reads and processes a message:

1. It MUST check that its own identifier is NOT already in the `ack` array
2. It MUST append its identifier to the `ack` array
3. It MUST write the updated message back to the bus file

A namespace MUST NOT process the same message twice (idempotency via `ack` check).

### 9.3. Expiration

A message is expired when:

```
current_date > date(ts) + ttl days
```

Expired messages SHOULD be moved from the bus file to the archive file. The archive file uses the same JSONL format.

### 9.4. Broadcast Completion

A broadcast message (`dst = "*"`) has no single "done" state. Implementations MAY define a policy for when a broadcast is considered fully consumed (e.g., all known namespaces are in `ack`).

## 10. Validation Algorithm

The following pseudocode defines the validation procedure for a HERMES message. A conforming implementation MUST reject messages that fail validation.

```
function validate(message) -> (valid: bool, errors: list[string]):
    errors = []

    -- 1. Structure check
    if message is not a JSON object:
        return (false, ["Not a JSON object"])

    required_fields = {"ts", "src", "dst", "type", "msg", "ttl", "ack"}
    actual_fields = set(message.keys())

    if actual_fields != required_fields:
        missing = required_fields - actual_fields
        extra = actual_fields - required_fields
        if missing: errors.append("Missing fields: " + str(missing))
        if extra: errors.append("Extra fields: " + str(extra))
        return (false, errors)

    -- 2. Type checks
    if not is_string(message.ts):
        errors.append("ts must be a string")
    if not is_string(message.src):
        errors.append("src must be a string")
    if not is_string(message.dst):
        errors.append("dst must be a string")
    if not is_string(message.type):
        errors.append("type must be a string")
    if not is_string(message.msg):
        errors.append("msg must be a string")
    if not is_integer(message.ttl):
        errors.append("ttl must be an integer")
    if not is_array(message.ack):
        errors.append("ack must be an array")

    if errors: return (false, errors)

    -- 3. Value constraints
    if not matches(message.ts, /^\d{4}-\d{2}-\d{2}$/):
        errors.append("ts must be YYYY-MM-DD format")
    else if not is_valid_calendar_date(message.ts):
        errors.append("ts is not a valid calendar date")

    if not matches(message.src, /^[a-z0-9][a-z0-9-]{0,63}$/):
        errors.append("src must be 1-64 lowercase alphanumeric/hyphen chars")

    if message.dst != "*":
        if not matches(message.dst, /^[a-z0-9][a-z0-9-]{0,63}$/):
            errors.append("dst must be '*' or 1-64 lowercase alphanumeric/hyphen chars")

    if message.src == message.dst:
        errors.append("src must not equal dst")

    valid_types = {"state", "alert", "event", "request",
                   "data_cross", "dispatch", "dojo_event"}
    if message.type not in valid_types:
        errors.append("type must be one of: " + str(valid_types))

    if len(message.msg) == 0:
        errors.append("msg must not be empty")

    -- 3a. Payload length depends on encoding mode (Section 12)
    encoding = message.get("encoding", "raw")
    if encoding == "raw" or encoding is absent:
        if len(message.msg) > 120:
            errors.append("msg exceeds 120 character limit (got " + str(len(message.msg)) + ")")
    -- For "cbor" and "ref", no length limit on msg

    if contains_control_chars(message.msg):
        errors.append("msg must not contain control characters")

    if message.ttl < 1:
        errors.append("ttl must be >= 1")

    -- 4. Ack array validation
    seen_acks = set()
    for i, entry in enumerate(message.ack):
        if not is_string(entry):
            errors.append("ack[" + str(i) + "] must be a string")
        else if not matches(entry, /^[a-z0-9][a-z0-9-]{0,63}$/):
            errors.append("ack[" + str(i) + "] is not a valid namespace identifier")
        else if entry in seen_acks:
            errors.append("ack contains duplicate: " + entry)
        else:
            seen_acks.add(entry)

    -- 5. Encoding field validation (Section 12)
    if "encoding" in message:
        valid_encodings = {"raw", "cbor", "ref"}
        if message.encoding not in valid_encodings:
            errors.append("encoding must be one of: " + str(valid_encodings))

    return (len(errors) == 0, errors)
```

## 11. Formal Grammar

The following ABNF grammar (RFC 5234) defines the structure of a serialized HERMES message:

```abnf
hermes-message = "{" ts-field "," src-field "," dst-field ","
                     type-field "," msg-field "," ttl-field ","
                     ack-field [ "," encoding-field ] "}"

ts-field   = DQUOTE "ts" DQUOTE ":" DQUOTE date-value DQUOTE
src-field  = DQUOTE "src" DQUOTE ":" DQUOTE ns-id DQUOTE
dst-field  = DQUOTE "dst" DQUOTE ":" DQUOTE dst-value DQUOTE
type-field = DQUOTE "type" DQUOTE ":" DQUOTE type-value DQUOTE
msg-field  = DQUOTE "msg" DQUOTE ":" DQUOTE msg-value DQUOTE
ttl-field  = DQUOTE "ttl" DQUOTE ":" ttl-value
ack-field  = DQUOTE "ack" DQUOTE ":" ack-value
encoding-field = DQUOTE "encoding" DQUOTE ":" DQUOTE encoding-value DQUOTE

date-value = 4DIGIT "-" 2DIGIT "-" 2DIGIT
             ; Valid calendar date in YYYY-MM-DD

ns-id      = 1*64( ALPHA / DIGIT / "-" )
             ; Lowercase letters, digits, hyphens

dst-value  = "*" / ns-id

type-value = "state" / "alert" / "event" / "request"
           / "data_cross" / "dispatch" / "dojo_event"

msg-value  = 1*120( VCHAR / SP )
             ; Printable characters and spaces, no control chars
             ; Length limit relaxed when encoding is "cbor" or "ref"

ttl-value  = 1*DIGIT
             ; Positive integer >= 1

ack-value  = "[" [ DQUOTE ns-id DQUOTE *( "," DQUOTE ns-id DQUOTE ) ] "]"

encoding-value = "raw" / "cbor" / "ref"

; JSON whitespace between tokens is permitted per RFC 8259
; but SHOULD be minimized in bus files for compactness
```

Note: This ABNF describes the *logical* structure. Actual serialization MUST conform to RFC 8259 (JSON), which governs escaping, Unicode handling, and whitespace rules.

## 12. Examples

### 12.1. Valid Messages

**State change:**
```json
{"ts":"2026-03-15","src":"engineering","dst":"operations","type":"state","msg":"API v2.1 deployed to production, rollback window 48h","ttl":7,"ack":[]}
```

**Broadcast alert:**
```json
{"ts":"2026-03-15","src":"finance","dst":"*","type":"alert","msg":"Q1 budget 92% consumed, discretionary spending freeze effective Monday","ttl":5,"ack":["engineering","operations"]}
```

**Data crossing with verified amount:**
```json
{"ts":"2026-03-15","src":"finance","dst":"operations","type":"data_cross","msg":"March cloud infrastructure cost $8,247.33 @billing-system","ttl":7,"ack":[]}
```

**Data crossing with projection:**
```json
{"ts":"2026-03-15","src":"finance","dst":"operations","type":"data_cross","msg":"PROJ: Q2 cloud cost ~$26,000 based on current growth rate @billing-system","ttl":7,"ack":[]}
```

**Request:**
```json
{"ts":"2026-03-15","src":"operations","dst":"engineering","type":"request","msg":"Need capacity estimate for Black Friday traffic by March 22","ttl":5,"ack":[]}
```

**Dispatch:**
```json
{"ts":"2026-03-15","src":"controller","dst":"security","type":"dispatch","msg":"Review PR #847 for auth bypass risk, priority high","ttl":3,"ack":[]}
```

**Dojo event:**
```json
{"ts":"2026-03-15","src":"controller","dst":"*","type":"dojo_event","msg":"Agent monitoring-bot earned badge [RELIABILITY] after 30d zero-downtime","ttl":3,"ack":[]}
```

**Informational event:**
```json
{"ts":"2026-03-15","src":"engineering","dst":"*","type":"event","msg":"Database migration completed, 2.3M rows migrated in 47 minutes","ttl":3,"ack":[]}
```

### 12.2. Invalid Messages

**Missing field (`ack` absent):**
```json
{"ts":"2026-03-15","src":"engineering","dst":"operations","type":"state","msg":"Deployed v2","ttl":7}
```
Validation error: `Missing fields: {'ack'}`

**Extra field:**
```json
{"ts":"2026-03-15","src":"engineering","dst":"operations","type":"state","msg":"Deployed v2","ttl":7,"ack":[],"priority":"high"}
```
Validation error: `Extra fields: {'priority'}`

**Payload exceeds 120 characters:**
```json
{"ts":"2026-03-15","src":"engineering","dst":"operations","type":"state","msg":"This message is intentionally written to be excessively long so that it exceeds the one hundred and twenty character limit imposed by the specification","ttl":7,"ack":[]}
```
Validation error: `msg exceeds 120 character limit (got 152)`

**Self-addressed message:**
```json
{"ts":"2026-03-15","src":"engineering","dst":"engineering","type":"state","msg":"Internal note","ttl":7,"ack":[]}
```
Validation error: `src must not equal dst`

**Invalid type:**
```json
{"ts":"2026-03-15","src":"engineering","dst":"operations","type":"warning","msg":"Something happened","ttl":7,"ack":[]}
```
Validation error: `type must be one of: {state, alert, event, request, data_cross, dispatch, dojo_event}`

**Invalid TTL (zero):**
```json
{"ts":"2026-03-15","src":"engineering","dst":"operations","type":"event","msg":"Something happened","ttl":0,"ack":[]}
```
Validation error: `ttl must be >= 1`

**Duplicate ack entries:**
```json
{"ts":"2026-03-15","src":"engineering","dst":"*","type":"event","msg":"Release complete","ttl":3,"ack":["operations","operations"]}
```
Validation error: `ack contains duplicate: operations`

**Broadcast as source:**
```json
{"ts":"2026-03-15","src":"*","dst":"engineering","type":"alert","msg":"System alert","ttl":5,"ack":[]}
```
Validation error: `src must be 1-64 lowercase alphanumeric/hyphen chars`

## 13. Security Considerations

### 13.1. No Credentials in Payloads

HERMES messages MUST NOT contain credentials, secrets, API keys, tokens, passwords, or any authentication material. The `msg` field is designed for operational signals, not data transfer of sensitive material.

### 13.2. Namespace Firewall

Deployments SHOULD enforce a namespace crossing policy that defines which namespaces are permitted to exchange `data_cross` messages. The policy is deployment-specific and outside the scope of this specification, but implementations MUST provide a mechanism to define and enforce it.

### 13.3. Bus File Access Control

The bus file is a shared resource. Deployments MUST use operating system file permissions (or equivalent) to restrict read/write access to authorized processes only. A namespace agent SHOULD only have the minimum permissions required:

- Read access to the bus file (to consume messages)
- Append access to the bus file (to emit messages)
- Write access to update `ack` arrays

### 13.4. Integrity

This specification does not define a message signing or integrity mechanism. Deployments operating in adversarial environments SHOULD layer integrity protection (e.g., file-system-level checksums, signed commits if the bus file is in a Git repository).

### 13.5. Denial of Service

A misbehaving namespace could flood the bus file with messages. Implementations SHOULD enforce rate limits per namespace and MAY reject messages from namespaces that exceed a configured threshold.

### 13.6. Ack Array Tampering

A malicious process could modify `ack` arrays to suppress or replay message delivery. Deployments concerned with this threat SHOULD use append-only storage or version-controlled bus files where history is preserved.
