## 1. Abstract

This specification defines the session lifecycle, delivery guarantees, and
message management protocol for HERMES, a file-based inter-namespace
communication system for AI agents.

Drawing from the design principles of TCP (RFC 793), this standard establishes
a SYN/FIN/ACK handshake around agent sessions, time-to-live (TTL) semantics for
messages, and a SYNC router function for consistency enforcement. Unlike TCP,
HERMES operates over a shared append-only bus file rather than a network socket,
making it suitable for AI agent orchestration across isolated execution contexts
(namespaces).

The protocol provides **at-least-once** delivery semantics. It does not
guarantee exactly-once delivery; consumers MUST be prepared to handle duplicate
messages idempotently.

---

## 2. Terminology

| Term          | Definition                                                        |
|---------------|-------------------------------------------------------------------|
| **Namespace** | An isolated execution context with its own configuration, permissions, and bus access scope. |
| **Bus**       | An append-only JSONL file serving as the shared message transport. |
| **Packet**    | A single JSON object (one line) in the bus file.                  |
| **Session**   | A bounded period of agent activity within a namespace.            |
| **SYN**       | Session initialization handshake: read and filter the bus.        |
| **FIN**       | Session teardown handshake: write state changes and ACK consumed messages. |
| **SYNC**      | Router function that audits global consistency.                   |
| **ACK**       | Acknowledgement that a namespace has consumed a message.          |
| **TTL**       | Time-to-live: maximum age in days before a message expires.       |
| **SYNC HEADER** | A metadata block in each namespace's configuration file reflecting its last-known state. |
| **Archive**   | A secondary bus file where expired messages are moved.            |
| **Firewall**  | Namespace-level access control that restricts credential and service crossing. |

## 3. Session Lifecycle

An Amaru session follows a three-phase lifecycle modeled after TCP's connection
management. Every session MUST execute SYN at start and FIN at close. The active
phase between them is unconstrained.

### 3.1. State Diagram

```
                    ┌─────────────┐
                    │   CLOSED    │
                    └──────┬──────┘
                           │ Agent starts in namespace X
                           v
                    ┌─────────────┐
                    │  SYN_RECV   │  Read bus, filter, display
                    └──────┬──────┘
                           │ Pending messages consumed & ACKed
                           v
                    ┌─────────────┐
                    │   ACTIVE    │  Normal session work
                    │             │  (reads, writes, computations)
                    └──────┬──────┘
                           │ Session ending
                           v
                    ┌─────────────┐
                    │  FIN_WAIT   │  Harvest, write packets,
                    │             │  update SYNC HEADER, ACK
                    └──────┬──────┘
                           │ All FIN steps completed
                           v
                    ┌─────────────┐
                    │   CLOSED    │
                    └─────────────┘
```

### 3.2. Lifecycle Invariants

1. **No session without SYN.** An agent MUST NOT perform work in a namespace
   without first executing the SYN protocol.
2. **No exit without FIN.** An agent MUST NOT terminate a session without
   executing the FIN protocol, even if the session produced no state changes.
3. **Atomic FIN.** All FIN operations (harvest, bus writes, SYNC HEADER update,
   ACK) MUST complete as a logical unit. A partial FIN leaves the system in an
   inconsistent state that SYNC will detect and flag.
4. **One session per namespace.** A namespace MUST NOT have concurrent active
   sessions writing to the same bus file.

## 4. SYN Protocol

### 4.1. Purpose

SYN initializes a session by reading the bus, filtering for relevant messages,
and presenting them to the human operator.

### 4.2. Procedure

```
SYN(namespace X):
    1. OPEN bus file for reading
    2. PARSE each line as a JSON packet
    3. FILTER packets where:
         (packet.dst == X  OR  packet.dst == "*")
         AND  X NOT IN packet.ack[]
    4. SORT filtered packets by packet.ts ascending
    5. COMPUTE count = |filtered packets|
    6. IF count > 0:
         a. DISPLAY "[HERMES] {count} pending messages"
         b. LIST each message: ts, src, type, summary
         c. For each message where age(packet.ts) > 3 days:
              FLAG "[STALE] message from {src}, {age} days old"
    7. ELSE:
         DISPLAY "[HERMES] No pending messages"
    8. TRANSITION to ACTIVE state
```

### 4.3. Stale Message Detection

A message is considered **stale** when it has remained unacknowledged for more
than 3 days. Stale messages trigger an alert during SYN to ensure they are not
silently ignored.

```
stale(packet) := (today - packet.ts) > 3 days
                 AND namespace NOT IN packet.ack[]
```

Stale detection is informational. The human operator decides whether to act on,
defer, or dismiss a stale message.

### 4.4. Filter Semantics

The filter `dst == X OR dst == "*"` implements unicast and broadcast delivery:

- **Unicast** (`dst == "namespace_name"`): Message intended for a single
  namespace.
- **Broadcast** (`dst == "*"`): Message intended for all namespaces.

A message is considered **consumed** by namespace X when X appears in the
`ack[]` array. Consumed messages are excluded from SYN output.

## 5. FIN Protocol

### 5.1. Purpose

FIN gracefully closes a session by persisting state changes, acknowledging
consumed messages, and updating the namespace's SYNC HEADER.

### 5.2. Procedure

```
FIN(namespace X, session S):
    1. EXECUTE session harvest (implementation-defined)
         - Record: what was built, decisions, insights, pending items
    2. IF session S produced state changes:
         a. CONSTRUCT packet(s) with:
              ts     = current date (YYYY-MM-DD)
              src    = X
              dst    = target namespace | "*"
              type   = "state" | "alert" | "event" | "data_cross"
              msg    = payload string
              ttl    = default for type (see Section 7)
              ack    = []
         b. APPEND packet(s) to bus file (one JSON object per line)
    3. UPDATE SYNC HEADER in namespace X's configuration:
         - INCREMENT version number
         - SET last_sync = current date
         - SET state = summary of current namespace state
         - SET pending_out = count of messages written this session
         - SET pending_in = count of unACKed messages targeting X
    4. FOR each message consumed during this session:
         ADD X to packet.ack[] in the bus file
    5. TRANSITION to CLOSED state
```

### 5.3. Packet Construction Rules

- The `src` field MUST be the current namespace identifier.
- The `dst` field MUST be a valid namespace identifier or `"*"`.
- The `type` field MUST be one of the defined message types (Section 7.2).
- The `msg` field SHOULD be a concise, self-contained payload.
- Projections (future estimates, forecasts) MUST use the `PROJ:` prefix in the
  `msg` field.
- The `ack` array MUST be initialized as empty `[]`.

### 5.4. SYNC HEADER Update

The SYNC HEADER is a metadata block embedded in each namespace's configuration
file. It provides a snapshot of the namespace's last-known synchronization
state. See Section 10 for the full format specification.

## 6. SYNC Protocol (Router Function)

### 6.1. Purpose

SYNC is a global consistency auditor. It reads all namespace configurations and
the full bus, detects inconsistencies, and proposes corrections. SYNC does not
modify state autonomously; all corrections require human approval.

### 6.2. Procedure

```
SYNC():
    1. READ all SYNC HEADERs from each namespace configuration
    2. READ full bus file
    3. FOR each namespace N:
         a. COMPUTE expected pending_in:
              count of packets where (dst == N OR dst == "*")
                                 AND N NOT IN ack[]
                                 AND NOT expired(packet)
         b. COMPARE with N.sync_header.pending_in
         c. IF mismatch: FLAG "desync: {N} reports {reported}, actual {computed}"
    4. FOR each packet P in bus:
         a. IF expired(P): FLAG "expired: {P.ts} {P.src}->{P.dst} ({P.type})"
         b. IF age(P) > 3 AND P.ack == []: FLAG "orphan: unACKed for {age} days"
    5. FOR each namespace N:
         a. IF N.sync_header.last_sync is stale (> 7 days):
              FLAG "stale header: {N} last synced {last_sync}"
    6. PRESENT all flags to human operator
    7. PROPOSE fixes:
         - Update mismatched SYNC HEADERs
         - ACK messages consumed but not yet acknowledged
         - Archive expired messages
    8. WITH human approval:
         a. UPDATE SYNC HEADERs
         b. APPEND X to ack[] for consumed messages
         c. MOVE expired messages to archive file
```

### 6.3. SYNC Invariants

1. **Read-only by default.** SYNC MUST NOT modify any state without explicit
   human approval.
2. **Full visibility.** SYNC reads across all namespaces. This is the only
   operation permitted to cross namespace boundaries.
3. **Idempotent.** Running SYNC multiple times without intervening changes MUST
   produce identical results.

## 7. TTL Specification

### 7.1. Definition

TTL (Time-to-Live) defines the maximum age of a message in days, measured from
the `ts` (timestamp) field. A message is **expired** when:

```
expired(packet) := (today - packet.ts) > packet.ttl
```

### 7.2. Defaults by Message Type

| Type          | TTL (days) | Description                                    |
|---------------|:----------:|------------------------------------------------|
| `state`       | 7          | Namespace state snapshot. Long-lived.           |
| `alert`       | 5          | Requires attention. Medium urgency.             |
| `event`       | 3          | Informational event. Short-lived.               |
| `data_cross`  | 7          | Cross-namespace data transfer. Long-lived.      |
| `dispatch`    | 3          | Task routing directive. Short-lived.            |
| `dojo_event`  | 3          | System governance event. Short-lived.           |

### 7.3. Custom TTL

Implementations MAY specify a custom TTL per packet by setting the `ttl` field
to a positive integer. The custom value overrides the type default.

### 7.4. Expiration Handling

Expired messages are:

1. **Flagged** by SYNC during consistency audits.
2. **Archived** (moved from the active bus to the archive file) upon human
   approval.
3. **Never deleted.** The archive serves as a permanent historical record.

## 8. ACK Mechanism

### 8.1. Design

HERMES uses an **array-based acknowledgement** model. Each packet contains an
`ack` field: an array of namespace identifiers that have consumed the message.

```json
{
  "ts": "2026-02-28",
  "src": "alpha",
  "dst": "*",
  "type": "state",
  "msg": "Deployment completed.",
  "ttl": 7,
  "ack": ["beta", "gamma"]
}
```

In this example, namespaces `beta` and `gamma` have consumed the message.
Namespace `alpha` (the sender) is implicitly aware and does not need to ACK its
own messages, though it MAY appear in `ack[]` if it re-consumed the message in
a later session.

### 8.2. ACK Timing

ACKs are written during the FIN phase of a session. A namespace SHOULD NOT
delay ACKs beyond the session in which a message was consumed.

### 8.3. ACK Semantics

- **Presence in `ack[]`** indicates the namespace has read and processed the
  message.
- **Absence from `ack[]`** means the message is still pending for that
  namespace.
- A namespace appearing in `ack[]` more than once has no additional semantic
  meaning. Implementations SHOULD deduplicate but MUST tolerate duplicates.

### 8.4. Full ACK

A message is **fully acknowledged** when all intended recipients have ACKed:

- **Unicast**: `ack[]` contains the `dst` namespace.
- **Broadcast**: `ack[]` contains all active namespaces. Since the set of
  active namespaces is dynamic, broadcast messages rely on TTL expiration
  rather than full-ACK detection for lifecycle management.

## 9. Delivery Guarantees

### 9.1. At-Least-Once Delivery

HERMES guarantees **at-least-once** delivery. A message written to the bus
will be presented to its destination namespace during SYN at least once,
provided:

1. The destination namespace starts a session before the message expires.
2. The bus file is not corrupted (see Section 12).

### 9.2. Why Not Exactly-Once

Exactly-once delivery requires transactional semantics across the write (FIN)
and read (SYN) boundaries. Since HERMES operates over plain files without
locking or transactions, the following scenarios can produce duplicates:

- A session crashes after consuming a message but before writing the ACK.
- A SYNC operation re-presents a message that was consumed but whose ACK was
  lost due to a partial FIN.

Consumers MUST handle duplicate delivery idempotently. Implementations SHOULD
use the combination of `(ts, src, type, msg)` as a deduplication key.

### 9.3. Ordering

Messages are ordered by `ts` (date granularity). Messages with the same `ts`
are presented in bus-file order (append order). HERMES does not guarantee
strict total ordering across concurrent writers; within a single session's
FIN, messages are ordered by their append sequence.

## 10. SYNC HEADER Format

### 10.1. Purpose

The SYNC HEADER is an inline metadata block in each namespace's configuration
file. It provides a fast-readable summary of synchronization state without
requiring a full bus scan.

### 10.2. Format

The SYNC HEADER is a Markdown table embedded in the namespace configuration
file, delimited by a comment marker:

```markdown
## SYNC HEADER
<!-- HERMES Reliable Transport (ARC-0793) -- do not edit manually -->
| Field       | Value                              |
|-------------|------------------------------------|
| version     | <monotonically increasing integer> |
| last_sync   | <YYYY-MM-DD>                       |
| state       | <free-text summary of namespace>   |
| pending_out | <count of messages sent this FIN>  |
| pending_in  | <count of unACKed inbound messages> |
```

### 10.3. Field Definitions

| Field         | Type    | Description                                              |
|---------------|---------|----------------------------------------------------------|
| `version`     | integer | Monotonically increasing counter, incremented each FIN.  |
| `last_sync`   | date    | Date of the most recent FIN (YYYY-MM-DD).                |
| `state`       | string  | Human-readable summary of namespace state.               |
| `pending_out` | integer | Count of packets written to the bus during the last FIN. |
| `pending_in`  | integer | Count of unACKed packets targeting this namespace.       |

### 10.4. Version Monotonicity

The `version` field MUST be strictly increasing. Each FIN increments it by
exactly 1. SYNC uses version comparisons to detect missed sessions (gaps in
the version sequence may indicate sessions that failed to complete FIN).

## 11. Mathematical Foundations

### 11.1. Queue Model

The Amaru bus can be modeled as a single-server queue where:

- **Arrivals** = messages appended during FIN phases across all namespaces.
- **Service** = messages consumed (ACKed) during SYN phases.
- **Queue** = unACKed messages currently in the bus.

This maps to an **M/M/1 queue** under the simplifying assumption that arrivals
and service follow Poisson processes.

### 11.2. Little's Law

Little's Law relates the average number of messages in the system (L), the
average arrival rate (lambda), and the average time a message spends in the
system (W):

```
L = lambda * W
```

Where:
- **L** = average number of unACKed messages in the bus.
- **lambda** = average number of messages appended to the bus per day.
- **W** = average time (in days) a message remains unACKed.

**Operational interpretation:** If the average message lifetime (W) exceeds the
TTL, the bus will accumulate expired messages faster than SYNC can archive them.
A healthy system maintains `W < min(TTL)` for all message types.

### 11.3. Stability Condition

For the queue to remain stable (not grow without bound):

```
lambda < mu
```

Where **mu** is the average service rate (messages ACKed per day). When
`lambda >= mu`, the bus grows monotonically, indicating that namespaces are
producing messages faster than they are consuming them.

**Monitoring heuristic:** If `pending_in` across all SYNC HEADERs trends
upward over 3 consecutive SYNC cycles, the system is approaching instability.

### 11.4. TTL as a Safety Valve

TTL expiration acts as a **bounded buffer** mechanism. Even if `lambda > mu`
temporarily, expired messages are eligible for archival, preventing unbounded
bus growth. The effective queue size is bounded by:

```
L_max = lambda * max(TTL)
```

This bound assumes all messages use the maximum TTL and none are ACKed, which
represents the worst-case scenario.

## 12. Error Handling

### 12.1. Stale Messages

**Definition:** A message is stale when `age > 3 days` and `ack[] `does not
contain the destination namespace.

**Handling:**
1. SYN flags stale messages with a `[STALE]` marker.
2. SYNC reports stale messages as potential orphans.
3. Human operator decides: ACK (dismiss), act on, or leave pending.

### 12.2. Missing SYNC HEADERs

**Definition:** A namespace configuration file lacks a SYNC HEADER block.

**Handling:**
1. SYNC flags the namespace as `"no SYNC HEADER"`.
2. Proposed fix: insert a default SYNC HEADER with `version=0`,
   `last_sync=never`, `state=unknown`, `pending_out=0`,
   `pending_in=<computed>`.
3. Applied only with human approval.

### 12.3. Bus Corruption

**Definition:** A line in the bus file cannot be parsed as valid JSON.

**Handling:**
1. The corrupted line is logged with its line number and raw content.
2. The line is skipped for processing. SYN and SYNC continue with remaining
   valid lines.
3. SYNC reports the corruption count.
4. Manual repair is required. Implementations MUST NOT auto-repair corrupted
   lines, as the original intent cannot be reliably inferred.

### 12.4. Partial FIN

**Definition:** A session completes some but not all FIN steps (e.g., packets
written but SYNC HEADER not updated).

**Handling:**
1. SYNC detects the inconsistency via version/pending_in mismatches.
2. The next SYN in the affected namespace may re-present already-consumed
   messages (at-least-once delivery).
3. SYNC proposes reconciliation: update SYNC HEADER, write missing ACKs.

### 12.5. Clock Skew

**Definition:** The `ts` field is set to a date in the future or significantly
in the past relative to the current date.

**Handling:**
1. Implementations SHOULD warn if `|ts - today| > 1 day` for newly created
   packets.
2. TTL calculations use the `ts` field as-is. Clock skew may cause premature
   expiration or delayed expiration.
3. Implementations MUST NOT auto-correct timestamps.

## 13. Security Considerations

### 13.1. Namespace Firewall

The bus is a shared transport, but namespace isolation MUST be enforced at the
application layer:

1. **Credential isolation.** Messages MUST NOT contain credentials, API keys,
   tokens, or secrets. The bus file may be readable by all namespaces.
2. **Service boundary.** Each namespace has a set of permitted and prohibited
   external services. The bus MUST NOT be used to circumvent these boundaries.
3. **Data classification.** Cross-namespace data transfers (`data_cross` type)
   carry categorized data only (e.g., expense amounts with category labels),
   never raw credentials or service handles.

### 13.2. Bus File Permissions

The bus file SHOULD be readable and writable only by the agent runtime user.
Recommended permissions: `600` (owner read/write only).

### 13.3. Archive File Permissions

The archive file contains historical messages and SHOULD have the same
permission model as the active bus file.

### 13.4. Projection Integrity

Messages containing projections (prefixed with `PROJ:`) represent estimates
and forecasts. Implementations SHOULD clearly distinguish projections from
factual state updates in all display contexts to prevent decision-making based
on unverified data.

### 13.5. Human-in-the-Loop

HERMES is an **informational protocol.** It informs the human operator; it does
not autonomously execute decisions. The following operations MUST require
explicit human approval:

- Archiving expired messages
- Updating SYNC HEADERs via SYNC
- Acting on alert-type messages
- Any cross-namespace data transfer

### 13.6. Denial of Service

A misbehaving namespace could flood the bus with messages, degrading
performance for all namespaces. Mitigations:

1. **Rate awareness.** SYNC SHOULD flag namespaces whose `pending_out`
   consistently exceeds the system average by more than 2 standard deviations.
2. **TTL enforcement.** Short-lived message types (`event`, `dispatch`) expire
   quickly, limiting the impact of message floods.
3. **Bus size monitoring.** Implementations SHOULD warn when the active bus
   exceeds a configurable size threshold (recommended default: 1000 lines).

## 14. Packet Schema

### 14.1. JSON Schema

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "HERMES Packet",
  "type": "object",
  "required": ["ts", "src", "dst", "type", "msg", "ttl", "ack"],
  "properties": {
    "ts": {
      "type": "string",
      "pattern": "^\\d{4}-\\d{2}-\\d{2}$",
      "description": "ISO 8601 date (YYYY-MM-DD)"
    },
    "src": {
      "type": "string",
      "minLength": 1,
      "description": "Source namespace identifier"
    },
    "dst": {
      "type": "string",
      "minLength": 1,
      "description": "Destination namespace identifier or '*' for broadcast"
    },
    "type": {
      "type": "string",
      "enum": ["state", "alert", "event", "data_cross", "dispatch", "dojo_event"],
      "description": "Message type determining TTL default and semantics"
    },
    "msg": {
      "type": "string",
      "description": "Message payload. Projections prefixed with 'PROJ:'"
    },
    "ttl": {
      "type": "integer",
      "minimum": 1,
      "description": "Time-to-live in days"
    },
    "ack": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Array of namespace identifiers that have consumed this message"
    }
  },
  "additionalProperties": false
}
```

### 14.2. Example Packets

**State update (unicast):**
```json
{"ts":"2026-02-28","src":"alpha","dst":"beta","type":"state","msg":"Sprint 2 started. 5 tickets in backlog.","ttl":7,"ack":[]}
```

**Alert (broadcast):**
```json
{"ts":"2026-02-28","src":"gamma","dst":"*","type":"alert","msg":"Dependency vulnerability detected. Review required.","ttl":5,"ack":["alpha"]}
```

**Projection:**
```json
{"ts":"2026-02-28","src":"delta","dst":"epsilon","type":"data_cross","msg":"PROJ: Estimated Q2 cost $4,200 based on current trajectory.","ttl":7,"ack":[]}
```

## 15. Conformance

An implementation is conformant with ARC-0793 if it:

1. Executes SYN at session start per Section 4.
2. Executes FIN at session close per Section 5.
3. Uses the packet schema defined in Section 14.
4. Respects TTL semantics per Section 7.
5. Implements at-least-once delivery per Section 9.
6. Maintains SYNC HEADER per Section 10.
7. Enforces namespace firewall rules per Section 13.1.
8. Requires human approval for all state-modifying operations per Section 13.5.

Optional conformance:

- SYNC router implementation (Section 6).
- Mathematical monitoring heuristics (Section 11).
- Bus size monitoring (Section 13.6).

## Appendix A: Protocol Comparison with TCP

| Aspect              | TCP (RFC 793)              | HERMES (ARC-0793)                     |
|---------------------|----------------------------|---------------------------------------|
| Transport medium    | Network socket             | Append-only file (JSONL)              |
| Connection          | SYN/SYN-ACK/ACK 3-way     | SYN (read bus) / FIN (write bus)      |
| Ordering            | Sequence numbers           | Timestamp + append order              |
| Reliability         | Retransmission             | At-least-once via re-presentation     |
| Flow control        | Window size                | TTL expiration                        |
| Congestion control  | Slow start, AIMD           | Bus size monitoring, rate awareness   |
| Teardown            | FIN/FIN-ACK                | FIN (harvest + write + ACK)           |
| Scope               | Point-to-point             | Unicast + broadcast                   |
| Error detection     | Checksum                   | JSON parse validation                 |
| Human-in-the-loop   | No                         | Yes (all state changes)               |
